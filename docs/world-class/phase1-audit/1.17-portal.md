## 1. Обзор модуля

### Масштаб
| Метрика | Значение |
|---------|----------|
| Backend Java файлов | ~108 |
| Frontend TSX файлов | 18 страниц + types.ts + index.ts |
| API файл | portal.ts (~290 LOC) |
| Domain entities | 24 (с enum-ами) |
| Repositories | 11 |

| Frontend routes | 13 зарегистрированных + 5 без маршрутов |
| i18n keys | portal.* (используются через t()) |

### Что реализовано
- **Отдельная авторизация** для портальных пользователей (register/login/JWT)
- **4 роли портала**: CUSTOMER, CONTRACTOR, SUBCONTRACTOR, SUPPLIER
- **Проекты**: просмотр назначенных проектов с уровнями доступа (VIEW_ONLY, LIMITED, FULL)
- **Документы**: sharing, download tracking, expiration
- **Сообщения**: bidirectional messaging (portal ↔ internal), threads
- **Задачи**: назначение, статусы, приоритеты, дедлайны
- **КС-2 черновики**: создание подрядчиком, submit → review → approve/reject workflow
- **Рекламации клиентов**: CLM-{seq} нумерация, SLA по приоритету, state machine
- **Прогресс-снапшоты**: еженедельные фотоотчёты, milestone tracking
- **Электронная подпись**: request → sign/reject → track
- **Вехи клиента**: видимые заказчику milestones
- **Админка портала**: управление пользователями, доступом
- **Настройки/брендирование**: логотип, цвета, visibility toggles
- **Frontend**: 18 полноценных страниц с React Query, dark mode, i18n

### Архитектура
```
PortalAuthService → JWT (ROLE_PORTAL_*) → PortalControllers → PortalServices → DB
                                        → PortalDataProxyController → Native SQL (!!!)
Frontend: portalApi → useQuery/useMutation → Pages с dark mode + i18n
```

#### BUG-10: SLA calculation error в ClientClaimService
**Файл**: `ClientClaimService.java:59,134`

```java
// При создании: SLA = now + priority.days (корректно)
claim.setSlaDeadline(now.plus(priority.getSlaDays(), ChronoUnit.DAYS));

// При triage с изменением priority: SLA = now + new_days (НЕКОРРЕКТНО)
// Должно быть: SLA = createdAt + new_days
```

SLA «усыхает» если приоритет повышен через 3 дня после создания.
Пример: LOW (14d) → создан 1 марта → triage 5 марта → пересчёт → SLA = 5 марта + 3d = 8 марта (вместо 1 марта + 3d = 4 марта)

**Severity**: HIGH — некорректные SLA дедлайны → контрактные споры

## 3. Анализ по ролям

### 👷 СМЕТЧИК
- **КС-2 черновики**: State machine (DRAFT→SUBMITTED→REVIEW→APPROVED/REJECTED) корректна
- **Суммы**: totalAmount хранится в БД, но НЕТ позиционного расчёта (lines_json — JSON blob)
- **НДС**: НЕ считается отдельно в КС-2 drafts (отсутствует vatRate/vatAmount)
- **Оценка**: 4/10 — нет строчной детализации, нет автоматических расчётов

### 👷 ПРОРАБ
- **Ежедневные отчёты**: PortalDailyReportsPage есть, но route не зарегистрирован → 404
- **Дефекты**: PortalDefectsPage есть, но route не зарегистрирован → 404
- **Фотоотчёты**: PortalPhotoReportsPage есть, file upload не работает
- **RFI**: PortalRfiPage есть, но route не зарегистрирован → 404
- **Оценка**: 3/10 — 4 из 5 полевых страниц недоступны

### 💼 БУХГАЛТЕР
- **Invoices**: создание, submit, delete — работает
- **Contracts**: просмотр с payment progress — работает
- **КС-2**: есть workflow, но нет НДС расчётов
- **НДС**: полностью отсутствует в financial dashboard
- **Оценка**: 4/10 — нет НДС, нет акта сверки, нет печатных форм

### 🔒 БЕЗОПАСНИК
- **SQL Injection**: 7 мест в PortalDataProxyController — КРИТИЧНО

## 6. Проблемы архитектуры

1. **PortalDataProxyController — anti-pattern**: ~450 LOC raw SQL в контроллере. Нет service layer, нет repository. Бизнес-логика в web layer.
2. **portalUserId не в JWT**: Фундаментальная проблема — невозможно validate identity server-side. Нужно: embed portalUserId в JWT claims → extract через SecurityUtils.
3. **3 entity без organizationId**: PortalMessage, PortalProject, PortalDocument — невозможна tenant isolation на уровне БД. Нужна миграция + @Filter.
4. **Dual query pattern**: PortalDataProxyController дублирует queries из основных модулей (contracts, invoices) → divergence risk.
5. **No PortalSecurityUtils**: Нет утилиты для извлечения portal user context из JWT. Каждый controller изобретает своё решение.

## 8. Что сделано хорошо

1. **Полноценная отдельная авторизация** — JWT с ROLE_PORTAL_* ролями, не shared с internal users
2. **State machines**: КС-2 draft (DRAFT→SUBMITTED→REVIEW→APPROVED/REJECTED) и ClientClaim (SUBMITTED→TRIAGED→ASSIGNED→IN_PROGRESS→VERIFICATION→CLOSED) — правильные бизнес-процессы
3. **SLA tracking**: ClaimPriority с автоматическим расчётом дедлайна (LOW=14d, MEDIUM=7d, HIGH=3d, CRITICAL=1d)
4. **Frontend quality**: 18 страниц с consistent design — dark mode, i18n, React Query, loading/error states
5. **Rich domain model**: 24 entity с proper enums, 34 DTO — правильное разделение domain/transport
6. **Access levels**: VIEW_ONLY, LIMITED, FULL — гранулярный доступ к проектам
7. **Document expiration**: sharing с TTL, download counter — правильная модель для sensitive docs
8. **Claim activity log**: полный audit trail для рекламаций (comments, status changes, photos)
9. **Progress snapshots**: периодические отчёты с фото, milestone summary — ценно для заказчика
10. **Bidirectional messaging**: portal ↔ internal users с threads
