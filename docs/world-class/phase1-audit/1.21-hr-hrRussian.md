## 1. Общий вердикт

HR-модуль — один из наиболее полных в системе. 6 субмодулей покрывают полный кадровый цикл от рекрутинга до расчёта зарплаты. **PayrollService** — лучший расчётный движок в системе: прогрессивный НДФЛ (5 ступеней, НК РФ ст.224 ред. 2024), порог ОПС (ФЗ-178 2025), ст.152/153/154 ТК РФ для сверхурочных/ночных/праздничных. **Employee entity** — reference implementation: PII-шифрование (@EncryptedFieldConverter), @Filter tenant isolation, валидация СНИЛС (@ValidSnils).

Критические проблемы: **8 entities без tenant isolation** (recruitment, leave, payroll), 2 Kanban-доски — пустые оболочки без API, **PayrollService нарушает модульную изоляцию** (импорты из 3 чужих модулей).

## 3. Найденные баги

### P1 — Critical

#### BUG-124: [SECURITY] Recruitment — 3 entity БЕЗ organizationId/tenant filter
**Файлы**: `modules/recruitment/domain/Applicant.java:20-79`, `JobPosition.java`, `Interview.java`
**Проблема**: Entities Applicant, JobPosition, Interview НЕТ поля `organizationId`, НЕТ `@Filter(name = "tenantFilter")`. RecruitmentService не фильтрует по организации.
**Импакт**: Любой authenticated user видит кандидатов, вакансии, собеседования ВСЕХ организаций. Утечка ПДн (ФИО, email, phone, salary, resume).
**Воспроизведение**: GET /api/v1/recruitment/applicants — вернёт ВСЕ applicants всех орг.

#### BUG-125: [SECURITY] Leave — 3 entity БЕЗ organizationId/tenant filter
**Файлы**: `modules/leave/domain/LeaveRequest.java:34-67`, `LeaveAllocation.java`, `LeaveType.java`
**Проблема**: НЕТ поля `organizationId`, НЕТ `@Filter`. LeaveService.listLeaveRequests() (строка 104) вызывает `findAll(pageable)` — возвращает ВСЕ leave requests глобально.
**Импакт**: Персональные данные отпусков сотрудников видны другим организациям.

#### BUG-126: [SECURITY] Payroll — 2 entity БЕЗ organizationId/tenant filter
**Файлы**: `modules/payroll/domain/PayrollCalculation.java:33-121`, `PayrollTemplate.java`
**Проблема**: НЕТ `organizationId`, НЕТ `@Filter`. PayrollService.listTemplates() и listCalculations() возвращают данные всех орг.
**Импакт**: **КРИТИЧЕСКИ ОПАСНО** — зарплатные данные (grossPay, netPay, deductions) = наиболее чувствительная информация. Cross-tenant утечка зарплат = нарушение ФЗ-152, потенциальный иск.

#### BUG-127: [SECURITY] EmploymentContractService.createContract() — НЕ устанавливает organizationId
**Файл**: `modules/hrRussian/service/EmploymentContractService.java:54-68`
**Проблема**: Entity EmploymentContract имеет @Filter и organizationId поле, но builder в createContract() не вызывает `.organizationId(SecurityUtils.requireCurrentOrganizationId())`. Контракт создаётся с `organizationId = null` → tenant filter не находит его.
**Импакт**: Созданные трудовые договоры «исчезают» после создания (tenant filter отсекает null).

#### BUG-128: [SECURITY] 15+ GET endpoints без @PreAuthorize в HR-модулях
**Файлы**:
- `LeaveController.java:47-52,55-60,82-92,94-99,147-156,160-169,171-176,178-185` — 8 GET endpoints без @PreAuthorize (leave requests, types, allocations, balances)
- `PayrollController.java:48-57,61-69,72-77,109-118,120-125` — 5 GET endpoints без @PreAuthorize (payroll calculations, templates)
- `RecruitmentController.java:50-60,62-67,109-119,121-138,133-138,170-175,179-188,190-195` — 10+ GET endpoints без @PreAuthorize (applicants, jobs, interviews, stages)
**Проблема**: Любой authenticated user (включая VIEWER) может видеть зарплаты, отпуска, данные кандидатов.
**Импакт**: VIEWER роль видит зарплатные расчёты и данные рекрутинга.

#### BUG-129: [ARCH] PayrollService — cross-module imports нарушают изоляцию модулей
**Файл**: `modules/payroll/service/PayrollService.java:4-9`
**Проблема**: Импортирует из 3 чужих модулей:
- `calendar.service.WorkCalendarService` — нормативные часы
- `finance.domain.BudgetCategory`, `finance.domain.BudgetItem`, `finance.repository.BudgetItemRepository` — бюджет ФОТ
- `hr.domain.TimesheetStatus`, `hr.repository.TimesheetRepository` — табели
**Импакт**: Нарушение модульной изоляции (CLAUDE.md: «модули изолированы, не импортировать между модулями»). Tight coupling: изменение в finance/hr/calendar ломает payroll.

#### BUG-130: [DATA] LeaveBoardPage + ApplicantBoardPage — пустые оболочки без API
**Файлы**: `frontend/src/modules/leave/LeaveBoardPage.tsx:38`, `frontend/src/modules/recruitment/ApplicantBoardPage.tsx:40`
**Проблема**: `useState<LeaveCard[]>([])` / `useState<ApCard[]>([])` — инициализируются пустым массивом, НЕТ useQuery, НЕТ API вызовов. Доски ВСЕГДА пустые.
**Дополнительно**: Используют HTML5 Drag & Drop (onDragStart/onDrop) вместо @dnd-kit → сломано на мобильных.
**Импакт**: Пользователь видит пустые Kanban-доски без данных. Обещанная функциональность не работает.

#### BUG-131: [DATA] LeaveService.listLeaveRequests() — findAll() без tenant filter
**Файл**: `modules/leave/service/LeaveService.java:104`
**Проблема**: При отсутствии employeeId/approverId/status фильтра, вызывает `leaveRequestRepository.findAll(pageable)` — стандартный JPA findAll без любых фильтров. Поскольку LeaveRequest не имеет @Filter, это возвращает ВСЕ записи ВСЕХ организаций.
**Импакт**: Открыв список отпусков без фильтра → видны отпуска сотрудников всех организаций.

#### BUG-135: [CALC] PayrollService — socialRate 2.9% = ОСС только ВНиМ, нет НС и ПЗ
**Файл**: `PayrollService.java:94`
**Проблема**: Default socialRate = 2.9% (ОСС по временной нетрудоспособности). Но для строительства обязательна доп. ставка по НС и ПЗ (несчастные случаи и профзаболевания) = 0.2%–8.5% в зависимости от класса профриска (VII-XXXII для строительства = 0.7%–4.1%). Не учтена.
**Импакт**: Работодатель недоплачивает взносы НС и ПЗ → штраф ФСС.

#### BUG-136: [CALC] PayrollService.calculateBasePay PIECEWORK — возвращает baseSalary вместо расчёта
**Файл**: `PayrollService.java:389`
**Проблема**: `case PIECEWORK -> template.getBaseSalary()` — сдельная оплата просто возвращает фиксированную сумму из шаблона. В реальности сдельщик получает ставку × выработку (количество единиц).
**Импакт**: Сдельная оплата не считается — подставляется фиксированная сумма.

#### BUG-137: [DATA] Employee.employeeNumber — global UNIQUE constraint
**Файл**: `modules/hr/domain/Employee.java:29`
**Проблема**: `@Index(name = "idx_employee_number", columnList = "employee_number", unique = true)` — табельный номер уникален через ВСЕ организации. Org A и Org B не могут иметь сотрудника №001.
**Импакт**: Multi-tenant коллизия табельных номеров.

#### BUG-138: [SECURITY] LeaveController — createLeaveRequest/submit/cancel БЕЗ @PreAuthorize
**Файл**: `modules/leave/web/LeaveController.java:101-108,110-115,138-143`
**Проблема**: POST /requests, PUT /requests/{id}/submit, PUT /requests/{id}/cancel — без @PreAuthorize. Любой authenticated user (включая VIEWER) может создавать и отменять заявки на отпуск.

#### BUG-139: [UX] PayrollService — nightHours и holidayHours не передаются из calculateFromTimesheetData()
**Файл**: `PayrollService.java:318-319`
**Проблема**: `null, null, // nightHours, holidayHours — надо передать вручную` — при автоматическом расчёте из табеля ночные и праздничные часы не извлекаются. Всегда = 0.
**Импакт**: Расчёт из табеля занижает зарплату (нет доплат за ночные/праздничные).

#### BUG-140: [ARCH] EmploymentContractService.getContractOrThrow() — findById без org filter
**Файл**: `modules/hrRussian/service/EmploymentContractService.java:105-108`
**Проблема**: `contractRepository.findById(id)` — ищет по ID без проверки organizationId. Даже при наличии @Filter на entity, findById(UUID) обходит Hibernate @Filter (filters работают только на коллекционных запросах, не на findById).
**Импакт**: IDOR — зная UUID контракта другой организации, можно получить/удалить/расторгнуть его.

#### BUG-141: [DATA] PayrollService.bulkCalculate — N+1 проблема
**Файл**: `PayrollService.java:267-274`
**Проблема**: Итерирует по списку расчётов в цикле, каждый вызов calculatePayroll() делает: 1 SELECT template, 1 SELECT cumulative income, 1 INSERT calculation. При 100 сотрудниках = 300+ queries в одной транзакции.
**Импакт**: Slow bulk payroll при 100+ сотрудниках.

#### BUG-142: [SECURITY] PayrollService.updateLaborBudgetActual — swallows exceptions
**Файл**: `PayrollService.java:351-353`
**Проблема**: `catch (Exception e) { log.error(...) }` — проглатывает ВСЕ исключения при обновлении бюджета. Если finance module сломался, зарплата утверждается, но бюджет ФОТ не обновляется. Расхождение данных.

#### BUG-143: [CALC] PayrollService — OPS threshold hardcoded 2 225 000
**Файл**: `PayrollService.java:562`
**Проблема**: `final BigDecimal OPS_THRESHOLD = new BigDecimal("2225000")` — порог ОПС 2025 года. С 2026 года порог будет другой (ежегодно меняется постановлением Правительства). Hardcode = ручное обновление каждый год.

#### BUG-144: [DATA] hrRussian — нет печатных форм Т-2, Т-1, Т-5, Т-6, Т-8
**Проблема**: Приказы (PersonnelOrder), личные карточки (PersonalCard), отпуска (Vacation) — все CRUD без генерации унифицированных форм. Для российского кадрового учёта обязательны: Т-1 (приём), Т-2 (личная карточка), Т-5 (перевод), Т-6 (отпуск), Т-8 (увольнение).
**Импакт**: Кадровик не может распечатать обязательные документы.

#### BUG-145: [UX] TimesheetT13Page — ячейки с кодами (Я/Б/О/К) без валидации ввода
**Проблема**: Пользователь может ввести произвольный текст в ячейки табеля Т-13. Нет ограничения по допустимым кодам (ГОСТ Р 7.0.97-2016: Я, Н, В, К, Б, О, ОО, ДО, НН, ПР, Г и т.д.).

- ✅ ст.152 ТК РФ: первые 2 часа × 1.5, далее × 2.0

**Праздничные** (PayrollService.java:448-455):

## 7. Архитектура (взгляд архитектора)

### 7.1 Положительное

- **Employee entity** — reference implementation: PII encryption, @Filter, computed fullName, SNILS validation
- **PayrollService** — enterprise-level: progressive tax, cumulative calculation, budget integration, timesheet integration
- **Status machines**: PayrollCalculationStatus.canTransitionTo(), LeaveRequest FSM, EmploymentContract.isExpired/isOnProbation
- **Audit trail**: все CUD-операции логируются через AuditService
- **Soft delete**: все entities используют BaseEntity.softDelete()

### 7.2 Проблемы

| Проблема | Severity | Описание |
|----------|----------|----------|
| Cross-module imports в PayrollService | P1 | 3 модуля: finance, hr, calendar. Нарушает принцип изоляции |
| 8 entities без tenant isolation | P1 | Recruitment (3), Leave (3), Payroll (2) |
| Employee.employeeNumber global UNIQUE | P2 | Коллизия между тенантами |
| PayrollTemplate.code global check | P2 | existsByCode без org filter |
| findById() обходит @Filter | P2 | EmploymentContractService: IDOR |
| 0 тестов для 4 из 5 субмодулей | P3 | hrRussian, leave, recruitment, payroll |

### 7.3 Cross-module dependency graph

```
payroll ──imports──> finance.domain.BudgetItem
payroll ──imports──> finance.domain.BudgetCategory
payroll ──imports──> finance.repository.BudgetItemRepository
payroll ──imports──> hr.domain.TimesheetStatus
payroll ──imports──> hr.repository.TimesheetRepository
payroll ──imports──> calendar.service.WorkCalendarService
```

**Рекомендация**: Ввести domain events или application service layer:
- Payroll APPROVED → publish `PayrollApprovedEvent` → finance module listens and updates budget
- calculateFromTimesheetData() → вынести в межмодульный application service

### 8.2 Уникальные фичи PRIVOD (чего нет у конкурентов)

1. **Payroll→Budget ФОТ integration** — утверждённая зарплата автоматически отражается в бюджете проекта. Ни один конкурент этого не делает.
2. **Crew timesheets** — табели учёта времени бригад (не только индивидуальные).
3. **Certification Matrix** — визуальная матрица допусков с countdown до истечения.
4. **Seniority+Leave calculator** — расчёт стажа с правом на отпуск.
5. **PII encryption at rest** — AES шифрование персональных данных в PostgreSQL.

### 8.3 Критические gap-ы

1. **Печатные формы** — 0 из 7 обязательных (Т-1, Т-2, Т-3, Т-5, Т-6, Т-8, расчётный листок)
2. **Расчёт больничных/отпускных** — нет расчёта по среднему заработку (ст.139 ТК РФ)
3. **Аванс** — нет расчёта (обязательно по ст.136 ТК РФ)
4. **6-НДФЛ / РСВ / СЗВ-М** — нет генерации отчётных форм для ФНС/ПФР
5. **Электронный кадровый документооборот** — нет ЭЦП на кадровых документах (ФЗ-377)
