## 1. Что сделано хорошо (5 сильных сторон)

### 1.1. Продуманные конечные автоматы (State Machines)
Каждая ключевая сущность (Rfi, Issue, Submittal, ChangeEvent, ChangeOrder, ChangeOrderRequest, ApprovalInstance) имеет метод `canTransitionTo(status)` с явными допустимыми переходами. Это лучше, чем у Procore, где статусы часто hardcoded на фронтенде.

### 1.2. Полная цепочка Change Management
ChangeEvent → ChangeOrderRequest → ChangeOrder → ChangeOrderItem — 4-уровневая иерархия с автоматическим пересчётом revised contract amount. `sumTotalAmountByContractIdAndStatus()` — нативный SQL для точности. Procore не имеет отдельного ChangeEvent.

### 1.3. Workflow Engine с Auto-Approval + SLA + Batch
- AutoApprovalRule с threshold-based auto-approve (сумма < порог → автоматическое утверждение)
- SLA tracking с deadline + isOverdue + escalation
- Batch decision (массовое одобрение/отклонение)
- Delegation + escalation
- Это enterprise-level функциональность.

### 1.4. Cross-Linking между сущностями
- Issue → linkedRfiId, linkedSubmittalId
- ChangeEvent → linkedRfiId, linkedIssueId
- ChangeEvent → ChangeOrderRequest → ChangeOrder
- ApprovalChain → Specification (cascade status)
- Это позволяет отследить полную цепочку: RFI → Issue → Change Event → Change Order.

### 1.5. Аналитика Change Management
`ChangeManagementAnalyticsService` предоставляет 3 аналитических endpoint'а:
- Budget Impact (additions/deductions по типу, monthly trend)
- Schedule Impact (affected WBS nodes, critical path)
- Trends (monthly event/order counts, cumulative cost)

## 2. Найденные баги

### P1 — Critical

#### PROC-B01. [SECURITY] Системная cross-tenant утечка: pmWorkflow (RFI, Issues, PM-Submittals)
**Все** read-операции RfiService, IssueService, PmSubmittalService **не фильтруют по organizationId**. Entity имеет `@Filter(name = "tenantFilter")`, но Hibernate Filter opt-in и нигде не активирован.
- `RfiService.java:42-55` — listRfis Specification без org predicate
- `RfiService.java:222-224` — getRfiOrThrow findById без org check
- `IssueService.java:42-54` — listIssues аналогично
- `IssueService.java:221-224` — getIssueOrThrow аналогично
- `PmSubmittalService.java:50-63` — listSubmittals аналогично
- `PmSubmittalService.java:271-274` — getSubmittalOrThrow аналогично
- **Воздействие**: любой authenticated user видит RFI/Issues/Submittals ВСЕХ организаций

#### PROC-B02. [SECURITY] Системная cross-tenant утечка: changeManagement
- `ChangeOrderService.java:45-47` — listAll() возвращает ВСЕ change orders
- `ChangeOrderService.java:284-287` — getChangeOrderOrThrow findById без org
- `ChangeEventService.java:33-43` — listChangeEvents без org в Specification
- `ChangeOrderRequestService.java:161-164` — getChangeOrderRequestOrThrow findById без org
- **Воздействие**: аналогично PROC-B01

#### PROC-B03. [SECURITY] ChangeOrderRequest entity — ZERO tenant isolation
`ChangeOrderRequest.java` — нет поля `organizationId`, нет `@Filter(name = "tenantFilter")`. Entity полностью не скопирована к tenant. Невозможно добавить фильтрацию без миграции.

#### PROC-B04. [SECURITY] IDOR: Path variable игнорируется при создании sub-resource
3 контроллера передают `request` в сервис, игнорируя `@PathVariable`:
- `RfiController.java:117` — POST `/api/pm/rfis/{rfiId}/responses` → `rfiService.addResponse(request)` использует `request.rfiId()` из body. Атакующий может POST на `/rfis/A/responses` с `rfiId: B` в body → ответ добавляется к чужому RFI.
- `IssueController.java:117` — POST `/api/pm/issues/{issueId}/comments` → аналогично
- `PmSubmittalController.java:165` — POST `/api/pm/submittals/{submittalId}/reviews` → аналогично

#### PROC-B05. [SECURITY] IDOR: ApprovalStep approve/reject без ownership check
`ApprovalService.java:98-143` — `approveStep(stepId, comment)` и `rejectStep(stepId, comment)` находят step по ID и меняют статус. **Нет проверки** что step принадлежит организации вызывающего или что вызывающий — designated approver.

#### PROC-B06. [BUG] ChangeOrderDetailPage status mutation вызывает CREATE вместо UPDATE
`ChangeOrderDetailPage.tsx:56-58` — `statusMutation` вызывает `changeManagementApi.createChangeOrder({ id: orderId, status } as any)`. **Создаёт новый CO** вместо изменения статуса существующего. `as any` скрывает ошибку от TypeScript.

#### PROC-B07. [BUG] ChangeEventDetailPage status mutation вызывает CREATE вместо UPDATE
`ChangeEventDetailPage.tsx:41-43` — Аналогично PROC-B06: `changeManagementApi.createChangeEvent({ id: eventId, status } as any)`.

#### PROC-B08. [STUB] ChangeOrderBoardPage — 100% не функционален
`ChangeOrderBoardPage.tsx:49` — `const [items, setItems] = useState<CoCard[]>([])` — нет useQuery, нет API call. Kanban-доска всегда пуста. Полная заглушка.

## 7. Подробная оценка по ролям

### СМЕТЧИК 👷 (4/10)
- Change Order → revised contract amount вычисляется
- Но: в analytics originalAmount берётся из CO (не контракт), recalculate per-CO а не per-contract
- НДС: VatCalculator 20% (должен быть 22% с 2026)
- Нет автоматического пересчёта сметы при CO

### ПРОРАБ 📋 (5/10)
- RFI/Issues/Submittals — полный lifecycle
- Kanban boards для визуального управления
- Но: RFI board не работает на мобильных (HTML5 DnD)
- CO Board полностью пуст (нет API)
- Нет offline mode
- Нет email уведомлений
- Нет фото-вложений

### БУХГАЛТЕР 💰 (5/10)
- CO → revised contract amount
- Но: CO не автоматически обновляет бюджет (manual trigger)
