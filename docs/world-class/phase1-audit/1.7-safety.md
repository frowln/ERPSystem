## 2. Оценка по ролям

### БЕЗОПАСНИК: 5/10

**Критические проблемы:**

| # | Проблема | Файл | Приоритет |
|---|----------|------|-----------|
| B1 | **15+ GET endpoints без @PreAuthorize** — инциденты, инспекции, метрики, риск-скоринг, СИЗ, СОУТ доступны без RBAC проверки | SafetyIncidentController:42-56,124-139, SafetyRiskScoringController:36-82, SafetyMetricsController:39-45, SafetyPpeController:40-55 | **P2** |
| B2 | **SafetyComplianceService.hasUpcomingTraining()** — ГЛОБАЛЬНАЯ проверка вместо per-employee. Блокирует создание инструктажей если у ЛЮБОГО сотрудника в ЛЮБОЙ организации есть предстоящий тренинг | SafetyComplianceService.java:413-421 | **P1** |
| B3 | **SafetyComplianceService.parseJsonStringArray()** — regex-based JSON parsing (`replaceAll("[\\[\\]\"\\s]", "")`) вместо Jackson ObjectMapper. Ломается на escaped quotes, вложенных объектах, значениях с запятыми | SafetyComplianceService.java:442-459 | **P2** |
| B4 | **SafetySoutController** — class-level `@PreAuthorize("isAuthenticated()")` — только проверка аутентификации, нет RBAC. Любой аутентифицированный user может CRUD СОУТ карты | SafetySoutController.java:28 | **P2** |
| B5 | **safetyChecklist.ts — 100% localStorage** — контрольные списки безопасности (6 категорий, 16 пунктов) хранятся только в браузере. Нет бэкенда, нет tenant isolation, данные теряются при очистке | safetyChecklist.ts:20 | **P2** |

### ПРОРАБ: 7.5/10

**Сильные стороны:**
- 13 навигационных пунктов — все ключевые модули доступны из сайдбара
- Полный workflow инцидентов: REPORTED → UNDER_INVESTIGATION → CORRECTIVE_ACTION → RESOLVED → CLOSED
- IncidentInvestigateWizard (5 шагов): данные → расследование → причины → корректирующие действия → завершение
- Форма Н-1 (AccidentActN1Page) с печатью
- Инструктажи по ГОСТ 12.0.004-2015 (5 типов: вводный, первичный, повторный, внеплановый, целевой)
- Матрица сертификатов (CertificationMatrixPage) — наглядная таблица сотрудник × сертификат
- LTIFR/TRIR метрики на отдельном дашборде

**Проблемы:**
- SafetyBoardPage использует HTML5 DnD (не @dnd-kit) — **не работает на мобильных** (iOS/Android)
- Нет фотофиксации инцидентов (нет upload фото)
- Нет QR-кодов для СИЗ
- Нет оффлайн-режима (стройка без интернета)

### АРХИТЕКТОР: 7.5/10

**Сильные стороны:**
- Правильная tenant isolation через @Filter + SecurityUtils.requireCurrentOrganizationId()
- Audit logging на всех CRUD операциях (auditService.logCreate/logStatusChange/logDelete)
- Soft-delete на всех entity (deleted: bool)
- State machine validation (AccidentActStatus.canTransitionTo())
- Batch queries для избежания N+1 (projectRepository.findNamesByIdsAndOrganizationId)
- JSONB для гибких данных (participants, certificate types, factors)
- 16+ enums с displayName — правильная типизация

**Проблемы:**
- **SafetyRiskScoringService** (~500 LOC) — 30+ hardcoded threshold values (нормализация, веса). Не конфигурируется per-tenant
- **SafetyComplianceService** (461 LOC) — граничит с god-class. Содержит auto-scheduling, compliance checking, access blocking, prescription tracking
- **SafetyTrainingService:72-75** — `ObjectMapper.class.getDeclaredConstructor().newInstance()` (reflection) для парсинга JSON, при этом Spring может inject ObjectMapper
- `countOverdueTrainings()` — STUB, всегда возвращает 0. Training compliance factor в risk scoring = всегда 0 risk

### СМЕТЧИК / БУХГАЛТЕР: N/A (не применимо к модулю safety)

Модуль safety не содержит финансовых расчётов. Единственная численная логика — risk scoring (0-100 баллов), LTIFR/TRIR расчёт. Эти метрики корректны по формулам, но thresholds hardcoded.

### ПРОДАКТ МЕНЕДЖЕР: 7/10

**Что есть (соответствие Gold Standard 1.11):**
- [x] Инструктажи — 5 типов, журнал с подписями (электронными), автоматическое назначение по правилам
