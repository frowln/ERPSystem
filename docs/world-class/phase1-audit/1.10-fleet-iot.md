## 2. Что работает хорошо ✅

### 2.1 Fleet Backend — сильнейший модуль техники
- **МДС 81-3.2001 калькулятор маш.-часа** (MachineHourCalculatorService): 5-компонентная формула (амортизация + ГСМ + ТО + страхование + оператор). Правильные расчёты по российскому стандарту
- **Own vs Rent анализ**: сравнение себестоимости владения с рыночной арендой, рекомендация (OWN/RENT/EQUAL), процент экономии — отличная бизнес-фича
- **Путевой лист по Приказу Минтранса 390/159**: медосмотр, механик, расход по норме, отклонение расхода — грамотная российская специфика
- **Compliance Dashboard**: overdue maintenance + expiring insurance + tech inspection в одном виде — удобно для механика
- **Maintenance Schedule Rules Engine**: правила по моточасам / пробегу / дням с lead-time оповещениями — правильный подход к превентивному ТО
- **Vehicle lifecycle**: AVAILABLE → IN_USE → MAINTENANCE → REPAIR → DECOMMISSIONED — полный жизненный цикл

### 2.2 IoT Equipment — грамотная GPS подсистема
- **Geofence с двумя типами зон**: круговые (Haversine) + полигональные (ray casting) — правильные алгоритмы
- **Batch telemetry ingestion**: принимает массив точек за один запрос — эффективно для IoT-шлюзов
- **Real-time geofence checking**: проверка при каждом ingest — мгновенные алерты о нарушениях
- **Полная tenant isolation** на equipment IoT: @Filter + organizationId на всех 4 таблицах

### 2.3 Frontend — богатый набор страниц
- **21 страница fleet**: от реестра техники до QR-сканера — покрытие функциональности лучше большинства конкурентов
- **WaybillsPage (705 строк)**: полноценный ESM-2 интерфейс с создание → выпуск → выполнение → завершение
- **UsageLogListPage**: 3 вкладки (журнал, калькулятор маш.-часа, Own vs Rent) — всё в одном месте
- **Kanban доска ТО** (FleetMaintenanceBoardPage): drag-and-drop статусов техники
- **QR Scanner**: работает через BarcodeDetector API с fallback на ручной ввод
- **i18n**: ~95% покрытие, все строки через t()
- **Dark mode**: все страницы поддерживают

### 2.4 Архитектура
- **React Query** для серверного состояния, **React Hook Form + Zod** для форм — правильные паттерны
- **Soft deletes** на всех entities — защита от случайного удаления
- **Audit logging** на всех мутациях — хорошая трассируемость
- **RBAC на всех fleet controllers** — ADMIN, FLEET_MANAGER, PROJECT_MANAGER + специализированные роли (OPERATOR, FOREMAN, SAFETY_OFFICER)

## 3. Найденные баги 🐛

### P1 — Critical

#### P1-1: IoT Sensor модуль — ZERO tenant isolation (5 таблиц)
- **Файлы**: `modules/iot/domain/IoTDevice.java`, `IoTSensorData.java`, `IoTAlert.java`, `IoTAlertRule.java`, `IoTDashboard.java`
- **Миграция**: `V48__iot_tables.sql` — таблицы без `organization_id`
- **Сервис**: `IoTDeviceService.java` — ни один метод не фильтрует по organizationId
- **Контроллер**: `IoTController.java` — нет SecurityUtils.requireCurrentOrganizationId()
- **Проблема**: Любой аутентифицированный пользователь с ролью ADMIN или ENGINEER видит ВСЕ IoT-устройства, данные сенсоров и алерты ВСЕХ организаций. Полная утечка данных между тенантами.
- **Контраст**: Equipment IoT (V218) ПРАВИЛЬНО реализован с organization_id на всех 4 таблицах.
- **Оценка ущерба**: Если два клиента используют IoT-сенсоры, один видит данные другого. При наличии конкурирующих строительных компаний — утечка бизнес-информации.

#### P1-2: Frontend↔Backend path mismatch — GPS tracking endpoints НЕ СУЩЕСТВУЮТ
- **Frontend ожидает**: `/fleet/gps/statuses`, `/fleet/gps/vehicles/{id}/track`, `/fleet/gps/geofence-alerts`
- **Backend имеет**: `/api/iot/equipment/devices`, `/api/iot/equipment/devices/{id}/location`, `/api/iot/equipment/alerts`
- **Файлы**: `frontend/src/api/fleet.ts` (getGpsStatuses, getVehicleTrack, getGeofenceAlerts), `backend/.../iot/web/IotEquipmentController.java`
- **Результат**: GpsTrackingPage вызывает несуществующие endpoints → 404 → fallback на localStorage → **пользователь видит пустую карту или мусорные данные из localStorage другой сессии**. Backend driver rating endpoints (`/fleet/drivers/ratings`, `/fleet/drivers/{id}/detail`) тоже не существуют.

#### P1-3: VehicleAssignment + FuelRecord — нет organizationId
- **Файлы**: `modules/fleet/domain/VehicleAssignment.java`, `modules/fleet/domain/FuelRecord.java`
- **Проблема**: Обе entities не имеют поля `organizationId` и не используют `@Filter(tenantFilter)`. Tenant isolation работает ТОЛЬКО косвенно через Vehicle (у которого есть orgId). Но:
  - `FuelRecordRepository.sumCostByProjectId()` — агрегирует по projectId БЕЗ фильтра организации
  - `VehicleAssignmentRepository.findActiveAssignmentsByProject()` — ищет по projectId БЕЗ orgId
  - Если projectId совпадает у двух организаций (UUID collision маловероятно, но repo не защищён архитектурно) — cross-tenant leakage

#### P2-1: iotApi.getAlertRules() — вызывается в AlertsPage, но НЕ определён в api.ts
- **Файл**: `frontend/src/modules/iot/AlertsPage.tsx` → вызывает `iotApi.getAlertRules()`
- **Файл**: `frontend/src/modules/iot/api.ts` → функция getAlertRules() отсутствует

- **Проблема**: Вкладки Assignments, Maintenance, Fuel, Inspections рендерят DataTable с `data=[]`. API-функции `getAssignments()`, `getMaintenanceRecords()`, `getFuelRecords()`, `getInspections()` существуют в fleet.ts, но НЕ вызываются на странице.
- **Для пользователя**: Открыл карточку экскаватора → видит пустые вкладки → думает, что данных нет → на самом деле данные есть, но не загружаются

#### P2-3: GpsTrackingPage — 100% localStorage mock
- **Файл**: `frontend/src/api/fleet.ts` (getGpsStatuses, getVehicleTrack, getGeofenceAlerts)
- **Проблема**: Все 3 GPS-функции возвращают `localStorage.getItem()` fallback. Backend IoT equipment endpoints существуют на другом пути. Пользователь видит либо пустую карту, либо данные от предыдущей dev-сессии.

#### P2-4: DriverRatingPage — backend endpoints не существуют
- **Файл**: `frontend/src/modules/fleet/DriverRatingPage.tsx`
- **Backend**: НЕТ `/fleet/drivers/ratings` и `/fleet/drivers/{id}/detail` контроллеров
- **Проблема**: API делает try/catch с fallback на localStorage. Пользователь видит пустой рейтинг водителей.

#### P2-5: AlertsPage acknowledge — toast-заглушка
- **Файл**: `frontend/src/modules/iot/AlertsPage.tsx`
- **Проблема**: Кнопка "Acknowledge" показывает toast, но НЕ вызывает `iotApi.acknowledgeAlert()`. Пользователь думает что подтвердил алерт, но в базе он остаётся ACTIVE.

#### P2-6: No @Scheduled job для maintenance due detection
- **Файл**: `modules/fleet/service/MaintenanceScheduleService.java`
- **Проблема**: `getDueMaintenanceItems()` выполняется ТОЛЬКО при API-запросе (GET /fleet/maintenance-schedule/due). Нет Quartz/Spring @Scheduled для проактивного обнаружения просроченного ТО. Если пользователь не открывает страницу compliance — он не узнает о просроченном ТО.

#### P2-7: IoT sensor_data — нет стратегии партиционирования/retention
- **Файл**: `V48__iot_tables.sql` → таблица `iot_sensor_data`
- **Проблема**: Time-series данные растут неограниченно. На стройке с 50 датчиками и 5-секундным интервалом → 864,000 записей/день → 26M/месяц. Без партиционирования или cleanup — деградация производительности и заполнение диска.

#### P2-8: Geofence violation logic — ложные EXITED алерты для STORAGE зон
- **Файл**: `modules/iot/service/IotService.java`, метод `checkGeofenceViolations()`
- **Логика**: `if (!isInside && zone.getZoneType() != RESTRICTED) → EXITED alert`
- **Проблема**: Для зоны типа STORAGE (склад) — любой выезд техники с территории склада генерирует EXITED алерт. Это нежелательное поведение: техника ДОЛЖНА покидать склад для работы. EXITED алерт логичен только для SITE (стройплощадка).

#### P2-9: GPS Timesheet (V219) — 4 таблицы без контроллера и сервиса в IoT
- **Миграция**: `V219__gps_timesheets.sql` создаёт 4 таблицы (site_geofences, gps_check_events, gps_timesheet_entries, gps_timesheet_summaries)
- **Код**: Существует отдельный модуль `gpsTimesheet` (domain + repo + service + controller), но он НЕ интегрирован с IoT equipment модулем. Два параллельных geofence-системы: одна для техники (IoT), другая для HR-табелей (gpsTimesheet) — дублирование логики.

#### P2-10: Hardcoded pagination `size: 200, page: 0` на большинстве fleet страниц
- **Файлы**: FleetListPage, MaintenancePage, FuelPage, WaybillListPage и др.
- **Проблема**: Загружает до 200 записей за раз без серверной пагинации. При 500+ единицах техники — медленная загрузка и высокое потребление памяти.

## 8. Расчёты (глазами сметчика) 🧮

### 8.1 Маш.-час (МДС 81-3.2001) — ✅ Корректно
```
hourlyRate = depreciation/h + fuel/h + maintenance/h + insurance/h + operator/h
```
- Амортизация: `(purchasePrice / usefulLifeYears) / annualWorkingHours` — правильно
- ГСМ: `fuelConsumptionRate × fuelPrice` — правильно (л/час × руб/л)
- Все компоненты BigDecimal с HALF_UP rounding, scale 2 — корректно для финансов
- **Замечание**: `annualWorkingHours` по умолчанию 1800ч/год. МДС 81-3.2001 рекомендует для разных типов техники разные нормативы (1500-2400). Можно добавить справочник.
- **Замечание**: `fuelPrice` по умолчанию 60 руб/л (hardcode в service). Цена дизеля в 2026 ≈ 70+ руб/л. Нужно брать актуальную цену.

### 8.2 Own vs Rent — ✅ Корректно
- Формулы правильные: ownRatePerHour vs marketRentalRatePerHour
- savingsPercent правильно считает от rent cost
- Рекомендация OWN/RENT/EQUAL — логика корректна

### 8.3 Расход по путевому листу — ✅ Корректно
```
fuelNorm = distance × fuelConsumptionRate / 100
fuelVariancePercent = ((fuelConsumed - fuelNorm) / fuelNorm) × 100
```
- Формула Минтранс 390 — правильная
- **Замечание**: Не учитывает коэффициенты (зимний +5-20%, городской +5-10%, горный +5-20%). Для стройки нужен хотя бы зимний коэффициент.

### 8.4 Амортизация — ✅ Корректно
```
annualDepreciation = purchasePrice × depreciationRate / 100
currentValue = purchasePrice - totalDepreciation (min 0)
```
- Линейный метод — самый распространённый в строительстве
- **Замечание**: Нет ускоренного метода (коэффициент 2-3 для работы в агрессивных условиях) и нелинейного метода

### 8.5 Стоимость назначения — ⚠️ Упрощено
```
totalCost = dailyRate × ChronoUnit.DAYS.between(startDate, now) + 1
```
- Считает календарные дни, не рабочие. Если техника назначена на месяц — оплата за 30 дней, а не за 22 рабочих.
