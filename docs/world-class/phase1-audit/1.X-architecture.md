# 1.X — Архитектура, инфраструктура, безопасность, production-readiness

> **Дата**: 2026-03-18
> **Общая оценка**: **6.0/10** — "Крепкий фундамент, критические пробелы в tenant isolation и secrets management"
> **Критичных багов**: 12 P1, 18 P2, 14 P3

---

## РЕЗЮМЕ

Платформа PRIVOD имеет архитектурный фундамент enterprise-уровня: модульный монолит Spring Boot с 3-слойной tenant isolation, JWT+TOTP 2FA, HikariCP, партиционирование таблиц, полноценный DR-план. Однако **53% entities (324/608) не имеют organizationId**, MinIO/Redis без tenant namespace, PostgreSQL RLS отсутствует, секреты в git, 6 мест SQL injection в портале.

## 1. МУЛЬТИТЕНАНТНОСТЬ

### 1.1. Реализация изоляции данных

**Архитектура**: 3-слойная модель (лучше чем у 80% SaaS):

| Слой | Механизм | Файл | Покрытие |
|------|----------|------|----------|
| 1. Service | `SecurityUtils.requireCurrentOrganizationId()` | `SecurityUtils.java:45-50` | ~70% сервисов |
| 2. Repository | Explicit `findByOrganizationId*` queries | Повсеместно | ~60% repos |
| 3. ORM | Hibernate `@Filter("tenantFilter")` | `package-info.java:10-14` | **284/608 entities (47%)** |

**TenantFilterInterceptor** (`TenantFilterInterceptor.java:43-47`):
- Перехватывает ВСЕ `/api/**` запросы (кроме `/api/auth/**`, `/api/health/**`, `/api/payments/webhook/**`)
- Извлекает `organizationId` из JWT через `SecurityUtils`
- Включает Hibernate filter `tenantFilter` с параметром `organizationId`
- **Регистрация**: `WebConfig.java:33-39`

### 1.2. Критические пробелы

**324 entities (53%) БЕЗ organizationId / @Filter:**

| Категория | Кол-во | Риск | Примеры |
|-----------|--------|------|---------|
| Системные (Organization, Role, Permission) | ~20 | NONE | Ожидаемо |
| Дочерние (через parent filtering) | ~80 | LOW | JournalLine, DocumentRevision, TransmittalItem |
| **Бизнес-entities БЕЗ tenant isolation** | **~120+** | **CRITICAL** | IoT Sensor (5 таблиц), BIM (11 entities), Leave (3), Payroll (2), Recruitment (3), Design (3), Portal (3), Dashboard/KPI (3), Maintenance (5), Email (1) |
| Settings/Config | ~50 | MEDIUM | IntegrationConfig, WebhookConfig, FeatureFlag |
| Audit/Logging | ~30 | MEDIUM | AuditLog, ActivityLog (нужен orgId для tenant-scoped audit trail) |

**Вердикт**: organizationId отсутствует у ~120 бизнес-entities — это **системная уязвимость P1**. Данные одного тенанта видны другому через прямые GET-запросы.

### 1.3. MinIO (файловое хранилище)

**Текущая реализация** (`S3StorageService.java:47-57`):
- Единый bucket `privod-uploads` для ВСЕХ тенантов
- Ключ файла: `{keyPrefix}/{UUID}.{extension}` (например `attachments/abc-123.pdf`)
- **НЕТ tenant prefix** в пути

**Проблема**: Знание UUID чужого файла = доступ к файлу. Защита только на уровне `FileAttachment` entity (имеет @Filter), но прямой доступ к MinIO API обходит это.

**Рекомендация**: Изменить формат ключа на `{keyPrefix}/{organizationId}/{UUID}.{extension}`

### 1.4. Redis (кэш)

**Текущая реализация** (`CacheConfig.java:14-20`):
- **Не Redis!** Используется `ConcurrentMapCacheManager` (in-memory)
- Кэши: `permissionModelAccess`, `permissionFieldAccess`, `permissionRecordFilter`
- **НЕТ tenant namespace** в ключах кэша
- Redis используется ТОЛЬКО для WebSocket broadcast (`RedisWebSocketBroadcast.java:36`) и rate limiting

**Проблема**: Кэш прав доступа shared между тенантами. User с ролью ADMIN в Tenant A может получить cached permissions от ADMIN в Tenant B.

**Рекомендация**: Добавить `organizationId` в ключи кэша: `permissionModelAccess:{orgId}:{roleId}`

### 1.5. PostgreSQL RLS

**Статус**: **НЕ РЕАЛИЗОВАН**

Ни одной `CREATE POLICY` или `ALTER TABLE ... ENABLE ROW LEVEL SECURITY` во всех 300+ миграциях. Это означает:
- Если Hibernate filter не активирован (баг, прямой SQL, JDBC query) — данные утекают
- Native queries в PortalDataProxyController **обходят** @Filter (6 мест SQL injection!)

**Рекомендация**: Добавить RLS как 4-й слой защиты:
```sql
ALTER TABLE projects ENABLE ROW LEVEL SECURITY;
CREATE POLICY tenant_isolation ON projects
  USING (organization_id = current_setting('app.current_tenant')::uuid);
```

### 1.6. Тесты tenant isolation

**Файл**: `TenantIsolationTest.java` (707 строк, 16 тест-кейсов)
- Project, Task, Budget, Estimate, User, Dashboard, Cross-Reference, UUID Enumeration
- **Покрытие**: ~8 модулей из ~85 (10%)
- **Не покрыты**: BIM, IoT, Leave, Payroll, Recruitment, Portal, Maintenance, Design, Workflow

**CORS проблема** (`SecurityConfig.java:105`, `WebConfig.java:26-27`):
```java
configuration.setAllowedHeaders(List.of("*"));   // ← wildcard

```
RFC 6454: `allowedHeaders(*)` + `credentials(true)` позволяет атакующему инжектировать произвольные заголовки в cross-origin запросах.
