## НАЙДЕННЫЕ БАГИ

### P1 — Critical

#### BUG-E1: searchNormativeRates() возвращает HARDCODED mock данные — не реальные расценки
**Суть**: Метод `EstimateAdvancedService.searchNormativeRates()` (строка 501-531) **всегда** возвращает одни и те же 3 захардкоженные расценки (ГЭСН/ФЕР/ТЕР 06-01-001-01 «Устройство бетонных фундаментов»), независимо от поискового запроса. Поиск по реальной нормативной базе **не реализован** — stub.

**Файл**: `backend/src/main/java/com/privod/platform/modules/estimate/service/EstimateAdvancedService.java:508-531`
```java
// Stub: return placeholder normative rate data
results.add(new NormativeRateSearchResponse(
    "ГЭСН 06-01-001-01", "Устройство бетонных фундаментов", ...));
```

**Влияние**: Сметчик получает фейковые расценки. Любая смета на основе «поиска по нормативной базе» содержит неправильные цены. **Критическая бизнес-функция полностью не работает.**

**Приоритет**: P1 — сметчик не может работать с системой.

**Файл**: `backend/src/main/java/com/privod/platform/modules/estimate/service/EstimateAdvancedService.java:537-553`

```java
// Stub: return realistic result
```

**Влияние**: Пользователь загружает ЛСР, видит «Импорт завершён», но данные **потеряны**. Смета не создаётся.

**Приоритет**: P1 — потеря данных пользователя.

#### BUG-E5: НДС в LocalEstimate берётся из VatCalculator.DEFAULT_RATE = 20%, а не 22%
**Суть**: `LocalEstimate.vatRate` инициализируется через `VatCalculator.DEFAULT_RATE`, который до сих пор = `20.00` (баг BUG-F1 из аудита finance). Все новые локальные сметы считаются с НДС 20% вместо 22%. Тест `LocalEstimateServiceTest` (строка 82) тоже использует `20.00`.

**Файл**: `backend/src/main/java/com/privod/platform/modules/estimate/domain/LocalEstimate.java:88`
```java
private BigDecimal vatRate = VatCalculator.DEFAULT_RATE; // 20.00, а нужно 22.00
```

**Влияние**: Итоговая сумма сметы с НДС занижена на ~1.6% ((122-120)/120). На смете 100M руб — ошибка ~1.6M руб.

**Приоритет**: P1 — финансовая ошибка.

**Влияние**: Пользователь получает «зелёную галочку» на невалидной смете.

#### BUG-E8: EstimateService.listEstimates() — нет tenant isolation
**Файл**: `EstimateService.java:58-75`

В Criteria Query нет предиката `organizationId = currentTenantId`. Хотя `@Filter(tenantFilter)` аннотирован на entity, он работает только если активирован в `Session`. **Если Hibernate filter не активирован** (зависит от конфигурации — нужно проверить), данные одного тенанта видны другому.

В отличие от `LocalEstimateService.listEstimates()` (строка 135), который **явно** фильтрует по `organizationId`, `EstimateService` полагается только на `@Filter` аннотацию.

**Влияние**: Потенциальная cross-tenant утечка данных.

---

#### BUG-E9: CompetitiveListService.createFromPurchaseRequest() — UUID.randomUUID() fallback для specificationId
**Файл**: `CompetitiveListService.java:94`

1. **Конкурентные листы (КЛ)** — лучше конкурентов: weighted scoring (50/25/15/10), auto-ranking, min proposals validation, VE integration, apply to КП

3. **ARPS XML import** — полная реализация с XXE prevention, multi-format tag support, section/position hierarchy

5. **Versioning** — JSON snapshots для Estimate, doc versions для Specification
6. **Architecture** — clean layered design, DTO separation, audit trail на всех операциях
