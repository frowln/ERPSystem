## 1. КЛЮЧЕВАЯ НАХОДКА: Две конкурирующие системы согласования

### Система A: `approval` module (ApprovalChain + ApprovalStep)
- **Миграции**: V1058 (chains), V1151 (SLA на steps)
- **Entity**: ApprovalChain (organizationId, entityType, entityId, status String), ApprovalStep (stepOrder, approverName/Role, status String, SLA, @Version)
- **Паттерн**: Простая линейная цепочка. Статусы — raw Strings (`"PENDING"`, `"APPROVED"`, `"REJECTED"`), не enum
- **Cascade**: Работает ТОЛЬКО с Specification (hardcoded в `cascadeEntityStatus()`)
- **Используется**: ApprovalTimeline component, SpecificationDetailPage, approvalsApi.ts (85 LOC)

### Система B: `workflowEngine` module (ApprovalInstance + ApprovalDecision + WorkflowDefinition + WorkflowStep)
- **Миграции**: V56 (definitions, steps, transitions, automation), V75 (auto-approval rules), V143 (approval instances, decisions)
- **Entity**: ApprovalInstance (organizationId, workflowDefinitionId, currentStepId, status Enum, SLA, escalation, delegation), ApprovalDecision (approverId, decision Enum, comments)
- **Паттерн**: Полноценный workflow engine. Статусы — enum (IN_PROGRESS, APPROVED, REJECTED, CANCELLED, ESCALATED). Поддерживает SLA, делегирование, эскалацию, batch decisions, notifications
- **Используется**: ApprovalInboxPage, WorkflowDesignerPage, workflowApi (api.ts, 107 LOC)

### Проблема: Архитектурный долг

| Критерий | Система A (approval) | Система B (workflowEngine) |
|----------|---------------------|---------------------------|
| Статусы | String literals | Proper Enums |
| Tenant isolation | @Filter на Chain, НЕТ на Step | @Filter на Instance |
| SLA | slaHours + deadline на Step | slaDeadline на Instance + auto-escalation |
| Делегирование | ❌ | ✅ delegatedTo |
| Эскалация | escalatedAt поле, нет логики | ✅ escalateOverdue() scheduler |
| Batch operations | ❌ | ✅ batchDecision() |
| Notifications | ❌ | ✅ NotificationService |
| Audit logging | ❌ | ✅ AuditService |
| Cascade to entity | Только Specification | Нет (TODO) |
| Optimistic locking | ✅ @Version | ❌ |
| Controller auth | @PreAuthorize (ADMIN, PM, FM) | @PreAuthorize class-level (ADMIN, PM, CM, FM) |
| Frontend integration | ApprovalTimeline (436 LOC) | ApprovalInboxPage (868 LOC) |
| Entity types supported | SPECIFICATION only (hardcoded) | CONTRACT, INVOICE, PO, PR, CO, PAYMENT, BUDGET, DOCUMENT |

**Вывод**: Система B (workflowEngine) — более зрелая и функциональная. Система A (approval) — legacy, привязана только к Specification. Нужна миграция на единую систему.

## 2. Что сделано хорошо (7 сильных сторон)

- Entity type color-coding (CONTRACT=blue, INVOICE=emerald, PURCHASE_REQUEST=violet, etc.)
- Metric cards: pending, overdue, approved, avg wait time
- Sort: overdue first → SLA deadline soonest → created date
- Full dark mode support
- i18n: ВСЁ через `tp()` helper (100+ keys)
- Cancel with confirmation dialog
- Fallback to per-item requests if batch API fails

### 2.3. AutoApprovalRuleService — образцовый tenant isolation
- `findAll()`: JPA Specification с 6 predicate фильтрами (notDeleted, entityType, isActive, organization, project, search)
- `getRuleOrThrow()`: явная проверка organizationId с "not found" для cross-tenant (не "access denied" — не выдаёт существование)

- Audit logging на все CRUD операции

### 2.4. ConditionBuilder — визуальный условный конструктор (393 LOC)

- Дублирование групп, добавление/удаление условий
- Полная i18n (40+ keys)
- Полный dark mode

### 2.5. WorkflowDesignerPage — визуальный конструктор маршрутов (391 LOC)
- Create/edit workflow definitions с name, entityType, description

- Visual flow: step cards с connector arrows

`ApprovalInboxPage.tsx:198-214` — если batch API (`batchDecision`) вернёт ошибку, fallback на `Promise.allSettled()` с per-item requests. Graceful degradation — лучший pattern в проекте.

## 2. Найденные баги

### P1 — Critical

#### WF-B01. [SECURITY] ApprovalService.approveStep/rejectStep — ZERO ownership check
`ApprovalService.java:98-143` — `approveStep(stepId, comment)` и `rejectStep(stepId, comment)` делают `stepRepository.findById(stepId)` без ЛЮБОЙ проверки:
1. Нет проверки что step принадлежит организации вызывающего
2. Нет проверки что вызывающий — designated approver
3. ApprovalStep НЕ наследует BaseEntity → нет `organizationId`, нет `@Filter`
4. Любой пользователь с ролью ADMIN/PM/FM может approve/reject ЛЮБОЙ step ЛЮБОЙ организации по UUID

**Пример атаки**: POST `/api/approvals/steps/{victim-step-uuid}/approve` → step утверждён → chain пересчитывается → Specification меняет статус на APPROVED.

**Impact**: Cross-tenant approval manipulation. Одна компания может утверждать спецификации другой.

#### WF-B02. [SECURITY] ApprovalService.getChainById — cross-tenant access
`ApprovalService.java:91-95` — `getChainById(id)` вызывает `chainRepository.findByIdAndDeletedFalse(id)` БЕЗ проверки organizationId. Хотя ApprovalChain имеет organizationId и @Filter, findById обходит @Filter (filter opt-in, не active by default).

**Impact**: Любой authenticated user может просмотреть approval chain любой организации.

#### WF-B03. [SECURITY] WorkflowDefinitionService CRUD — cross-tenant manipulation
`WorkflowDefinitionService.java:50-112` — `findById()`, `update()`, `delete()`, `toggleActive()`, `getSteps()`, `replaceSteps()` — все делают raw `findById()` без проверки organizationId.

- `findAll()` (line 37-39) — **правильно** фильтрует по org
- Но `findById(id)` (line 50-57) — **НЕ проверяет** org → кросс-тенант чтение
- `update(id, request)` (line 78-92) — **НЕ проверяет** org → кросс-тенант изменение
- `delete(id)` (line 94-100) — **НЕ проверяет** org → кросс-тенант удаление
- `toggleActive(id)` (line 102-112) — **НЕ проверяет** org → кросс-тенант активация
- `getSteps(id)` (line 117-123) — **НЕ проверяет** org, WorkflowStep НЕ имеет organizationId
- `replaceSteps(id, requests)` (line 125-153) — **НЕ проверяет** org → кросс-тенант замена шагов

**Impact**: Workflow definitions можно читать, изменять, удалять, активировать/деактивировать через UUID guessing. Атакующий может активировать workflow в чужой организации → все начатые approvals пойдут по его маршруту.

#### WF-B04. [SECURITY] AutomationExecution findAll без org filter
`WorkflowDefinitionService.java:158-166` — когда `ruleId == null`, вызывает `automationExecutionRepository.findAll(pageable)`. AutomationExecution НЕ имеет organizationId → все executions всех организаций в ответе.

**Impact**: Information leakage — видно какие автоматизации запускались во всех организациях.

#### WF-B05. [SECURITY] ApprovalInstanceService.submitDecision — нет проверки approver identity
`ApprovalInstanceService.java:192-290` — `submitDecision()` проверяет что instance принадлежит org и что статус IN_PROGRESS, но **НЕ проверяет** что вызывающий userId — designated approver для текущего шага. Любой пользователь с ролью PM/FM/CM может approve/reject любой approval instance своей организации.

Метод `isApproverForCurrentStep()` (line 498-532) используется ТОЛЬКО в `getPendingForUser()` для фильтрации inbox. Но `submitDecision()` его НЕ вызывает.

**Impact**: Менеджер может утвердить документ на этапе, предназначенном для бухгалтера. Нарушение принципа разделения полномочий (segregation of duties).

#### WF-B06. [SECURITY] ApprovalInstance.getHistory — cross-tenant data leak
`ApprovalInstanceService.java:338-350` — `getHistory(entityId, entityType)` вызывает `findByEntityIdAndEntityTypeAndDeletedFalse()` без проверки organizationId. Все approval instances для entity (включая другие организации) возвращаются.

#### WF-B15. [PERF] N+1 в ApprovalInstanceService.getPendingForUser

1. `resolveWorkflowName(workflowDefinitionId)` → findById

20 definitions = 21 query. Steps загружаются полностью (не COUNT).

## 4. Проблемы безопасности (summary)

| # | Тип | Описание | Приоритет |
