## 2. АРХИТЕКТУРА — оценка 6.5/10

### Сильные стороны
- **Правильная модульная структура**: каждый подмодуль изолирован (domain/repository/service/web/dto)
- **State machines**: LeaveRequest (5 статусов), PayrollCalculation (4 статуса), CompletionAct (4 статуса), SelfEmployedPayment (5 статусов) — все с canTransitionTo() методами
- **Soft-delete паттерн** во всех entities
- **Audit logging** через AuditService во всех CRUD операциях
- **Optimistic locking** (`version` field) на контрактных entities
- **@PreAuthorize** на write endpoints

### Проблемы

#### GOD-CLASSES:
- `SelfEmployedService.java` — 675 LOC, управляет 5 entity типами (worker, contractor, payment, registry, act). Должен быть разбит минимум на 3 сервиса
- `PayrollService.java` — ~600 LOC, содержит расчёт зарплаты + бюджетную интеграцию + timesheet интеграцию

#### ДУБЛИРОВАНИЕ:
- **SelfEmployed**: два параллельных entity для одной концепции:
  - `SelfEmployedContractor` (V79, без tenant isolation) — глобальный реестр
  - `SelfEmployedWorker` (V1134, с tenant isolation) — tenant-scoped worker
  - Нет связи между ними, разные API endpoints, разные DTO
  - Пользователь должен работать с обоими — UX confusion

#### CROSS-MODULE ЗАВИСИМОСТИ:
- `PayrollService` импортирует `TimesheetRepository` и `BudgetItemRepository` — нарушение модульной изоляции
- Правильный вариант: event-based или facade interface

## 3. БЕЗОПАСНОСТЬ — оценка 4.5/10 (КРИТИЧНО)

### P1 — Tenant Isolation ПОЛНОСТЬЮ ОТСУТСТВУЕТ в 3 из 4 модулей

#### BUG-HR-01: Leave — НЕТ organizationId в entities [P1] [SECURITY]
- **LeaveType, LeaveRequest, LeaveAllocation** — нет поля `organizationId`, нет `@Filter(tenantFilter)`
- V1138 добавляет org_id к sick_leaves/vacations, но **НЕ** к leave_types/leave_requests/leave_allocations
- Все repository queries возвращают данные ВСЕХ организаций
- **Файл**: `LeaveType.java`, `LeaveRequest.java`, `LeaveAllocation.java`
- **Impact**: cross-tenant data leak — отпуска сотрудников одной компании видны другой

#### BUG-HR-02: Payroll — НЕТ organizationId в entities [P1] [SECURITY]
- **PayrollTemplate** и **PayrollCalculation** — нет `organizationId`, нет `@Filter`
- `findByDeletedFalse()` возвращает шаблоны и расчёты ВСЕХ организаций
- **Файл**: `PayrollTemplate.java`, `PayrollCalculation.java`, `PayrollTemplateRepository.java`
- **Impact**: зарплатные данные (оклады, НДФЛ, ФОТ) видны между тенантами — максимальный риск утечки PII

#### BUG-HR-03: Recruitment — НЕТ organizationId в entities [P1] [SECURITY]
- **Applicant, JobPosition, Interview, RecruitmentStage** — нет `organizationId`, нет `@Filter`
- GET `/v1/recruitment/applicants` возвращает кандидатов ВСЕХ организаций (с PII: ФИО, email, телефон, ожидаемая зарплата)
- **Файл**: `Applicant.java`, `JobPosition.java`, `Interview.java`, `ApplicantRepository.java`
- **Impact**: PII кандидатов утекает между тенантами + нарушение 152-ФЗ

#### BUG-HR-04: SelfEmployed Contractor — НЕТ tenant isolation [P1] [SECURITY]
- `SelfEmployedContractor` (V79) не имеет organizationId и @Filter
- `SelfEmployedPayment` не имеет @Filter (extends BaseEntity)
- `SelfEmployedRegistry` не имеет @Filter
- **Частично** исправлено: `SelfEmployedWorker` и `CompletionAct` ИМЕЮТ organizationId + @Filter
- **Файл**: `SelfEmployedContractor.java`, `SelfEmployedPayment.java`, `SelfEmployedRegistry.java`
- **Impact**: подрядчики, платежи и реестры видны между тенантами

### P1 — Unauthenticated Read Access to PII

#### BUG-HR-05: 28+ GET endpoints без @PreAuthorize [P1] [SECURITY]
Все GET endpoints в 4 модулях доступны любому аутентифицированному пользователю (включая VIEWER):

| Контроллер | GET endpoints без @PreAuthorize | Данные |
|------------|-------------------------------|--------|
| LeaveController | 7 (types×2, requests×2, allocations×3, balances×1) | отпуска сотрудников |
| PayrollController | 4 (templates×2, calculations×2) | зарплаты, НДФЛ |
| SelfEmployedController | 7 (contractors×2, payments×2, registries×3) | ИНН, банк. реквизиты |
| RecruitmentController | 10 (applicants×2, vacancies×3, stages×1, interviews×4) | PII кандидатов |

- **Файл**: `LeaveController.java`, `PayrollController.java`, `SelfEmployedController.java`, `RecruitmentController.java`
- **Impact**: VIEWER может видеть зарплаты, кандидатов, подрядчиков — нарушение RBAC

### P1 — IDOR / Authorization Bypass

#### BUG-HR-06: Leave approverId from query parameter [P1] [SECURITY]
- `PUT /requests/{id}/approve?approverId=XXX` — approverId передаётся как query param
- Нет проверки что текущий пользователь = approver
- Нет проверки что approver является руководителем сотрудника
- User может approve свой собственный запрос на отпуск
- **Файл**: `LeaveController.java` approve/refuse endpoints

#### BUG-HR-07: Leave createRequest — no employee ownership check [P1] [SECURITY]
- `POST /requests` принимает любой `employeeId` без проверки, что текущий пользователь = этот сотрудник или его руководитель
- User может создать заявку на отпуск от имени другого сотрудника
- **Файл**: `LeaveService.java:createLeaveRequest()`

### P2 — Security Issues

#### BUG-HR-08: Recruitment PII хранится plain text [P2] [SECURITY]
- applicants: partner_name, email, phone, salary — VARCHAR, не зашифровано
- SelfEmployedWorker: V1171 расширяет INN/phone/email до TEXT для будущего шифрования, но шифрование не реализовано
- **Impact**: утечка БД = утечка всех PII → штраф по 152-ФЗ
- **Файл**: `V68__recruitment_leave_tables.sql`, `Applicant.java`

#### BUG-HR-09: SelfEmployed INN validation — только формат, нет контрольной суммы [P2] [VALIDATION]
- ИНН валидируется как 12 цифр, но нет проверки контрольной суммы по алгоритму ИФНС
- Можно ввести 123456789012 — система примет
- **Файл**: `SelfEmployedService.java:verifyNpd()`, `CreateContractorRequest.java`

#### BUG-HR-10: Payroll — NPE risk в calculateBasePay() [P2] [BUG]
- `workCalendarService.getNormativeHoursForMonth(year, month)` может вернуть null
- `normHours.compareTo(BigDecimal.ZERO)` → NPE
- **Файл**: `PayrollService.java:370`

#### BUG-HR-11: LeaveType validity period не проверяется [P2] [LOGIC]
- `validityStart`, `validityEnd` хранятся но НЕ валидируются при createLeaveRequest
- Сотрудник может запросить отпуск по expired leave type
- **Файл**: `LeaveService.java:createLeaveRequest()`

#### BUG-HR-12: Recruitment — нет state machine validation [P2] [LOGIC]
- `updateApplicantStatus()` разрешает переход из ЛЮБОГО статуса в ЛЮБОЙ
- NEW → WON без собеседования — нарушение процесса
- **Файл**: `RecruitmentService.java:updateApplicantStatus()`

#### BUG-HR-13: Payroll — race condition при approve [P2] [DATA]
- `checkLeaveBalance()` → `updateAllocationUsedDays()` — не атомарная операция
- Два одновременных approve могут превысить лимит отпуска
- **Файл**: `LeaveService.java:approveLeaveRequest()`

#### BUG-HR-14: Payroll frontend↔backend type mismatch [P2] [API]
- Frontend: `PayrollTemplateType = 'SALARY' | 'HOURLY' | 'PIECE_RATE' | 'MIXED'`
- Backend: `PayrollType = SALARY | HOURLY | PIECEWORK | BONUS | OVERTIME`
- Frontend fields `nightShiftMultiplier`, `holidayMultiplier`, `regionalCoefficient`, `northernAllowance` не существуют в backend
- **Файл**: `frontend/src/modules/payroll/types.ts` vs `PayrollType.java`

| Отпуска | Запросы + allocations | Полный расчёт + табель | Нет расчёта отпускных |
| Самозанятые | Реестр + акты | Нет (отдельный модуль) | PRIVOD лучше |

### vs PlanRadar / Procore
| Функция | PRIVOD | PlanRadar | Procore |
|---------|--------|-----------|---------|

**PRIVOD имеет уникальное преимущество**: встроенный HR + самозанятые + рекрутинг для строительных компаний, чего нет у западных конкурентов (Procore, PlanRadar, Autodesk Build).
