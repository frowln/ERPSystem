## 3. Найденные баги

### P1 — Critical

#### BUG-124: Дублирование BidPackage entity — 2 JPA класса маппят одну таблицу
**Файлы**:
- `modules/portfolio/domain/BidPackage.java:32` — без @Filter, без organizationId
- `modules/bidManagement/domain/BidPackage.java:32` — с @Filter, с organizationId

**Проблема**: Два JPA entity (`BidPackage` и `BidMgmtBidPackage`) маппят таблицу `bid_packages`. Portfolio-версия НЕ ИМЕЕТ tenant isolation (нет organizationId поля, нет @Filter). При использовании через PortfolioService, tenant isolation обеспечивается косвенно через Opportunity check, но при прямом запросе через репозиторий — cross-tenant access.

**Миграционный конфликт**: V55 создаёт таблицу, V1059 использует `CREATE TABLE IF NOT EXISTS` (no-op), V1121 добавляет недостающие колонки задним числом.

**Риск**: Confusion для разработчиков, двойная attack surface, потенциальный data corruption при одновременном доступе через оба модуля.

---

#### BUG-125: BidInvitation — НЕТ organizationId, НЕТ @Filter
**Файл**: `modules/bidManagement/domain/BidInvitation.java:29`

**Проблема**: Entity не имеет поля `organization_id` и аннотации `@Filter`. Tenant isolation обеспечивается только через `ensurePackageAccessible()` в service — проверяет что BidPackage принадлежит org. Но `invitationRepo.findByIdAndDeletedFalse(invId)` (BidManagementService:151) не проверяет что invitation принадлежит тому же package.

**Атака**: Злоумышленник знающий invId из другой организации может обновить/удалить чужое приглашение, передав свой packageId.

**Пример**: `PUT /api/bid-packages/{myPackageId}/invitations/{otherOrgInvitationId}` — `ensurePackageAccessible(myPackageId)` пройдёт, а `invitationRepo.findByIdAndDeletedFalse(otherOrgInvitationId)` вернёт чужое приглашение.

---

#### BUG-126: BidEvaluation — НЕТ organizationId, НЕТ @Filter
**Файл**: `modules/bidManagement/domain/BidEvaluation.java:27`

**Проблема**: Аналогично BidInvitation — нет tenant isolation на уровне entity. `evaluationRepo.findByBidPackageIdAndDeletedFalse(packageId)` фильтрует по packageId, но при получении evaluation по ID прямой запрос не проверит организацию.

---

#### BUG-127: BidManagement listPackages — cross-tenant при фильтре по projectId
**Файл**: `modules/bidManagement/service/BidManagementService.java:51-63`

```java
if (projectId != null) {
    packages = packageRepo.findByProjectIdAndDeletedFalseOrderByCreatedAtDesc(projectId);
} else {
    packages = packageRepo.findByOrganizationIdAndDeletedFalseOrderByCreatedAtDesc(orgId);
}
return packages.stream()
        .filter(p -> p.getOrganizationId().equals(orgId))  // in-memory filter!
        .map(...)
```

**Проблема**: Когда `projectId != null`, загружаются ВСЕ пакеты этого проекта из БД (включая чужие org), затем фильтруются in-memory. Если projectId валиден, но принадлежит другой организации — данные загрузятся в RAM перед фильтрацией. Information leakage через timing attack.

**Исправление**: Использовать `findByProjectIdAndOrganizationIdAndDeletedFalse(projectId, orgId)`.

## 10. Уникальные фичи (лучше конкурентов)

1. **Go/No-Go Scorecard** (CRM + Portfolio) — 15-критерийная модель с весами (opportunity 35%, win 35%, delivery 30%). Ни Procore, ни PlanRadar не имеют встроенного Go/No-Go scoring.

2. **Leveling Matrix** (BidManagement) — weighted multi-criteria vendor evaluation с normalized scores. Реальный bid leveling tool, обычно доступный только в standalone procurement software.

3. **Pricing Chain Integration** (CP → Budget) — `pushToFinancialModel()`, `applyBidWinner()`, `selectPriceFromCL()` — полная цепочка от тендера до бюджета. Уникально для российского рынка.

4. **DaData/Checko Integration** — автопроверка контрагента по ИНН (банкротство, санкции, дисквалификация, mass directors, illegal financing). Лучше чем manual checks у конкурентов.

5. **Lead → Counterparty → КП → Contract chain** — полная цепочка конвертации с audit trail на каждом шаге.
