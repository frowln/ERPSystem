## 1. Архитектурный обзор

### 1.1 Что входит в аудит

Модуль «Документы» — один из самых масштабных в системе, объединяющий 6 подсистем:

| Подмодуль | Назначение | Backend | Frontend |
|-----------|-----------|---------|----------|
| **document** | Общий документооборот, файлы, версии, доступ | 34 файла, 2300+ LOC | 10 файлов |
| **cde** | ISO 19650 CDE, контейнеры, трансмиттали, архив | 25+ файлов | 10 файлов |
| **pto** | ПТО: акты, наряд-допуски, КС-6, лаб.испытания | 40+ файлов | 15 файлов |
| **dataExchange** | Импорт/экспорт данных (CSV, XLSX, PDF, JSON) | 15+ файлов, **НЕТ контроллеров** | 11 файлов |
| **integration1c** | 1С интеграция (OData, SOAP, sync) | 20+ файлов | 8 файлов |
| **edo** | ЭДО (Диадок, СБИС) | 10+ файлов | — (через integration1c UI) |

## 2. Детальный анализ подмодулей

### 2.1 Document (Общий документооборот) — 7.5/10

**Сильные стороны:**
- ✅ **Полноценный tenant isolation**: organizationId + @Filter на Document и DrawingMarkup
- ✅ **State machine** с валидацией переходов: DRAFT → UNDER_REVIEW → APPROVED → ACTIVE → ARCHIVED
- ✅ **Document versioning**: parentVersionId цепочка, auto-increment docVersion, getDocumentHistory()
- ✅ **Document access control**: DocumentAccess entity (VIEW/EDIT/FULL), grant/revoke per user
- ✅ **Storage abstraction**: S3StorageService с поддержкой MinIO, presigned URLs, content sniffing (Tika)
- ✅ **File validation**: FileValidationService — MIME whitelists, max 5GB, context-aware (document/messaging/bim/image)
- ✅ **Drawing markup**: CRUD для аннотаций на чертежах (RECTANGLE, CIRCLE, ARROW, TEXT, CLOUD, FREEHAND)
- ✅ **Audit logging**: все CRUD + status changes логируются через AuditService
- ✅ **19 DocumentCategory**: CONTRACT, APPENDIX, ESTIMATE, LOCAL_ESTIMATE, SPECIFICATION, DRAWING, DESIGN_DOC, PERMIT, ACT, INVOICE, COMMERCIAL_PROPOSAL, PROTOCOL, CORRESPONDENCE, CERTIFICATE, SCHEDULE, PHOTO, REPORT, TECHNICAL, OTHER

**Проблемы:**
- ⚠️ 8 GET endpoints без @PreAuthorize (list, getById, download-url, download, history, search, expiring, comments)
- ⚠️ DocumentAccess и DocumentComment — нет прямого @Filter (только через FK cascade на Document)
- ⚠️ UnifiedDocumentsPage навигация: GENERAL тип переходит на `/documents` вместо `/documents/{id}/edit`

### 2.2 CDE (ISO 19650) — 7.0/10

**Сильные стороны:**
- ✅ **ISO 19650 lifecycle**: WIP → SHARED → PUBLISHED → ARCHIVED (one-way)
- ✅ **Transmittal workflow**: DRAFT → ISSUED → ACKNOWLEDGED → RESPONDED → CLOSED
- ✅ **Revision management**: RevisionSet + DocumentRevision с CURRENT/SUPERSEDED
- ✅ **Archive policies**: per-org, auto-archive expired documents, configurable retention
- ✅ **Audit trail**: DocumentAuditEntry с JSONB details, per-action IP tracking
- ✅ **Unique constraints**: documentNumber per project, transmittalNumber per project

**КРИТИЧЕСКИЕ проблемы:**
- 🔴 **DocumentContainer — НЕТ organizationId, НЕТ @Filter** → cross-tenant data leak
- 🔴 **DocumentRevision — НЕТ organizationId** → наследует через FK, но нет прямой изоляции
- 🔴 **Transmittal — НЕТ @Filter(tenantFilter)** → from/to org есть, но нет tenant filter
- ⚠️ Document Locking (V136: locked_by_id, locked_at, lock_expires_at) — **колонки в БД, но НЕ реализовано** в service
- ⚠️ Revision Approval (V136: approval_status, approval_comment) — **колонки в БД, но НЕ реализовано** в service
- ⚠️ 10+ GET endpoints без @PreAuthorize (list, getById, revisions, audit, transmittal items)

### 2.3 PTO (Производственно-технический отдел) — 6.5/10

**Сильные стороны:**
- ✅ **Полный набор российских форм**: PtoDocument, WorkPermit, ActOsvidetelstvovanie, HiddenWorkAct, LabTest, KS-6/KS-6a, KS-11
- ✅ **Code generation**: PtoCodeGenerator с паттерном DOC-yyyyMMdd-nnnnn
- ✅ **WorkPermit safetyMeasures**: JSONB с @JdbcTypeCode(SqlTypes.JSON) ✅
- ✅ **HiddenWorkAct**: полная цепочка (photoIds, signatures, materials, geodesy)
- ✅ **ItdValidationPage**: 7 категорий правил (даты, подписи, ссылки, полнота, последовательность, материалы, геодезия)
- ✅ **KS-6/KS-6a**: monthly volume tracking with decade breakdown

**КРИТИЧЕСКИЕ проблемы:**
- 🔴 **PtoDocument — НЕТ organizationId, НЕТ @Filter** → cross-tenant data leak
- 🔴 **WorkPermit — НЕТ organizationId, НЕТ @Filter** → cross-tenant access к наряд-допускам!
- 🔴 **LabTest — НЕТ organizationId, НЕТ @Filter** → cross-tenant lab results
- 🔴 **Ks6Journal — НЕТ organizationId, НЕТ @Filter** → cross-tenant журнал работ
- 🔴 **PtoCodeGenerator: AtomicLong(System.currentTimeMillis() % 100000)** — сбрасывается при рестарте JVM → коллизии кодов
- ⚠️ PtoDocumentBoardPage — HTML5 DnD (не @dnd-kit), мобильные сломаны
- ⚠️ Некоторые JSONB поля без @JdbcTypeCode (HiddenWorkAct.photoIds)

### 2.4 DataExchange — 3.0/10 (INCOMPLETE)

**Что есть:**

### 2.5 Integration1C — 6.0/10

**Сильные стороны:**
- ✅ **OneCODataClient**: РЕАЛЬНЫЙ OData v3 клиент с Cyrillic, pagination, proper Basic Auth

- ✅ **Sync logs**: полный аудит trail

**Проблемы:**

### 2.6 EDO (Электронный документооборот) — 4.5/10

**Сильные стороны:**
- ✅ **EdoConfig** + **EdoDocument**: правильная структура с organizationId + @Filter
- ✅ **generateUpdXml()**: РЕАЛЬНАЯ генерация УПД XML по формату ФНС 5.02
- ✅ **EdoService architecture**: правильная абстракция provider (DIADOK/SBIS/KONTUR)
- ✅ **Status lifecycle**: DRAFT → SENT → DELIVERED → SIGNED_BY_COUNTERPARTY → REJECTED → ERROR

**КРИТИЧЕСКИЕ проблемы:**
- 🔴 **DiadokClient = 100% MOCK**: authenticate() → "mock-diadok-token", sendDocument() → "diadok-doc-{UUID}", downloadSignedDocument() → empty byte[]
- ⚠️ 4 GET endpoints без @PreAuthorize (document history, check status, download, config get)
- ⚠️ generateUpdXml() — buyer INN/KPP не резолвится из Invoice (используются дефолты)
- ⚠️ Нет SBIS/Kontur клиентов — только Diadok stub

### 2.7 Инфраструктура хранения — 8.5/10

**Сильные стороны:**
- ✅ **S3StorageService**: полная реализация AWS SDK v2, совместимо с MinIO
- ✅ **FileValidationService**: content sniffing через Apache Tika, context-aware whitelists
- ✅ **Presigned URLs**: time-limited download URLs для безопасной отдачи файлов
- ✅ **Auto bucket creation**: создаёт bucket при инициализации
- ✅ **Storage abstraction**: интерфейс StorageService позволяет подменять реализацию

**Замечания:**
- ⚠️ Max 5GB без multipart upload — для BIM-моделей может быть мало
- ⚠️ Нет lifecycle policies (auto-delete temp files, archive old files)

### 2.8 DaData/Checko — 8.0/10

**Сильные стороны:**
- ✅ **DaDataService**: реальный API, ИНН/ОГРН lookup, маппинг полей
- ✅ **ChekkaService**: risk scoring (0-100), 8 весовых факторов, financials, directors, founders
- ✅ **24h in-memory cache**: ConcurrentHashMap с TTL
- ✅ **SME category**: micro/small/medium classification

**Замечания:**
- ⚠️ Cache — нет eviction strategy, потенциальный memory leak при 100K+ ИНН
- ⚠️ Нет fallback между DaData и Checko (независимые вызовы)

## 4. Оценка по ролям

### 4.1 СМЕТЧИК
- DocumentCategory включает ESTIMATE, LOCAL_ESTIMATE, COMMERCIAL_PROPOSAL — документы привязаны к проекту ✅
- Нет автоматической привязки документа к смете/ФМ при загрузке — ручная работа ⚠️
- КС-6/КС-6а работают, но нет связи с КС-2 → нет автоматического заполнения ❌

### 4.2 ПРОРАБ
- PTO документы (акты, наряд-допуски, КС-6) — полный набор ✅
- HiddenWorkAct с фото, материалами, геодезией — правильная модель ✅
- Offline mode — отсутствует для PTO документов ❌
- Мобильный DnD сломан (PtoDocumentBoardPage) ❌
- ItdValidation — отличная идея для проверки полноты ИД, но demo data ⚠️

### 4.3 БУХГАЛТЕР
- 1С интеграция: OData + SOAP клиенты реально работают ✅
- Import контрагентов по ИНН — работает ✅
- Export КС-2/КС-3 в 1С — STUB (только sync log) ❌
- ЭДО (Диадок) — 100% mock, нельзя отправлять юридически значимые документы ❌
- УПД XML — генерируется по формату ФНС, но buyer INN hardcoded ⚠️

### 4.4 БЕЗОПАСНИК
- **10 P1 security bugs** — системная проблема tenant isolation в CDE + PTO
- DocumentAccess control (VIEW/EDIT/FULL) — есть, но GET endpoints open ⚠️
- File validation через Tika content sniffing — хорошо ✅
- Presigned URLs с TTL — хорошо ✅
- encryptedPassword в Integration1cConfig — не зашифрован ❌
- No audit on file downloads — нет логирования скачиваний ⚠️

### 4.5 АРХИТЕКТОР
- Document module: exemplary architecture — tenant isolation, state machine, versioning, audit ✅
- CDE ISO 19650 lifecycle — правильная модель ✅
- Storage abstraction — clean interface ✅
- DataExchange: services без controllers — **архитектурный дефект** ❌
- Duplicate EDO modules (russianDoc + edo) — нарушение DRY ⚠️
- PtoCodeGenerator: AtomicLong counter — антипаттерн в distributed системе ❌
- Integration1cMapper: String concatenation XML — уязвимо к injection ⚠️

### 4.6 UX ДИЗАЙНЕР
- UnifiedDocumentsPage: агрегация 4 типов документов в один view ✅
