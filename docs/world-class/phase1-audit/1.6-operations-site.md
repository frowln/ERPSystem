## ОБЩАЯ ОЦЕНКА: 6.0/10 — "Функциональный MVP с серьёзными дырами в tenant isolation"

**Статус**: `needs-work`

### Сильные стороны
- Полноценные CRUD для WorkOrder, DailyReport, Defect, Dispatch, M29, SiteAssessment
- Хорошая state machine (canTransitionTo) для всех сущностей
- Audit trail через AuditService на все операции
- Фронтенд: полная i18n, dark mode, empty states, loading states, responsive
- M29: 4-шаговый wizard (КС-2 → Материалы → Корректировка → Генерация) — бизнес-логика грамотная
- Defect: reinspection counter, SLA, drawing pin (x/y), contractor attribution — продвинутый функционал
- WorkOrder: auto-progress от DailyReport (heuristic, но рабочий)
- SiteAssessment: 12-критериальная система оценки + геотехника/экология/инженерные сети
- Бэкенд тесты: 7 файлов — лучше чем у большинства модулей

### Критические проблемы
- **3 сущности БЕЗ tenant isolation** (DispatchOrder, DailyReport, M29Document)
- **6 GET endpoints БЕЗ @PreAuthorize** (M29 list/get, Dispatch orders/routes)
- **Defect dashboard — cross-tenant aggregation** (countOpen/countOverdue без org filter)
- **DispatchService.orderSequence — AtomicLong in-memory** (сбрасывается при рестарте)
- **M29Document.name — global UNIQUE** (аналогичная проблема как Specification.name)
- **WorkOrderBoard — 100% mock** (пустой useState, нет API)

## 2. АНАЛИЗ ПО РОЛЯМ

### СМЕТЧИК (Calculations)
- **М-29 расчёт variance**: `M29Line.computeVariance()` — корректно считает отклонение planned vs actual

- **Проблема**: нет связи М-29 → Смета → ФМ для автоматического расчёта перерасхода по кост-кодам
- **Оценка**: 5/10 — базовые расчёты есть, нет интеграции с финансами

### ПРОРАБ (Field Usability)
- **DailyLog**: AI auto-fill (suggestDailyLog), voice input — отлично для поля
- **WorkOrder**: Kanban board — HTML5 DnD не работает на планшете/телефоне

- **Оценка**: 6/10 — база хорошая, но Kanban board не работает, офлайн не реализован

### БУХГАЛТЕР (Financial Correctness)
