## 2. Найденные баги

### P1 — Critical

#### BUG-D1: Все 3 entity БЕЗ organizationId — полное отсутствие tenant isolation [P1] [SECURITY]
**Файлы**: `DesignVersion.java`, `DesignReview.java`, `DesignSection.java`, `V70__dispatch_design_kpi_crew.sql`

Ни одна из 3 таблиц (design_versions, design_reviews, design_sections) не имеет колонки `organization_id`. В entities нет `organizationId` поля, нет `@Filter` аннотации. Все запросы в `DesignService` возвращают данные ВСЕХ организаций.

**Воздействие**: В SaaS-системе данные одного тенанта видны другому. Проектная документация — коммерческая тайна. Полная cross-tenant утечка.

**Доказательство**: `DesignService.listVersions()` при `projectId=null` вызывает `versionRepository.findByDeletedFalse(pageable)` — возвращает ВСЕ версии всех организаций. То же для `listReviews()` и `getSectionsForProject()` (projectId может принадлежать чужой организации).

#### BUG-D2: Frontend↔Backend field mismatch — фронтенд обращается к полям, которых нет в ответе [P1] [DATA]
**Файлы**: `frontend/src/modules/design/types.ts:3-22`, `backend/.../dto/DesignVersionResponse.java`

Frontend DesignVersion type ожидает поля:
- `number` — нет в backend (нет поля code/number в entity)
- `sectionName` — нет в backend (только discipline)
- `version` — нет в backend (есть versionNumber)
- `authorId` — нет в backend (есть только author: String)
- `authorName` — нет в backend (есть author: String)
- `projectName` — нет в backend
- `reviewCount` — нет в backend
- `approvedAt` — нет в backend

Frontend DesignReview type ожидает:
- `number` — нет в backend
- `versionTitle`, `versionNumber` — нет в backend
- `projectId`, `projectName` — нет в backend
- `dueDate` — нет в backend
- `markupCount` — нет в backend

Frontend DesignSection type ожидает:
- `leadDesignerId`, `leadDesignerName` — нет в backend
- `latestVersion`, `latestVersionStatus` — нет в backend
- `versionCount` — нет в backend (нужен JOIN-запрос)

**Воздействие**: Все 3 list-страницы будут показывать `undefined` в колонках. Поиск по `sectionName`, `authorName`, `projectName` crash при `.toLowerCase()` на undefined.

#### BUG-D3: Frontend API endpoint mismatch — updateVersionStatus вызывает несуществующий endpoint [P1] [API]
**Файл**: `frontend/src/api/design.ts:47-49`

`designApi.updateVersionStatus()` вызывает `PATCH /v1/design/versions/${id}/status` с body `{ status }`. Но backend НЕ имеет PATCH endpoint. Есть только POST endpoints для конкретных переходов:
- POST `/versions/{id}/submit-for-review`
- POST `/versions/{id}/approve`
- POST `/versions/{id}/supersede`
- POST `/versions/{id}/archive`

**Воздействие**: DesignVersionFormPage при обновлении получит 405 Method Not Allowed. Невозможно обновить версию через UI.

#### BUG-D4: DesignReviewBoardPage = 100% mock — пустой useState, нет API вызовов [P1] [STUB]
**Файл**: `frontend/src/modules/design/DesignReviewBoardPage.tsx:41`

```typescript
const [items, setItems] = useState<DrCard[]>([]);
```

Страница ВСЕГДА пустая. Нет useQuery, нет fetch, нет API-вызовов. Карточки никогда не загружаются. Доска показывает только 4 пустые колонки с текстом "Нет разделов" / "Перетащите сюда".

Также используется HTML5 DnD API (draggable, onDragStart, onDrop) вместо @dnd-kit — не работает на мобильных (iOS/Android).

#### BUG-D5: DesignSectionListPage вызывает getSections() без projectId → 400 Bad Request [P1] [API]
**Файл**: `frontend/src/modules/design/DesignSectionListPage.tsx:40-43`

```typescript

```

API функция `designApi.getSections()` из `src/api/design.ts:72-75` вызывается без параметров. Но в backend `DesignController.getSections()` строка 200: `@RequestParam UUID projectId` — обязательный параметр. Без него — 400 Bad Request.

**Воздействие**: Страница разделов ВСЕГДА ломается при загрузке.

### P2 — High

#### BUG-D6: Status enum mismatch Version — frontend 'REJECTED' не существует в backend [P2] [DATA]
**Файлы**: `frontend/src/modules/design/types.ts:1`, `backend/.../DesignVersionStatus.java`

Frontend определяет `DesignVersionStatus = 'DRAFT' | 'IN_REVIEW' | 'APPROVED' | 'SUPERSEDED' | 'REJECTED' | 'ARCHIVED'`. Backend enum: DRAFT, IN_REVIEW, APPROVED, SUPERSEDED, ARCHIVED — **нет REJECTED**. DB CHECK constraint тоже не включает REJECTED.

Фильтр "Отклонено" в списке версий никогда не покажет результатов. MetricCard "Отклонено" всегда 0.

#### BUG-D7: Status enum mismatch Review — frontend 'IN_PROGRESS' не существует в backend [P2] [DATA]
**Файлы**: `frontend/src/modules/design/types.ts:24`, `backend/.../DesignReviewStatus.java`

Frontend определяет `DesignReviewStatus = 'PENDING' | 'IN_PROGRESS' | 'APPROVED' | 'REJECTED' | 'REVISION_REQUESTED'`. Backend enum: PENDING, APPROVED, REJECTED, REVISION_REQUESTED — **нет IN_PROGRESS**. DB CHECK constraint тоже не включает IN_PROGRESS.

Таб "В работе" на странице рецензий всегда пустой.

#### BUG-D8: Duplicate API files — two competing design API implementations [P2] [ARCH]
**Файлы**: `frontend/src/api/design.ts`, `frontend/src/modules/design/api.ts`

Существуют ДВА файла с API функциями для design:

`src/api/design.ts` имеет НЕПРАВИЛЬНЫЙ endpoint `PATCH /status` (BUG-D3). `src/modules/design/api.ts` имеет ПРАВИЛЬНЫЕ endpoints (POST submit-for-review, POST approve, etc.).

Страницы импортируют НЕПРАВИЛЬНУЮ версию.

### P3 — Medium

#### BUG-D18: createVersion/updateVersion — один DTO для create и update [P3] [ARCH]
**Файл**: `backend/.../service/DesignService.java:97-113`

UpdateVersion использует `CreateDesignVersionRequest` — все поля с `@NotNull/@NotBlank`. Но update делает null-checks (`if (request.title() != null)`). При partial update `@NotBlank` на title потребует отправки ВСЕХ полей, даже если меняем только одно. Нужен отдельный UpdateDesignVersionRequest.

#### BUG-D19: DesignVersion.author — String, не связан с User entity [P3] [DATA]
**Файл**: `backend/.../domain/DesignVersion.java:49`

`author` хранится как VARCHAR(255) строка, а не как FK на users таблицу. Нет referential integrity. При удалении/переименовании пользователя — данные рассинхронизируются.

#### BUG-D20: DesignReview.reviewerName дублирует данные из users [P3] [DATA]
**Файл**: `backend/.../domain/DesignReview.java:39`

`reviewer_name` хранится прямо в таблице design_reviews рядом с reviewer_id. Денормализация для performance — ОК, но при смене ФИО рецензента данные рассинхронизируются навсегда.

#### BUG-D21: Нет пагинации в getReviewsForVersion — все рецензии в памяти [P3] [PERF]
**Файл**: `backend/.../service/DesignService.java:149-153`

`getReviewsForVersion()` возвращает `List<DesignReviewResponse>` — все рецензии одной версии. При большом количестве рецензий (enterprise) — всё в памяти.

#### BUG-D22: Нет пагинации в getSectionsForProject — все разделы в памяти [P3] [PERF]
**Файл**: `backend/.../service/DesignService.java:208-213`

`getSectionsForProject()` возвращает `List<DesignSectionResponse>`. Для крупных проектов с десятками разделов и подразделов — всё загружается сразу без пагинации.

#### BUG-D23: Нет routePermissions для design routes [P3] [SECURITY]
**Файл**: `frontend/src/config/routePermissions.ts`

Ни один design route не указан в routePermissions.ts. Значит ВСЕ роли (включая VIEWER) могут видеть design-страницы. Для проектной документации нужна как минимум роль ENGINEER.

#### BUG-D24: DesignVersionListPage client-side filtering и пагинация [P3] [PERF]
**Файл**: `frontend/src/modules/design/DesignVersionListPage.tsx:53-56`

```typescript

```

Загружает ВСЕ версии с первой страницей (size=20 по умолчанию backend), но фильтрует client-side. При >20 версий — фильтры работают только по первым 20 записям. Нужно передавать фильтры в params.

#### BUG-D25: DesignReview.completeReview — нет связи с DesignVersion автостатусом [P3] [LOGIC]
**Файл**: `backend/.../service/DesignService.java:182-201`

Когда ВСЕ рецензии для версии получают статус APPROVED, версия должна автоматически перейти в APPROVED. Сейчас рецензент должен вручную найти версию и отдельно её утвердить. Нет автоматизации workflow.

## 4. Анализ по ролям

### СМЕТЧИК 🔢
- Модуль design не содержит расчётов. Разделы ПД (АР, КР, ИОС) корректно соответствуют российским стандартам ГОСТ 21.101-2020.
- Нет привязки версий документации к сметным разделам.
- **Оценка**: 5/10 — базовые коды есть, но нет связи с финансами.

### ПРОРАБ 👷
- **Критично**: нет offline-режима. На стройке невозможно открыть последнюю версию чертежа без интернета.
- Нет мобильного UI для быстрого доступа к чертежам.
- Нет привязки замечаний (дефектов) к конкретной версии документации.
- **Оценка**: 3/10 — прорабу нечего делать в этом модуле.

### БУХГАЛТЕР 💰
- Модуль не затрагивает финансы. Нет замечаний.

### БЕЗОПАСНИК 🔒
- **CRITICAL**: Zero tenant isolation — 3 entity без organizationId (BUG-D1).
- 8 GET endpoints без @PreAuthorize (BUG-D11).
- No route permissions на frontend (BUG-D23).
- fileUrl хранится как строка — нет проверки на path traversal при формировании URL.
- **Оценка**: 2/10 — критические дыры.

### АРХИТЕКТОР 🏗️
- Backend архитектура чистая: правильное разделение domain/repository/service/web/dto.
- DesignService не god-class (296 LOC — нормально для 3 entity).
- Duplicate API files (BUG-D8) — нарушение SRP.
- Frontend↔Backend десинхронизация полей (BUG-D2) — architectural smell.
- Status machine правильная (canTransitionTo с exhaustive switch).
- Нет связи с другими модулями (BIM, documents, specifications) — isolated island.
- **Оценка**: 5/10 — backend ОК, frontend broken.

### UX ДИЗАЙНЕР 🎨

## 5. Дублирование с BIM модулем

В `modules/bim/` существует параллельная подсистема проектирования:
- `DesignPackage` — аналог DesignVersion (с approval workflow)
- `DesignDrawing` — отдельные чертежи внутри пакета
- `DesignDiscipline` enum — 10 дисциплин (vs string discipline в design)
- `DrawingAnnotation` — markup на чертежах (то, чего НЕТ в design)

**Проблема**: Два конкурирующих модуля для одной и той же задачи. BIM-модуль имеет больше функционала (drawings, annotations, packages), но design-модуль имеет разделы (sections) с иерархией и review workflow.

**Рекомендация**: Объединить design и bim.design* в единый модуль. DesignSection → DesignPackage, DesignVersion → DesignDrawing.
