## НАЙДЕННЫЕ БАГИ

### P1 — Critical

#### BUG-F1: НДС — Backend VatCalculator застрял на 20%, а с 2026 года НДС = 22%
**Суть**: С 01.01.2026 ставка НДС в России повышена до 22%. Фронтенд ФМ (0.22) и миграция V120 (DEFAULT 22.00) уже обновлены. Но бэкенд `VatCalculator.DEFAULT_RATE = 20.00` и несколько миграций для других таблиц (invoices, contracts, PO items) по-прежнему используют 20%.

**Рассогласованные файлы (нужно обновить до 22%)**:
- `backend/src/main/java/.../infrastructure/finance/VatCalculator.java:12` — `DEFAULT_RATE = new BigDecimal("20.00")`
- `frontend/src/modules/finance/InvoiceFormPage.tsx:100` — `vatRate: '20'`
- `backend/src/main/resources/db/migration/V13__finance_tables.sql:154` — `vat_rate DEFAULT 20.00` (invoices)
- `backend/src/main/resources/db/migration/V6__contract_tables.sql:60` — `vat_rate DEFAULT 20.00` (contracts)
- `backend/src/main/resources/db/migration/V91:50` — `vat_rate DEFAULT 20.00` (PO items)
- `backend/src/main/resources/db/migration/V208:104` — `vat_rate DEFAULT 20.00` (normative cost)
- `backend/src/main/resources/db/migration/V231:20` — `vat_rate DEFAULT 20.00` (invoice enhancements)

**Правильные файлы (уже 22%)**:
- `FmPage.tsx:220`, `FmItemsTable.tsx:136,234,416` — `0.22` ✅
- `budgetDetailTypes.ts:97` — `vatRate: '22'` ✅
- `V120__budget_items:13` — `DEFAULT 22.00` ✅

**Влияние**: Новые счета и контракты создаются с дефолтным НДС 20% вместо 22%. Расхождение frontend (22%) ↔ backend (20%) при создании invoice. Нужна миграция для обновления defaults + правка VatCalculator.

**Приоритет**: CRITICAL — финансовые документы создаются с неправильной ставкой НДС.

#### BUG-F2: NPE в BudgetService.recalculateActuals()
**Суть**: `budget.getActualRevenue()` может быть `null` (поле `actual_revenue` не имеет NOT NULL constraint, default не задан для старых записей). Вызов `.subtract()` на null → NullPointerException → 500 Internal Server Error.

**Файл**: `backend/src/main/java/com/privod/platform/modules/finance/service/BudgetService.java:294`
```java
budget.setActualMargin(budget.getActualRevenue().subtract(totalActualCost));
```

**Влияние**: Crash при пересчёте фактических данных бюджета. Пользователь не сможет обновить факт.

**Приоритет**: CRITICAL — runtime crash на основной бизнес-операции.

---

#### BUG-F3: N+1 запрос в getExpenses() — потенциальный OOM
**Суть**: Метод загружает budget items в цикле (1 запрос на каждый бюджет), затем фильтрует и пагинирует В ПАМЯТИ.

**Файл**: `backend/src/main/java/com/privod/platform/modules/finance/service/BudgetService.java:718-741`
```java
List<BudgetItem> allItems = budgetIds.stream()
    .flatMap(bId -> budgetItemRepository.findByBudgetIdAndDeletedFalseOrderBySequenceAsc(bId).stream())
    .toList(); // N+1: один запрос на каждый бюджет!
// ...
List<BudgetItem> pageItems = start < total ? filtered.subList(start, end) : List.of(); // In-memory pagination!
```

**Влияние**: На проекте с 50 бюджетами × 200 позиций = 10,000 объектов в памяти. При 100+ бюджетах — OutOfMemoryError.

**Приоритет**: CRITICAL — OOM на production с реальными данными.

---

#### BUG-F4: Дублирование CashFlowEntry при повторной оплате счёта
**Суть**: При вызове `InvoiceService.registerPayment()` создаётся `CashFlowEntry` без проверки на существование. Если метод вызван повторно (retry, race condition), записи дублируются.

**Файл**: `backend/src/main/java/com/privod/platform/modules/finance/service/InvoiceService.java:330-337`

**Влияние**: Искажение отчёта Cash Flow — двойной учёт оплаты.

**Приоритет**: CRITICAL — финансовая ошибка, невидимая пользователю.

## ИТОГ

### Что модуль делает хорошо:
1. **Чистая архитектура** — layered design, DTO separation, tenant isolation
2. **BigDecimal precision** — правильная работа с деньгами
3. **Backend tests** — 80 тестов с хорошим покрытием state machine
4. **KPI transparency** — 10 метрик на одном экране, прозрачные расчёты
5. **3-way matching** — Invoice ↔ Contract ↔ КС-2 ↔ Budget Item
6. **VatCalculator** — централизованный, правильный (20%), с extractVat()
7. **Audit trail** — все операции логируются
8. **Dark mode** — 924 класса, консистентный

### Что нужно исправить немедленно:
1. **НДС 22% → 20%** — влияет на все финансовые показатели
2. **NPE в recalculateActuals()** — crash на основной операции
3. **N+1 + in-memory pagination** — OOM при масштабировании
4. **CashFlowEntry дублирование** — финансовая ошибка

### Чего критически не хватает:
1. Реальный сметный движок (ГЭСН/ФЕР/ТЕР)
2. Авансы и удержания
3. 1С интеграция
4. EVM (CPI/SPI/EAC)
5. Separation of duties в approval workflow
