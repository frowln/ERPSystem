## 2. ОЦЕНКА ПО РОЛЯМ

### 🔨 ПРОРАБ (7/10)
**Closeout**: Отличный набор инструментов для сдачи объекта. Чек-листы пусконаладки (7 секций: конструкции, МЕР, пожарка, лифты, благоустройство, доступность, документация), пакеты передачи, гарантийные рекламации с workflow (OPEN→UNDER_REVIEW→APPROVED→IN_REPAIR→RESOLVED→CLOSED), трекер исполнительной документации по WBS. Пакет Стройнадзора — уникальная функция (автосборка из 5 источников: АОСР, QA, сертификаты, пусконаладка, ИД).

**Maintenance**: Базовый функционал есть (заявки, оборудование, ТО-графики), но Kanban доска пустая (нет API), детальных карточек заявок и оборудования нет.

### 📊 СМЕТЧИК (6/10)
- WarrantyClaim корректно отслеживает costOfRepair
- Нет привязки гарантийных расходов к бюджету проекта
- Нет автоматического расчёта гарантийных удержаний из контракта
- MaintenanceRequest.cost есть, но нет агрегации затрат на ТО по оборудованию/проекту

### 💰 БУХГАЛТЕР (5/10)
- Нет интеграции гарантийных обязательств с бухгалтерским учётом
- Нет провизий (резервов) на гарантийные ремонты (ФСБУ)
- Нет связи maintenance cost → проводки
- Нет печатных форм: акт приёмки объекта, дефектный акт, акт выявленных дефектов

### 🔒 БЕЗОПАСНИК (5/10)
**Closeout**: 10/11 entities с tenant isolation (отлично!). 1 дыра: StroynadzorPackageDocument без organizationId. AsBuiltTrackerController — 10 endpoints без @PreAuthorize.

### 🎨 UX ДИЗАЙНЕР (7/10)

## 5. ПРОБЛЕМЫ БЕЗОПАСНОСТИ

### Критические

1. **Maintenance — полное отсутствие tenant isolation** (5 entities × 24 метода × 22 endpoints)
   - Attack: `GET /api/v1/maintenance/requests` → все заявки ВСЕХ организаций
   - Attack: `DELETE /api/v1/maintenance/requests/{uuid}` → удаление заявки чужой организации
   - Attack: `GET /api/v1/maintenance/dashboard` → агрегированные метрики всех организаций
   - Рекомендация: добавить organizationId + @Filter на все 5 entities, обновить service/repository

2. **StroynadzorPackageDocument — нет tenant filter**
   - Indirect access через parent entity, но direct query возможен
   - Рекомендация: добавить organizationId, backfill из StroynadzorPackage

### Высокие

3. **AsBuiltTrackerController — 10 endpoints без RBAC**
   - POST/DELETE requirements и WBS links доступны VIEWER
   - Рекомендация: добавить @PreAuthorize на все write operations

4. **Closeout GET endpoints без @PreAuthorize** (20+ endpoints)
   - Pattern consistent с другими модулями, но defense-in-depth отсутствует
   - Tenant isolation через @Filter работает, но если session filter отключен — data leak

## 8. ЛУЧШИЕ НАХОДКИ (что сделано хорошо)

### 🏆 StroynadzorPackageService (9/10)
Уникальная функция — автоматическая сборка пакета документов для Стройнадзора:
- BFS обход WBS-дерева для сбора всех linked документов
- 5 источников: АОСР, Quality Checks, Certificates, Commissioning, AsBuilt
- TOC (table of contents) generation по категориям
- Missing documents report с completeness %
- Status flow: DRAFT → GENERATING → READY → SENT
- Нет аналогов у Procore, PlanRadar, Buildertrend

### 🏆 WarrantyClaim State Machine (8/10)
Правильный workflow: OPEN → UNDER_REVIEW → APPROVED → IN_REPAIR → RESOLVED → CLOSED (+ REJECTED ветка). Каждый transition с отдельным endpoint, валидацией, audit log.

### 🏆 AsBuiltTracker Quality Gate (8/10)
Per-WBS element tracking: link documents → track submission status → quality gate (all ACCEPTED = pass). Completeness % calculation. Уникальная интеграция с WBS.

### 🏆 Closeout Tenant Isolation (8/10)
10/11 entities с proper organizationId + @Filter. Все 7 services используют SecurityUtils.requireCurrentOrganizationId(). Лучший показатель среди всех проверенных модулей.

### 🏆 i18n Coverage (10/10)
498 + 118 = 616 i18n keys, 0 hardcoded строк в JSX. Структурно идентичные ru.ts и en.ts.

### 🏆 Maintenance RequestStatus State Machine (8/10)
canTransitionTo() с явными правилами (NEW→IN_PROGRESS→REPAIRED/SCRAP, terminal states block). processPreventiveSchedules() автоматически создаёт заявки из графиков.

### 🏆 Test Coverage (Closeout) (7/10)
6 test файлов покрывающих core services и controllers (CommissioningChecklist, HandoverPackage, WarrantyClaim). Каждый тест включает create, update, status transitions, 404 handling.
