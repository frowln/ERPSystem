## 2. Closing (КС-2 / КС-3): Подробный анализ

### 2.1 Backend

#### Entities
- **Ks2Document**: number, documentDate, projectId, contractId, status, totalAmount, totalWithVat, totalVatAmount, investorName, okdp, normCode, reportingPeriod, edoStatus, oneCPostingStatus, sourceDailyLogIds
- **Ks2Line**: ks2Id, sequence, specItemId, estimateLineId, name, quantity, unitPrice, amount, vatRate, vatAmount, unitOfMeasure
- **Ks3Document**: number, documentDate, periodFrom, periodTo, projectId, contractId, totalAmount, retentionPercent (default 5%), retentionAmount, netAmount, edoStatus, oneCPostingStatus
- **Ks3Ks2Link**: ks3Id, ks2Id (UNIQUE constraint)
- **ClosingDocumentStatus**: DRAFT → SUBMITTED → SIGNED → CLOSED (FSM с canTransitionTo)

#### Controllers & Endpoints (27 endpoints)
- **Ks2Controller** (`/api/ks2`): CRUD + lines + submit/sign/close — 12 endpoints
- **Ks3Controller** (`/api/ks3`): CRUD + link-ks2/auto-fill + submit/sign/close + create-invoice — 12 endpoints
- **Ks2PipelineController** (`/api/ks2-pipeline`): generate/batch-generate/preview/volumes — 4 endpoints
- **Ks6aController** (`/api/ks6a`): getEntries — **ЗАГЛУШКА** (returns empty list)
- **CorrectionActController** (`/api/correction-acts`): list (empty) + create (**throws UnsupportedOperationException**)

#### Service Logic
- **ClosingDocumentService** (660+ строк): CRUD для КС-2/КС-3, управление строками, пересчёт итогов, статусные переходы, создание счетов из КС-3
- **Ks2PipelineService** (330+ строк): автогенерация КС-2 из ежедневных журналов, сбор объёмов, матчинг со спецификацией, расчёт unit price
- **KsOneCExportService**: async экспорт SIGNED/CLOSED документов в 1С (OData формат)

#### Расчёты (проверка СМЕТЧИКОМ)
- **НДС КС-2**: `VatCalculator.vatAmount(totalAmount, vatRate)` = amount × rate / 100, HALF_UP → **Корректно** (НК РФ ст. 164)
- **Итоги КС-2**: SUM(line.amount) → totalAmount, totalAmount × vatRate → totalVatAmount, totalAmount + totalVatAmount → totalWithVat → **Корректно**
- **Удержание КС-3**: totalAmount × retentionPercent / 100 = retentionAmount, totalAmount - retentionAmount = netAmount → **Корректно**
- **Pipeline unitPrice**: plannedAmount / quantity из SpecItem → **Корректно для базовых случаев**
- **⚠️ VAT rate**: VatCalculator.DEFAULT_RATE = 20% — уже отмечено как P1 баг (с 2026 НДС=22%)

## 3. RussianDocs: Подробный анализ

### 3.1 Backend (25 entities, 14 repositories, 5 services)

#### Основные сущности
- **Upd** (УПД): number, date, seller/buyer, items (JSONB), totalAmount, vatAmount, status
- **Act** (Акт): number, date, contractId, executor/customer, items (JSONB), totalAmount, period
- **Torg12** (ТОРГ-12), **SchetFaktura** (Счёт-фактура), **Waybill** (ТТН): базовые entities
- **PowerOfAttorney** (Доверенность М-2): number, date, issuedTo, validUntil, materialList (JSONB)
- **InventoryAct**, **WriteOffAct**: инвентаризация и списание

#### ЭДО Entities (EDO v1 — в модуле russianDoc)
- **EdoDocument**: documentNumber, documentType, senderInn, receiverInn, status, xmlData
- **EdoTemplate**: code (unique), templateXml, isActive
- **EdoGeneratedDocument**: templateId, sourceDoc, generatedXml, status
- **EdoSignature**: documentId, signerName, certificateSerial, isValid
- **EdoExchangeLog**: аудит-трейл

#### OCR Entities
- **OcrTask**: fileUrl, status (PENDING→PROCESSING→COMPLETED/FAILED), recognizedText, recognizedFields (JSONB), confidence
- **OcrTemplate**, **OcrEstimateResult**

#### Контроллер RussianDocController
```
GET/POST/PUT/DELETE /api/russian-docs        — CRUD
PATCH              /api/russian-docs/{id}/status
GET                /api/russian-docs/{id}/export  — PDF/XLSX (не реализован на бэкенде!)
GET/POST/PUT/DELETE /api/russian-docs/upd     — УПД specific
```

### 3.2 Tenant Isolation & Security
- ✅ `@Filter(name = "tenantFilter")` на Upd, Act, OcrTask, PowerOfAttorney
- ✅ `SecurityUtils.requireCurrentOrganizationId()` в сервисах
- ✅ `@PreAuthorize` на всех мутирующих endpoints (ADMIN/PM/ACCOUNTANT/DOC_MANAGER)
- ✅ Защита от cross-tenant: `if (request.organizationId() != currentOrgId) → AccessDeniedException`

### 3.3 Проблемы
- **RussianDocService.getUpd()**: использует `updRepository.findById()` без tenant filter (строка 66) — inconsistency с другими методами
- **EdoService.applyTemplate()**: `String.replace()` вместо template engine — silent failure если placeholder отсутствует
- **OcrTask.totalLinesDetected, averageConfidence**: колонки существуют, но никогда не заполняются
- **Export endpoint**: `/api/russian-docs/{id}/export` объявлен во фронтенде, но **не реализован на бэкенде**

## 5. ЭДО (Электронный документооборот)

### 5.1 Backend — ДВЕ РЕАЛИЗАЦИИ

#### EDO v1 (в russianDoc module)
- EdoDocument, EdoTemplate, EdoSignature
- Простой template engine (`String.replace()`)
- 6 тестов

#### EDO v2 (отдельный `edo` module) — основной
- **EdoConfig**: per-org конфигурация провайдера (Diadok/SBIS/Kontur)
- **EdoDocument**: трекинг отправленных документов (status, counterpartyInn, externalId)
- **EdoService**: отправка КС-2/КС-3/Invoice через Diadok, проверка статуса, скачивание подписанных
- **DiadokClient**: **ПОЛНАЯ ЗАГЛУШКА** — `authenticate()` → mock token, `sendDocument()` → mock ID, `getDocumentStatus()` → always "DELIVERED", `downloadSignedDocument()` → empty byte[]
- **УПД XML генерация**: **РЕАЛИЗОВАНА** — Приказ ФНС № ЕД-7-26/970@, версия 5.02

#### EdoKs2ExportService (в integration module)
- Отдельный слой для экспорта КС-2/КС-3 через ЭДО
- Endpoints: `POST /api/integrations/edo-export/ks2/{id}/send`, `POST .../ks3/{id}/send`
- Webhook для входящих: `POST /api/integrations/edo-export/inbound`

### 5.2 Оценка
- Архитектура правильная (Config → Document → Status tracking)
- УПД XML — единственная реально работающая часть
- **Diadok REST API не вызывается** — всё mock
- **SBIS API не вызывается** — всё mock
- Нет retry-логики, нет webhook-обработки, нет журнала ошибок

## 6. КЭП (Квалифицированная электронная подпись)

### 6.1 Backend — полная архитектура
- **KepCertificate**: ownerId, serialNumber, issuer, validFrom/To, thumbprint, subject*, status (ACTIVE/EXPIRED/REVOKED/SUSPENDED), certificateData (PEM), qualified
- **KepSigningRequest**: documentModel, documentId, requesterId, signerId, status, dueDate, priority (NORMAL/HIGH/URGENT)
- **KepSignature**: certificateId, documentModel, documentId, signatureData (Base64), signatureHash (SHA-256), isValid
- **KepController**: 15 endpoints для сертификатов, запросов подписания, подписей
- **BouncyCastleCryptoProvider**: RSA + GOST алгоритмы, OCSP валидация
- **CryptoProviderService**: factory для CryptoPro/ViPNet/Rutoken/JaCarta

### 6.2 Проблемы
- **Нет CryptoPro JCP SDK**: подпись CMS (ГОСТ 34.10-2012) не реализована — это ключевое требование для госконтрактов
- **Нет HSM**: ключи хранятся в БД как PEM — в production нужен аппаратный HSM
- **CRL/OCSP refresh**: нет автоматической проверки отзыва сертификатов
- 22 backend теста — хорошее покрытие базовой логики

## 13. Итоговая оценка

### По ролям

| Роль | Оценка | Комментарий |
|------|--------|-------------|
| **СМЕТЧИК** | 6/10 | КС-2 расчёты верные, НДС корректный. Но VAT не показывается в API, PDF нет |

| **БУХГАЛТЕР** | 5/10 | Удержания есть, invoice создаётся. Но ЭДО mock, кумулятивного КС-3 нет, PDF нет |
| **БЕЗОПАСНИК** | 3/10 | **CRITICAL**: organizationId не ставится → tenant isolation не работает для closing |

| **UX ДИЗАЙНЕР** | 7/10 | Dark mode 100%, loading/empty/error states, Zod validation. Печать — стаб |
| **ПРОДАКТ** | 5/10 | 80% closing работает. Но ЭДО mock, execDocs mock, PDF нет — клиент не купит |
| **QA ИНЖЕНЕР** | 4/10 | 87 backend тестов ОК, но 0 frontend тестов на 5000 строк UI кода |

### Общая оценка: **5.5/10**

**Сильные стороны:**
- Правильная архитектура closing module (entity → repo → service → controller → DTO)
- КС-2 Pipeline из ежедневных журналов — инновационный функционал
- 4-этапный approval workflow
- Volume check vs смета
- КЭП инфраструктура (BouncyCastle, 22 теста)
- УПД XML генерация по ФНС стандарту
- 1С OData маппинг

**Критические слабости:**
- **organizationId не устанавливается** → tenant isolation сломан (P1)
- **5 execDocs страниц без бэкенда** → localStorage = потеря данных
- **DiadokClient = 100% mock** → ЭДО не работает
- **PDF/1С экспорт = toast-заглушки** → бухгалтер не может получить документ
- **КС-6а = пустой список** → обязательная форма не работает
- **0 frontend тестов** на 5000 строк UI кода
