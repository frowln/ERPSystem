## 1. Найденные баги

### P1 — Critical

#### BUG-C1: [SECURITY] ContractExt — системный bypass tenant isolation (IDOR)
**Все** 8 create-операций в contractExt НЕ устанавливают `organizationId` при создании записей И НЕ проверяют, что referenced `contractId`/`projectId` принадлежит организации текущего пользователя.

**Затронутые сервисы:**
- `ContractClaimService.create()` — `contractExt/service/ContractClaimService.java:41-61`
- `ContractSupplementService.create()` — `contractExt/service/ContractSupplementService.java:42-61`
- `ContractSlaService.create()` (предполагается аналогично)
- `ContractExtService.createGuarantee()` — `contractExt/service/ContractExtService.java:49-69`
- `ContractExtService.createMilestone()` — `contractExt/service/ContractExtService.java:81-98`
- `ContractExtService.createInsurance()` — `contractExt/service/ContractExtService.java:133-159`
- `LegalCaseService.create()` — `contractExt/service/LegalCaseService.java:53-75`
- `ContractToleranceService.create()` — `contractExt/service/ContractToleranceService.java:52-69`

**Вектор атаки**: Пользователь Org A, зная UUID контракта Org B, создаёт претензию/ДС/гарантию к чужому контракту. `organizationId = NULL` → запись "ничейная", потенциально видна всем.

**Воздействие**: Cross-tenant data manipulation. В строительном ERP это означает, что один подрядчик может подать фиктивные претензии к контракту конкурента.

#### BUG-C2: [SECURITY] ToleranceController.listAll() — full data leak
`ToleranceController.java` при запросе `GET /api/tolerances` без `projectId` вызывает `toleranceService.listAll()`, который возвращает **ВСЕ** допуски **ВСЕХ** организаций без фильтрации по tenant.

**Файл**: `contractExt/web/ToleranceController.java` (предположительная строка ~48)

#### BUG-C3: [CALC] НДС 20% вместо 22% в контрактах
`Contract.java:78` — `vatRate = VatCalculator.DEFAULT_RATE` = `20.00`. С 2026 НДС = 22%.
Каждый новый контракт по умолчанию создаётся с заниженной ставкой. При контракте 100M руб. → недобор НДС 2M руб.

**Файлы**: `contract/domain/Contract.java:78`, `infrastructure/finance/VatCalculator.java:12`

#### BUG-C4: [DATA] ContractSignWizard — данные подписания НЕ сохраняются
`ContractSignWizard.tsx:76-87` — `handleFinish()` вызывает только `changeContractStatus(contractId, 'SIGNED')`. Имя подписанта, роль, дата подписания, номер доверенности, загруженный файл скана — **ничего не передаётся на бэкенд**. Пользователь заполняет 3-шаговый визард, но данные теряются.

### P2 — High

#### BUG-C5: [DATA] Board page — несовпадение статусов с бэкендом
`ContractBoardPage.tsx:17` определяет `ContractStatus = 'DRAFT' | 'APPROVED' | 'ACTIVE' | 'COMPLETED' | 'TERMINATED'`, а бэкенд использует `ON_APPROVAL | SIGNED | CLOSED | CANCELLED` и другие. Контракты в статусах `ON_APPROVAL`, `LAWYER_APPROVED`, `MANAGEMENT_APPROVED`, `FINANCE_APPROVED`, `SIGNED`, `CLOSED` — **не отображаются ни в одной колонке**.

#### BUG-C6: [UX] Board page — HTML5 DnD не работает на мобильных
`ContractBoardPage.tsx:115-119` использует `draggable`/`onDragStart`/`onDrop` — HTML5 API, которое не работает на iOS/Android. Все остальные 17 Kanban-досок уже переведены на `@dnd-kit/core` с `TouchSensor`, но contracts board пропущена.

#### BUG-C7: [DATA] Вкладка "Документы" — заглушка
`ContractDetailPage.tsx:413-414` — `([] as any[]).map(...)` — таблица документов всегда пустая. Счётчики КС-2/КС-3 hardcoded (`value="1"` на строках 388-389).

#### BUG-C8: [LOGIC] Форма удаляет данные страхования/закупок
`ContractFormPage.tsx:188-199` и `221-232` — `createMutation` и `updateMutation` удаляют из payload поля `insuranceType`, `insurancePolicyNumber`, `insuranceAmount`, `procurementLaw`, `procurementMethod`, `tenderNumber`, `tenderJustification`. Пользователь заполняет секции "Страхование" и "Закупки" в форме, но данные **молча выбрасываются** перед отправкой.

#### BUG-C9: [PERF] Загрузка всех 500 контрактов в память
`ContractListPage.tsx:63` — `getContracts({ size: 500 })`. Нет серверной пагинации. При 500+ контрактах → проблемы с производительностью.

#### BUG-C10: [LOGIC] Approval logic bug — convoluted stage transition
`ContractService.java:523-534` — `determineStatusAfterApproval()` для stage "lawyer":
```java
boolean mgmtRequired = !contractApprovalRepository
    .existsByContractIdAndStageAndStatus(contract.getId(), "management", ApprovalStatus.PENDING)
    || contractApprovalRepository
    .existsByContractIdAndStageAndStatus(contract.getId(), "management", ApprovalStatus.APPROVED);
```
Логика: `!pending || approved` — это **всегда true** когда `management` стадия не существует (оба вернут false, `!false || false` = true). Потенциально пропускает management approval для типов контрактов, где management required, но ещё не создан approval record.

#### BUG-C11: [DATA] ДС не обновляет сумму контракта
`ContractSupplementService.java:42-61` — создаёт supplement с `amountChange` и `newTotalAmount`, но **не обновляет** поля `amount`/`totalWithVat` родительского контракта. Сумма контракта в реестре не отражает подписанные дополнительные соглашения.

#### BUG-C12: [UX] Модальное окно "Изменить статус" — bypass state machine
`ContractDetailPage.tsx:536-561` — модальное окно позволяет выбрать ЛЮБОЙ из 8 статусов, включая невалидные переходы. Бэкенд валидирует `canTransitionTo()`, но пользователь видит все статусы и получает неинформативную ошибку 500. Нужно показывать только допустимые переходы.

#### BUG-C13: [UX] ContractFormPage — форма не сбрасывается при загрузке данных
`ContractFormPage.tsx:126-172` — `defaultValues` устанавливаются при инициализации формы, но `existingContract` загружается асинхронно через useQuery. В режиме редактирования форма отображается с пустыми полями до загрузки данных, и `useForm.reset()` не вызывается при получении данных.

### P3 — Medium

#### BUG-C14: [CODE] ContractSignWizard — 8 `as any` cast
`ContractSignWizard.tsx:59-68` — маппинг полей через `(contractData as any).title`, `(contractData as any).startDate` и т.д. Поля не совпадают с ContractResponse (`title` → `name`, `startDate` → `plannedStartDate`, `contractor` → `partnerName`).

#### BUG-C15: [UX] Dark mode — неполные стили
`ContractListPage.tsx:170,178` — `text-neutral-600` без `dark:text-neutral-400` в ячейках колонок Project и Type. Текст невидим на тёмном фоне.

#### BUG-C16: [CODE] BankGuaranteesPage — inline API вместо contractExtApi
`BankGuaranteesPage.tsx:49` — `apiClient.get('/bank-guarantees')` вместо `contractExtApi.listGuarantees()`. Эндпоинт `/bank-guarantees` не существует в ContractExtController (правильный путь: `/contracts/{contractId}/guarantees`). Страница всегда показывает fallback/mock данные.

#### BUG-C17: [CODE] ReclamationsPage — inline API вместо contractExtApi
`ReclamationsPage.tsx:48-49` — аналогично BankGuaranteesPage, использует прямой `apiClient.get` вместо `contractExtApi.listClaims()`.

#### BUG-C18: [SECURITY] GET endpoints без @PreAuthorize
`ContractController.java` — методы `list()`, `getById()`, `getApprovals()`, `getDashboardSummary()`, `listTypes()` не имеют `@PreAuthorize`. Любой аутентифицированный пользователь (даже VIEWER) может видеть все контракты.

**Примечание**: Для чтения контрактов это может быть допустимо, но dashboard и approval history должны быть ограничены.

#### BUG-C19: [CODE] Number field — confusing UX
`ContractFormPage.tsx:18,187` — Zod-схема требует `number` (required), пользователь видит поле ввода номера, но в `createMutation` поле `number` удаляется (`delete payload.number`), т.к. бэкенд генерирует номер автоматически (`CTR-00001`). В режиме создания пользователь заполняет номер зря.

#### BUG-C20: [ARCH] 5 cross-module imports
- `ContractService.java` → `finance.service.BudgetItemSyncService`, `project.repository.ProjectRepository`
- `ContractBudgetItemService.java` → `finance.domain.BudgetItem`, `finance.repository.BudgetItemRepository`, `finance.repository.BudgetRepository`

## 8. Положительные стороны

Чтобы аудит был объективным — вот что сделано хорошо:

1. **State Machine** — `ContractStatus.canTransitionTo()` — чистая, полная реализация всех допустимых переходов
2. **Budget Item Linking** — `ContractBudgetItemService` — серьёзная валидация: проверка количества, суммы, дубликатов, покрытия, пересчёт contracted amount
3. **Multi-tenant в core** — `ContractService` правильно использует `SecurityUtils.requireCurrentOrganizationId()` + `getContractOrThrow(id, organizationId)` во всех операциях
4. **Approval Workflow** — 3-стадийная система согласования с настраиваемыми стадиями per contract type
5. **JPA Specifications** — гибкая фильтрация через `ContractSpecification` builder
6. **Versioning** — поддержка версий контракта (parentVersionId, docVersion)
7. **Direction** — CLIENT/CONTRACTOR разделение с корректным UI (tabs)
8. **Rich entity model** — contractExt содержит полный набор: supplements, claims, SLA, guarantees, insurance, milestones, legal cases, tolerances
9. **Audit trail** — все CRUD-операции логируются через AuditService
10. **Good UX components** — ContractCoverageBar, ContractDirectionTabs, ProcurementCompliancePanel — продуманные компоненты
