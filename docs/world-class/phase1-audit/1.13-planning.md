## 2. Анализ по ролям

### 2.1 СМЕТЧИК — Расчёты и формулы

**EVM расчёты (EvmAnalyticsService.java):**
- ✅ EAC Method 1 (CPI): `EAC = BAC / CPI` — корректно
- ✅ EAC Method 2 (SPI×CPI): `EAC = AC + (BAC - EV) / (CPI × SPI)` — корректно
- ✅ EAC Method 4 (Independent): `IEAC = AC + (BAC - EV)` — корректно
- ✅ VAC: `BAC - EAC` — корректно
- ✅ TCPI: `(BAC - EV) / (EAC - AC)` — корректно, с div-by-zero protection
- ⚠️ EAC Method 3 (Bottom-up): `eacBottom = bac` — **ЗАГЛУШКА** (line 162). Комментарий: "For now, uses BAC as the managerial estimate placeholder". Bottom-up EAC должен агрегировать фактические EAC по каждому WBS элементу снизу вверх
- ⚠️ WBS-level EVM (`getWbsEvmBreakdown`): `AC = EV` (line 218). Комментарий: "WbsNode has no actualCost field; derive as ev for now". Это делает CV = 0 ВСЕГДА на уровне WBS — бесполезно для cost variance анализа
- ⚠️ Confidence Bands: `optimistic = EV × 1.1`, `pessimistic = EV × 0.9` — **наивная формула** (line 250-255). Реальные confidence bands используют Monte Carlo или стандартное отклонение исторических CPI/SPI

**CPM расчёты (WbsNodeService.java):**
- ✅ Forward pass (Kahn's algorithm topological sort) — корректно
- ✅ Backward pass с Late Start/Late Finish — корректно
- ✅ Total Float = LS - ES — корректно
- ✅ Free Float = min(ES_succ - EF_current - 1) для FS — корректно
- ✅ Все 4 типа зависимостей (FS, SS, FF, SF) с lag — корректно реализованы
- ✅ Critical path = nodes where TotalFloat = 0 — корректно
- ⚠️ Duration = calendar days (не рабочие дни). Нет учёта выходных, праздников, нерабочих дней. Для проекта 200 дней это даёт ошибку ~57 дней (28%)

**refreshEvmSnapshotFromTasks():**
- ✅ % выполнения из задач: `DONE / (TOTAL - CANCELLED) × 100` — корректно
- ✅ BAC из суммы бюджетных позиций — корректно
- ⚠️ PV = EV (line 312) — упрощение. Без baseline calendar PV ≈ EV, что делает SPI = 1.0 ВСЕГДА. Примечание есть в коде
- ⚠️ AC = 0 (line 313) — "из платежей/счетов: будет интегрировано в Phase 5". CPI = 1.0 ВСЕГДА. Примечание есть

## 8. Вердикт

**Planning модуль — один из самых глубоко проработанных модулей в системе.**

Backend: 77 файлов, 5844 LOC — полноценная система управления проектами с CPM, EVM, ресурсами, skills matching, multi-project allocation. Архитектурно это на уровне Primavera P6 lite.

Frontend: 8 страниц, 4642 LOC — хорошая визуализация WBS, EVM, ресурсов. Но **Gantt — read-only** (нет CRUD, нет стрелок зависимостей, нет drag), что делает его скорее отчётом, чем инструментом планирования.

**Критические проблемы** (блокируют production):
1. 3 entities без tenant isolation (EvmSnapshot, ScheduleBaseline, ResourceAllocation) — data leak
2. 29 GET endpoints без @PreAuthorize
3. EVM авторасчёт: AC=0, PV=EV → CPI/SPI всегда 1.0

**Для первого платящего клиента** нужно:
1. Fix tenant isolation (6h)
2. WBS CRUD UI (1d)
3. S-Curve как Recharts Line chart (4h)
4. Реальный AC из Invoice/Payment в EVM (1-2d)
