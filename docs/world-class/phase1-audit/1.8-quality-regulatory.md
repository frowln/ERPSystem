## 2. Оценка по ролям

### 👷 ПРОРАБ (полевая работа) — 7/10
**Плюсы:**
- Дефекты с привязкой к чертежу (DefectPlanViewPage) — draggable pins на поэтажном плане
- Чек-листы по видам работ (12 типов: бетонирование, сварка, гидроизоляция, электрика...)
- Фотофиксация (поля photoUrls на дефектах, чек-листах, инспекциях)
- Punch-list с приоритетами (LOW/MEDIUM/HIGH/CRITICAL) и комментариями

**Минусы:**
- QualityBoardPage на HTML5 DnD — **не работает на мобильных** (прораб на стройке)
- PermitBoardPage — **пустой**, нет API, нельзя отследить статус разрешений
- Нет offline режима для полевых проверок
- Фото — только URL, нет загрузки файлов через MinIO в quality module

### 📐 СМЕТЧИК (расчёты) — 6/10
**Плюсы:**
- Tolerance rules с min/max/nominal — стандартные допуски по СНиП
- Deviation расчёт: `|measured - nominal| / (max - nominal) * 100%`
- Quality Gate completion %: корректная формула (docs + quality + volume)

**Минусы:**
- Нет стоимости дефектов в агрегации (NonConformance.cost единичный, нет dashboard суммирования)
- Нет привязки стоимости переделок к бюджету проекта
- Tolerance reference — только текст, нет связи с ГОСТ/СНиП справочником

### 💰 БУХГАЛТЕР (финансовые операции) — 5/10
**Плюсы:**
- Prescription.fineAmount + correctiveActionCost — фиксация штрафов от надзорных органов
- PrescriptionDashboard.totalFines — суммарные штрафы по организации

**Минусы:**
- Штрафы НЕ связаны с бюджетом проекта (нет auto-entry в CashFlow)
- Нет интеграции с 1С для штрафных проводок
- Нет отчётности по затратам на качество (CoQ — Cost of Quality)

### 🔒 БЕЗОПАСНИК (security) — 5/10

**Критические проблемы:**

1. **ReportingDeadline — НЕТ organizationId** (P1): entity не имеет поля tenant isolation. Все дедлайны глобальные. Тенант A видит дедлайны тенанта B. `ReportingCalendarService` — все 4 метода (`findAll`, `findByFilters`, `findUpcoming`, `findOverdue`) возвращают данные всех организаций.
   - `regulatory/domain/ReportingDeadline.java` — нет поля organizationId
   - `regulatory/service/ReportingCalendarService.java:47-74` — нет org filter

2. **20+ GET endpoints без @PreAuthorize** (P2): любой authenticated user видит все данные quality + regulatory
   - Quality: QualityCheckController (2 GET), NonConformanceController (2 GET), QualityGateController (4 GET), QualityCertificateController (4 GET), QualityChecklistController (3 GET), MaterialInspectionController (1 GET), SupervisionEntryController (1 GET), ChecklistTemplateController (1 GET)
   - Regulatory: RegulatoryInspectionController (4 GET), ConstructionPermitController (2 GET), RegulatoryReportController (2 GET), PrescriptionController (3 GET)

3. **QualityCertificateController expired/expiring** — глобальные без org filter: `getExpiredCertificates()` и `getExpiringCertificates()` возвращают сертификаты всех организаций

**Хорошие паттерны:**
- Все CREATE/UPDATE/DELETE endpoints имеют @PreAuthorize с proper roles
- Tenant isolation работает в большинстве service-методов (SecurityUtils.requireCurrentOrganizationId())
- Soft delete + @Filter(tenantFilter) на всех quality entities
- PunchListService полностью tenant-isolated

### 🏗️ АРХИТЕКТОР (паттерны) — 7.5/10
**Сильные стороны:**
- **Quality Gate Engine** — умная система: 3 dimension evaluation (docs, quality, volume), template application по WBS level pattern, progression blocking
- **State machines** — NonConformanceStatus.canTransitionTo() + PrescriptionStatus валидация переходов
- **Code generation** — QC-#####, NCR-#####, QCL-#####, SE-#####, REG-#####, PRE-##### через PostgreSQL sequences
- **Checklist architecture** — template → execution instance → items pattern (reusable, extensible)
- **BIM Integration** — defect_bim_links (V226) с camera_position_json для 3D навигации
- **Material Inspection auto-integration** — ACCEPTED → auto StockMovement(RECEIPT) cross-module

**Проблемы:**
- Дублирование: DefectService (в quality) + Defect entity (отдельная от NonConformance) + OpsService defect CRUD — три code path для дефектов
- PrescriptionService + RegulatoryInspectionService.addPrescription() — два entry point для предписаний
- Inspection.prescriptionsJson (JSONB) дублирует данные Prescription entity (FK inspection_id)
- ReportingDeadline без organizationId — архитектурный дефект

## 5. Сильные стороны модуля

### Quality — лучшее в модуле
1. **Quality Gate Engine** (QualityGateService) — уникальная система: оценка по 3 измерениям (docs, quality, volume), привязка к WBS, шаблоны по уровням — **лучше Procore** (у Procore нет WBS-привязанных gates)
2. **Checklist Architecture** — template → execution → items с per-item pass/fail/na + photo + inspector — production-ready
3. **DefectPlanViewPage** — draggable pins на поэтажном плане, create modal прямо из клика — уровень PlanRadar
4. **Prescription Dashboard** — totalActive/Overdue/Completed/Fines + byStatus + byBodyType + approachingDeadline — полный обзор для руководителя

### Regulatory — лучшее в модуле
1. **6 надзорных органов** (ГИТ, Ростехнадзор, Стройнадзор, МЧС, Роспотребнадзор, Экологический) — полный охват российской специфики
2. **Appeal workflow** — isAppealWindowOpen() с валидацией сроков, appealFiled флаг, status APPEALED — уникально для ERP
3. **Reporting Calendar** — дедлайны + submissions + частота + channel (электронно/бумага/ЭДО) — хорошая модель (если починить tenant isolation)
4. **152-ФЗ Compliance** (отдельный модуль) — DataConsent, DSAR, PII Audit Log — профессиональный уровень

### Архитектурные достижения
1. **Code generation** через PostgreSQL sequences (QC-#####, NCR-#####, REG-#####...) — надёжно, concurrent-safe
2. **State machines** с валидацией переходов (NonConformanceStatus.canTransitionTo)
3. **Material Inspection → auto StockMovement** — cross-module интеграция при ACCEPTED
4. **BIM defect linking** (V226) — camera_position_json для 3D навигации к дефекту
5. **i18n** — ~95% покрытие, 150+ ключей quality.* + regulatory.*
