## НАЙДЕННЫЕ БАГИ

### P1 — Critical

#### BUG-S1: SpecificationService.listSpecifications() — НЕТ tenant isolation
**Суть**: Метод `listSpecifications()` строит Criteria Query с предикатами `deleted=false`, `projectId`, `status`, но **нет** предиката по `organizationId`. Хотя entity аннотирован `@Filter(tenantFilter)`, это работает только если Hibernate Session filter активирован глобально. Если нет — данные одного тенанта видны другому.

**Файл**: `SpecificationService.java:46-62`
```java
predicates.add(cb.equal(root.get("deleted"), false));
if (projectId != null) { predicates.add(cb.equal(root.get("projectId"), projectId)); }
if (status != null) { predicates.add(cb.equal(root.get("status"), status)); }
// Нет: predicates.add(cb.equal(root.get("organizationId"), currentOrgId));
```

**Сравнение**: `CompetitiveListService.list()` (строка 66-73) **правильно** фильтрует — вызывает `findByOrganizationIdAndDeletedFalse()` когда projectId == null. SpecificationService — нет.

**Влияние**: Cross-tenant data leak. Любой аутентифицированный пользователь может увидеть спецификации чужой организации (особенно без projectId фильтра).

**Приоритет**: P1 — security breach в SaaS-системе.

---

#### BUG-S2: Specification.name — UNIQUE constraint через ВСЕ организации
**Суть**: Поле `name` в entity Specification имеет `unique = true` (строка 39), а auto-generated имя берётся из глобальной PostgreSQL sequence `seq_spec_name`. Это значит:
1. Два тенанта **не могут** иметь спецификацию с одинаковым именем (ошибка при параллельном создании невозможна из-за sequence, но...)
2. Номера спецификаций **утекают** между тенантами — тенант видит SPEC-00500, значит на платформе создано 500+ спецификаций. Information leakage.
3. При масштабировании с sharding/partitioning по tenant — UNIQUE constraint ломает partition isolation.

**Файл**: `Specification.java:39`
```java
@Column(name = "name", nullable = false, unique = true, length = 50)
private String name;
```

**Решение**: UNIQUE constraint должен быть составным: `(organization_id, name)`. Sequence должна быть per-tenant.

**Влияние**: Privacy leak + architectural anti-pattern для multi-tenant SaaS.

**Приоритет**: P1 — блокирует multi-tenant production deployment.

#### BUG-S5: PDF upload — нет ограничения размера файла, DoS vector
**Файл**: `SpecificationPdfParserService.java:241`
```java
PDDocument doc = Loader.loadPDF(pdfStream.readAllBytes())
```

`readAllBytes()` загружает весь PDF в память. Нет проверки размера файла в контроллере (только `isEmpty()` check). Злоумышленник может загрузить PDF 500MB+ → OOM → сервис падает.

**Файл**: `SpecificationController.java:171-178` — только проверка `file.isEmpty()`, нет проверки `file.getSize()`.

#### BUG-S13: N+1 в SpecificationService.listSpecifications()
**Файл**: `SpecificationService.java:58-61`
```java
return specificationRepository.findAll(spec, pageable).map(s -> {
    long cnt = specItemRepository.countBySpecificationIdAndDeletedFalse(s.getId()); // N+1!
    return SpecificationListResponse.fromEntity(s, cnt);
});
```

На каждую спецификацию — отдельный COUNT запрос. При 100 спецификациях на странице — 101 запрос.

**Решение**: `@Query("SELECT s, (SELECT COUNT(i) FROM SpecItem i WHERE i.specificationId = s.id AND i.deleted = false) FROM Specification s")` или пост-загрузка через `IN (ids)`.

#### BUG-S20: detectItemType() — default "EQUIPMENT" bias
**Файл**: `SpecificationPdfParserService.java:628-634`
```java
private String detectItemType(String name, String brand) {
    for (String kw : MATERIAL_KW)  { if (text.contains(kw)) return "MATERIAL"; }
    for (String kw : EQUIPMENT_KW) { if (text.contains(kw)) return "EQUIPMENT"; }
    return "EQUIPMENT";  // Default bias

```

MATERIAL проверяется первым — если item содержит и material, и equipment keyword, будет «MATERIAL». Default «EQUIPMENT» — bias для неизвестных позиций. Порядок проверки и default спорные.

#### BUG-S21: addEntry() позволяет unitPrice = 0 и quantity = 0
**Файл**: `CompetitiveListService.java:199-200`

## ИТОГ

### Что модуль делает хорошо:
1. **PDF-парсер ГОСТ Р 21.1101** — ColumnAwareStripper с 40+ единицами, 50+ аббревиатурами разделов ИОС. Лучший в классе.
2. **Конкурентные листы (КЛ)** — 5-step state machine, weighted scoring (50/25/15/10), auto-ranking, auto-select, apply to КП. Лучше Procore.
3. **Аналоги материалов** — уникальная функция (нет у конкурентов): substitution types, quality ratings, approval workflow.
4. **Версионирование** — docVersion + copy items + parentVersionId + version diff modal.
5. **Pricing chain** — Spec → КЛ → КП с backlink budgetItemId. Полная цепочка.
6. **Import conflict resolution** — при повторном импорте PDF/XLSX: new/changed/unchanged с выбором действия (replace/add/sum/skip).
7. **State machines** — чистые enum canTransitionTo() для Specification и CompetitiveList.
8. **Audit trail** — logCreate/logUpdate/logStatusChange на всех операциях.

### Что нужно исправить немедленно:
1. **Tenant isolation** — listSpecifications() без organizationId + getSpecificationOrThrow() без org check
2. **Global UNIQUE constraint** на name — ломает multi-tenant isolation
3. **8 GET endpoints без @PreAuthorize** — любой authenticated user видит все данные
4. **File upload без size limit** — DoS vector

### Чего критически не хватает:
1. **Тесты** — 9 тестов на ~46 backend файлов. PDF-парсер (680 LOC) без единого теста
2. **Экспорт спецификации** — нет PDF/Excel экспорта
3. **Тендерная рассылка** — кнопка есть, email не отправляется
4. **Batch operations** — sequential HTTP calls для import/push
