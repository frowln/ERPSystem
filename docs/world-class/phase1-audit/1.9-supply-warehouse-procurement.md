## 2. Архитектура бэкенда

### 2.1 Warehouse — СИЛЬНАЯ архитектура

**Tenant isolation**: Полная. Все entities имеют `@Filter(name="tenantFilter")`. Все services вызывают `SecurityUtils.requireCurrentOrganizationId()`. Есть tenant validators: `validateProjectTenant()`, `validateLocationTenant()`, `validateUserTenant()`.

**State machines**:
- StockMovementStatus: DRAFT→CONFIRMED→DONE→CANCELLED (с `canTransitionTo()`)
- InventoryCheckStatus: PLANNED→IN_PROGRESS→COMPLETED→CANCELLED
- WarehouseOrderStatus: DRAFT→CONFIRMED→CANCELLED
- LimitFenceSheetStatus: ACTIVE→CLOSED

**Бизнес-логика StockMovementService (666 LOC)**:
- RECEIPT/RETURN → addStock(destination) + createBatch() (ФСБУ 5/2019)
- ISSUE/WRITE_OFF → subtractStock(source)
- TRANSFER → subtract(source) + add(destination)
- ADJUSTMENT → adjustStock(location)
- Reverse операции для cancelMovement()
- Auto-receipt из PurchaseOrder: `createAutoReceiptFromDelivery()`
- Weighted-average price via StockBatch (средневзвешенная себестоимость)

**Constraints (DDL)**:
- UNIQUE: org+code (materials, locations)
- UNIQUE: org+number (movements, orders, sheets, checks)
- UNIQUE: org+material+location (stock_entries, stock_limits)
- CHECK: quantity ≥ 0, price ≥ 0
- FK: material_id, location_id, movement_id

**Вывод**: Warehouse — **лучший по архитектуре модуль** в проекте. Крепкий tenant isolation, корректные state machines, ФСБУ compliance.

### 2.2 Procurement — КРИТИЧЕСКАЯ проблема дублирования

**BLOCKER: Два модуля на одну таблицу `purchase_orders`**:

| | procurement | procurementExt |
|---|---|---|
| Entity class | `ProcurementPurchaseOrder` | `PurchaseOrder` |
| Table | `purchase_orders` | `purchase_orders` (SAME!) |
| Status enum | DRAFT, PENDING_APPROVAL, APPROVED, SENT_TO_SUPPLIER, PARTIALLY_DELIVERED, DELIVERED, CANCELLED | DRAFT, SENT, CONFIRMED, PARTIALLY_DELIVERED, DELIVERED, INVOICED, CLOSED, CANCELLED |
| Service | `PurchaseOrderService` (procurement) | `PurchaseOrderService` (procurementExt) |
| Repository | `PurchaseOrderRepository` | `PurchaseOrderRepository` (NO QUALIFIER!) |

**Проблемы**:
1. Когда procurement сохраняет `SENT_TO_SUPPLIER`, procurementExt не может распарсить (нет такого enum) → **Exception**
2. Spring не может автоматически разрешить `PurchaseOrderRepository` без qualifier → **BeanDefinitionStoreException**
3. Два service слоя пытаются управлять одними данными → **race conditions**

**ProcurementService (629 LOC)**: Хорошо структурирован. 10 статусных переходов с валидацией. `createFromSpecItems()` — автосоздание заявки из спецификации. Tenant isolation через `SecurityUtils.requireCurrentOrganizationId()`.

**PurchaseOrderService (459 LOC)**: Budget control (P0-WAR-2). Auto-receipt при доставке. Но `sendPriceRequests()` в ProcurementDirectoryService — **ЗАГЛУШКА** (только логирует).

### 2.3 Dispatch — НЕТ tenant isolation

**Два backend модуля для одного домена**:
- `ops.DispatchController` (`/api/dispatch`) — bean name `opsDispatchController`
- `procurementExt.DispatchController` (`/api/procurement-ext`) — bean name `procExtDispatchController`

**Разные status enum**:
- ops: PLANNED→DISPATCHED→IN_TRANSIT→DELIVERED→CANCELLED
- procurementExt: NEW→CONFIRMED→DISPATCHED→IN_TRANSIT→DELIVERED→CANCELLED

**Tenant isolation**: `dispatch_orders` имеет `project_id` (FK), но **НЕТ `organization_id`**. Фильтрация зависит от project access control, но прямой запрос по UUID обходит это. `DispatchRoute` также без org_id.

**Sequence**: `DispatchService.orderSequence = AtomicLong(System.currentTimeMillis() % 100000)` — **сбрасывается при рестарте JVM**. (Уже зафиксировано в BUG #76 из сессии 1.6, но не исправлено.)
