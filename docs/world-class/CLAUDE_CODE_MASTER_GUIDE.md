# PRIVOD → WORLD CLASS: Полная инструкция и промпты для Claude Code

---

## КАК ЭТО РАБОТАЕТ

### Проблема
Проект — 640k строк кода. Claude Code видит ~200k токенов за сессию.
Если дать ему "сделай всё" — он утонет и выдаст мусор.

### Решение: система "внешней памяти"
Claude Code работает сессиями. Каждая сессия — конкретная задача.
Результат каждой сессии — файл с выводами.
Следующая сессия читает предыдущие выводы.
Так знания НАКАПЛИВАЮТСЯ, а не теряются.

### Структура файлов (создай перед началом)

```
docs/
  world-class/
    00-INDEX.md              ← МОЗГ: сводка всего, обновляется каждой сессией
    01-GOLD-STANDARD.md      ← Эталон (файл, который я тебе уже дал)
    phase1-audit/            ← Результаты аудита
    phase2-plan/             ← Планы улучшений
    phase3-execution/        ← Логи выполнения
    BUGS.md                  ← Все найденные баги (накопительно)
    IMPROVEMENTS.md          ← Все идеи по улучшению (накопительно)
    COMPETITORS.md           ← Анализ конкурентов (накопительно)
```

### Порядок работы

```
ФАЗА 1: АУДИТ (Claude Code изучает и записывает, НЕ пишет код)
  ├── Сессия 1.0: Первичный обзор всего проекта
  ├── Сессии 1.1–1.N: Глубокий аудит по модулям (по одному за сессию)
  ├── Сессия 1.X: Аудит архитектуры и инфраструктуры
  ├── Сессия 1.Y: Анализ конкурентов через интернет
  └── Сессия 1.Z: Итоговый сводный отчёт

ФАЗА 2: ПЛАНИРОВАНИЕ (Claude Code строит план, НЕ пишет код)
  ├── Сессия 2.0: Приоритизация и roadmap
  └── Сессия 2.1: Декомпозиция на конкретные задачи

ФАЗА 3: ВЫПОЛНЕНИЕ (Claude Code пишет код)
  ├── Сессии 3.X: По одной задаче за сессию
  └── Каждая сессия заканчивается обновлением логов
```

```markdown
- Багов найдено: 0
- Улучшений предложено: 0
- Модулей проверено: 0 / ~95
```

```
3. Если есть предыдущие аудиты в docs/world-class/phase1-audit/ — прочитай их для контекста

ЗАДАЧА:
Глубокий аудит модуля [НАЗВАНИЕ МОДУЛЯ]. Изучи ВСЁ: бэкенд (domain, repository, service, web, dto), фронтенд (pages, components, hooks, types, api), тесты, миграции.

АНАЛИЗИРУЙ КАК:
- СМЕТЧИК: правильно ли считаются цифры? Корректны ли формулы? Соответствуют ли расчёты российским нормативам?
- ПРОРАБ: удобно ли пользоваться на стройке? Есть ли всё, что нужно в поле?
- БУХГАЛТЕР: корректны ли финансовые операции? Верны ли проводки? Правильная ли генерация КС-2/КС-3?

- АРХИТЕКТОР: правильные ли паттерны? Масштабируемо ли? Нет ли god-classes?
- UX ДИЗАЙНЕР: логичен ли интерфейс? Есть ли empty states? Loading states? Error handling?
- ПРОДАКТ МЕНЕДЖЕР: есть ли всё, что нужно пользователю? Чего не хватает по сравнению с конкурентами?
- QA ИНЖЕНЕР: достаточно ли тестов? Покрыты ли edge cases? Есть ли баги?

ЧТО ИСКАТЬ:
- Баги (логические ошибки, неправильные расчёты, сломанный UI)
- Заглушки (TODO, mock data, placeholder, hardcoded values)
- Дыры в безопасности (отсутствие @PreAuthorize, SQL injection, XSS)
- Отсутствующий функционал (по сравнению с GOLD-STANDARD)
- Проблемы UX (нет валидации, нет обработки ошибок, нет loading states)
- Проблемы производительности (N+1 queries, отсутствие пагинации, лишние запросы)
- Нарушение архитектуры (импорт между модулями, бизнес-логика в контроллере)
- Отсутствие тестов для критичной логики
- Некорректные российские формы (КС-2, М-29 и т.д.)

НА ВЫХОДЕ создай docs/world-class/phase1-audit/1.N-[module-name].md:
- Оценка модуля: 1-10
- Статус: production-ready / needs-work / major-rework / stub
- Найденные баги (с файлом и строкой)
- Отсутствующий функционал
- Проблемы безопасности
- Проблемы UX
- Проблемы архитектуры
- Конкретные рекомендации по улучшению (с приоритетом: critical / high / medium / low)

ОБЯЗАТЕЛЬНО ОБНОВИ:
- docs/world-class/BUGS.md — добавь найденные баги
- docs/world-class/IMPROVEMENTS.md — добавь предложения
- docs/world-class/00-INDEX.md — отметь модуль как проверенный, обнови статистику

НЕ ПИШИ КОД. Только анализ.
```

## СОВЕТЫ ПО ИСПОЛЬЗОВАНИЮ

### Как получить максимум от каждой сессии

1. **Один промпт = одна задача.** Не объединяй "проанализируй + исправь + оптимизируй" в одном промпте.

2. **Не торопись с Фазой 3.** Потрать 2-3 дня на полный аудит (Фаза 1). Это сэкономит недели в Фазе 3, потому что ты будешь знать, что чинить.

3. **Проверяй выводы Claude Code.** Он может ошибаться. Особенно в оценке "working vs stub". Открой модуль сам, убедись.

4. **INDEX — твой главный инструмент.** Если INDEX не обновлён — следующая сессия начнётся без контекста. Следи за этим.

5. **Если сессия пошла не туда** — останови, напиши "Стоп. Перечитай свою задачу. Ты отклонился". Это работает.

6. **Для больших модулей — разбивай.** Если модуль finance имеет 20+ файлов — раздели на 2 сессии: бэкенд и фронтенд.

7. **Веб-поиск.** Напоминай Claude Code использовать поиск: "Загугли, как Procore реализует [X]", "Найди актуальную форму КС-2 по ГОСТ".

### Порядок запуска промптов

```
День 1:     Промпт 1.0 (обзор проекта)
День 1-5:   Промпт 1.N × 15-20 раз (по модулю за сессию, начни с самых важных)
День 3:     Промпт 1.X (архитектура — можно параллельно с модулями)
День 4:     Промпт 1.Y (конкуренты)
День 5:     Промпт S.2 (безопасность)
День 5:     Промпт S.3 (российские формы)
День 6:     Промпт 1.Z (итоговый отчёт)
День 7:     Промпт 2.0 (roadmap)
День 8+:    Промпт 3.X (выполнение задач по плану)
Постоянно:  Промпт 3.F (свободное улучшение, когда есть время)
```

### Красные флаги — когда Claude Code тупит

- **Повторяет одно и то же** → "Ты уже написал это. Двигайся дальше."
- **Не читает файлы** → "Прочитай файл X прямо сейчас. Процитируй первые 3 строки."
- **Пишет код вместо анализа** → "НЕ ПИШИ КОД. Только анализ. Записывай в markdown."
- **Теряет контекст** → "Перечитай docs/world-class/00-INDEX.md. Что там написано?"
- **Делает слишком много за раз** → "Стоп. Сфокусируйся ТОЛЬКО на модуле X."
