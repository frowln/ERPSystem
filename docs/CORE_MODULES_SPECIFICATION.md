## 1. Общая архитектура и взаимосвязи

### 1.1 Иерархия модулей

Все перечисленные модули образуют единую экосистему, где **Финансовая модель (ФМ)** — центральный хаб ("single source of truth"), а остальные модули — спутники, которые питают ФМ данными и берут из неё контрольные цифры.

```

                        │  ПРОЕКТНАЯ ДОКУ- │

         │ (Что купить) │ │(Сколько  │  │  РАБОТ (Когда)   │
```

### 1.2 Ключевой принцип: единая структура

**Твоя идея верна на 100%**: все модули (ФМ, Договоры, КП, Сметы, Спецификация) имеют **одинаковую структуру** — разделы и позиции, разделённые на Материалы и Работы. Это не просто удобство, а стандарт индустрии — так работают Procore, 1С:УСО, АЛТИУС и другие ведущие системы.

Однако "одинаковая структура" не означает дублирование данных. Все модули **наследуют** структуру из ФМ (или из сметы, которая импортируется в ФМ), а не создают свою. Это обеспечивает:
- Прослеживаемость: от любого платежа можно дойти до позиции ФМ
- Контроль: бюджет всегда показывает, сколько законтрактовано/потрачено
- Защиту маржи: отслеживание маржи в реальном времени

### 1.3 Связи между модулями (data flow)

| Откуда | Куда | Что передаётся |
|--------|------|-----------------|
| Смета → ФМ | Плановые цифры | Сметная стоимость по позициям → столбец "Сметная цена" |
| Спецификация → ФМ | Объёмы материалов | Количество и техтребования материалов |
| Спецификация → Закупки | Что покупать | Позиции спецификации → заявки на закупку |
| КП → ФМ | Себестоимость | Утверждённая цена из КП → столбец "Себестоимость" в ФМ |
| Счета → КП | Цены поставщиков | Выбранный счёт → себестоимость позиции КП |
| ФМ → Договоры | Позиции и цены | Позиции ФМ → приложения к договорам |
| Договоры → ФМ | Законтрактованные суммы | Сумма договора → столбец "Законтрактовано" |
| КС-2/КС-3 → ФМ | Факт выполнения | Принятые объёмы → столбец "Факт" |
| Оплаты → ФМ | Факт оплаты | Оплаченные суммы → столбец "Оплачено" |

## 2. Финансовая модель (Бюджет)

### 2.1 Разделение на Материалы и Работы

**Твоя идея:** Разделить ФМ на две вкладки — Материалы и Работы.

**Уточнение по результатам исследования:** Помимо двух основных категорий, лучшие практики (Procore, Sage 300, 1С:УСО) выделяют **6 типов затрат**:

| Тип | Описание | Примеры |
|-----|----------|---------|
| **MATERIALS** (Материалы) | Физические материалы | Бетон, арматура, кабель, трубы |
| **EQUIPMENT** (Оборудование) | Оборудование к монтажу | Трансформаторы, насосы, щиты |
| **LABOR** (Работы) | Стоимость работ | Монтаж, бетонирование, сварка |
| **SUBCONTRACT** (Субподряд) | Работы субподрядчиков | Полный комплекс от субподрядчика |
| **OVERHEAD** (Накладные) | Накладные расходы | НР по МДС 81-33.2004 |
| **OTHER** (Прочее) | Прочие затраты | Транспорт, временные сооружения |

**Рекомендация:** В UI показывать две основные вкладки **Материалы** и **Работы** (как ты и хотел), но внутри каждой позиции хранить тип затрат (category). Это уже реализовано в текущем коде (`BudgetItem.category`).

> **Важно:** Оборудование — отдельная категория, потому что у него другие сроки поставки, другие правила закупки и другая наценка. Но в UI оно может показываться на вкладке "Материалы" с фильтром "Оборудование".

### 2.2 Разделы (секции)

**Твоя идея:** Разделы сейчас фиксированные (ЭМ, ЭО и т.д.). Нужно сделать кнопку "Добавить раздел", чтобы выбирать нужные.

**Это абсолютно правильно.** Все серьёзные конкуренты делают разделы настраиваемыми. Вот почему:

1. **Разные типы объектов** — разные разделы. Жилой дом не имеет раздела "ЭО" (тяжёлое электрооборудование), но завод — имеет. Дорожный проект имеет совсем другие разделы.
2. **У каждой компании** своя номенклатура разделов.
3. **Засорение** интерфейса пустыми разделами снижает удобство.

**Рекомендуемая реализация:**

```
Конфигуратор разделов:
├── Шаблоны (пресеты по типам объектов)
│   ├── Жилищное строительство (АР, КР, ЭМ, ЭО, ОВ, ВК, СС, ПС)
│   ├── Промышленное строительство (АР, КР, КМ, ЭМ, ЭО, ОВ, ВК, АПТ, ТМ, АСУ)
│   ├── Дорожное строительство (ЗП, ДО, ВО, ИО, Мосты)
│   └── Инженерные сети (НВК, НКн, ТС, ГС, ЭС)
├── Настройка раздела
│   ├── Код (например "ЭМ")
│   ├── Название (например "Электромонтажные работы")
│   ├── Ставка НР (накладные расходы, %)
│   ├── Ставка СП (сметная прибыль, %)
│   ├── Порядок сортировки
│   └── Цвет (для графиков/отчётов)
└── Кнопки управления
    ├── "Добавить раздел" → выбор из справочника или ввод нового
    ├── "Применить шаблон" → загрузка пресета
    └── "Скрыть пустые" → toggle
```

**Стандартный справочник разделов для строительства:**

| Код | Название | Категория |
|-----|----------|-----------|
| АР | Архитектурные решения | Общестрой |
| КР / КЖ | Конструкции железобетонные | Общестрой |
| КМ | Конструкции металлические | Общестрой |
| ЭМ | Электромонтажные работы | Инженерия |
| ЭО | Электрооборудование | Инженерия |
| ОВиК | Отопление, вентиляция, кондиционирование | Инженерия |
| ВК | Водоснабжение и канализация | Инженерия |
| СС | Слаботочные системы | Инженерия |
| ПС | Пожарная сигнализация | Безопасность |
| АПТ | Автоматическое пожаротушение | Безопасность |
| ТМ | Теплоснабжение | Инженерия |
| НВК | Наружные сети водоснабжения | Наружные |
| БЛ | Благоустройство | Наружные |
| ГП | Генплан | Общестрой |

### 2.3 Столбцы финансовой модели

**Твоя идея:** Три столбца — Себестоимость, Сметная цена, Цена заказчику.

**Это правильная база, но нужно расширить.** Вот полная таблица столбцов по лучшим практикам:

#### Основные ценовые столбцы (для каждой позиции)

| Столбец | Назначение | Источник данных |
|---------|------------|-----------------|
| **Ед. изм.** | Единица измерения | шт, м, м², м³, т, кг, компл |
| **Количество** | Объём | Из сметы / спецификации |
| **Себестоимость (за ед.)** | Фактическая закупочная цена | Из КП (выбранный счёт) |
| **Себестоимость (всего)** | Кол-во × себест. за ед. | Авторасчёт |
| **Сметная цена (за ед.)** | Цена по нормативам | Из сметы (ГЭСН/ТЕР/ФЕР + индексы) |
| **Сметная цена (всего)** | Кол-во × сметная за ед. | Авторасчёт |
| **Цена заказчику (за ед.)** | Контрактная цена | Из договора с заказчиком |
| **Цена заказчику (всего)** | Кол-во × цена заказчику | Авторасчёт |
| **Маржа (за ед.)** | Цена заказч. − Себестоимость | Авторасчёт |
| **Маржа (всего)** | Итоговая маржа | Авторасчёт |
| **Маржа (%)** | (Цена заказч. − Себест.) / Цена заказч. × 100 | Авторасчёт |

#### Столбцы исполнения (для отслеживания прогресса)

| Столбец | Назначение | Обновляется из |
|---------|------------|----------------|
| **Законтрактовано** | Сумма подписанных договоров | Договоры (статус ≥ SIGNED) |
| **Выполнено (факт)** | Принятые объёмы | КС-2 акты / накладные поставки |
| **Выставлено счетов** | Сумма выставленных счетов | Счета (статус ≥ APPROVED) |
| **Оплачено** | Фактически оплаченная сумма | Платежи (статус = PAID) |
| **Остаток (незаконтр.)** | Бюджет − Законтрактовано | Авторасчёт |
| **Прогноз (итого)** | Факт + Оставшиеся обязательства | Авторасчёт |

### 2.4 Валидации

**Твоя идея:** Цена заказчика не должна превышать сметную стоимость.

**Уточнение:** Это правило зависит от типа контракта:

| Тип контракта | Правило | Реализация |
|---------------|---------|------------|
| **Госконтракт (44-ФЗ)** | Цена заказчику ≤ Сметная цена | **Жёсткая блокировка** — система не даёт сохранить |
| **Коммерческий контракт** | Цена заказчику ≤ Сметная цена | **Мягкое предупреждение** — жёлтый индикатор, можно обосновать и сохранить |

**Дополнительные валидации:**

| Правило | Тип | Описание |
|---------|-----|----------|
| Себестоимость ≤ Цена заказчику | Предупреждение | Если нарушено — позиция убыточна (красный) |
| Маржа ≥ мин. порог (настраивается) | Предупреждение | Ниже порога = жёлтый, ниже 0% = красный |
| Сумма позиций = Сумма договора | Жёсткое | Отклонение ± 0.01₽ |
| Цены ≥ 0 | Жёсткое | Отрицательные цены запрещены |
| НДС единообразен | Жёсткое | Все цены либо с НДС, либо без |

### 2.5 Версионирование и снимки

**Три уровня версионирования (по Procore + 1С:УСО):**

1. **Версии бюджета** (крупные ревизии)
   - v1.0 — Начальный бюджет (по тендерной смете)
   - v2.0 — После рабочей документации
   - v3.0 — После крупного изменения объёмов

2. **Снимки** (фиксация на дату)
   - Ежемесячный автоматический снимок
   - Перед совещанием с заказчиком
   - Перед подписанием ДС (допсоглашения)

3. **Журнал изменений** (каждая правка)
   - Кто, когда, что изменил, обоснование

> **Уже реализовано:** `BudgetSnapshot` и `SnapshotComparison` — нужно лишь добавить автоматизацию снимков и визуальный diff.

## 3. Договоры

### 3.1 Разделение на Заказчика и Подрядчиков

**Твоя идея верна.** Договоры делятся на два направления:

| Направление | Назначение | Финансовый смысл |
|-------------|------------|------------------|
| **С заказчиком** (CLIENT) | Что нам платят | Доход / Выручка |
| **С подрядчиками/поставщиками** (CONTRACTOR) | Что мы платим | Расход / Себестоимость |

**Дополнительная классификация подрядных договоров:**

| Подтип | Описание | Пример |
|--------|----------|--------|
| SUPPLY | Договор поставки материалов | ООО "МеталлТорг" — арматура |
| SUBCONTRACT | Договор субподряда (работы) | ООО "ЭлектроМонтаж" — ЭМ работы |
| RENTAL | Аренда техники/оборудования | Кран, строительные леса |
| SERVICE | Услуги (проектирование, изыскания) | Геодезическая съёмка |

> **Уже реализовано:** Поле `contractDirection: 'CLIENT' | 'CONTRACTOR'` существует в текущих типах.

### 3.2 Структура договора = Структура ФМ

**Твоя идея верна.** Договор повторяет структуру ФМ: те же разделы (ЭМ, ЭО...), те же позиции, те же столбцы. Но есть важные отличия:

**Договор с заказчиком:**
- Позиции = позиции ФМ, столбец "Цена заказчику"
- Приложение к договору генерируется из ФМ
- Сумма договора = сумма всех "Цена заказчику (всего)"

**Договор с подрядчиком:**
- Позиции = подмножество позиций ФМ, столбец "Себестоимость"
- Один договор может покрывать часть позиции (см. п. 3.3)
- Сумма договора влияет на столбец "Законтрактовано" в ФМ

### 3.3 Одна позиция ФМ → Несколько договоров (N:M связь)

**Твоя идея абсолютно правильна.** Это ключевой сценарий реальной жизни. Примеры:

**Пример 1: Материалы**
```
Позиция ФМ: Арматура А500 d12, 100 тонн, бюджет 85,000 ₽/т = 8,500,000 ₽

Договор №1: ООО "НЛМК"    — 50 т × 83,000 = 4,150,000 ₽
Договор №2: ООО "Северсталь" — 30 т × 87,000 = 2,610,000 ₽
Договор №3: ИП Иванов      — 20 т × 89,000 = 1,780,000 ₽
────────────────────────────────────────────────────────────
Итого законтрактовано: 100 т                   8,540,000 ₽
Бюджет:                                        8,500,000 ₽
Перерасход:                                       40,000 ₽ ⚠️
```

**Пример 2: Работы**
```
Позиция ФМ: Монтаж металлоконструкций, 250 тонн

Договор №1: ООО "СтальМонтаж" — 150 т (корпус А)
Договор №2: ООО "МеталлСтрой" — 100 т (корпус Б)
```

**Также работает в обратную сторону** — один договор может покрывать несколько позиций ФМ:

```
Договор субподряда с ООО "ЭлектроМонтаж":
├── Позиция ФМ: Монтаж кабельных трасс → 2,000,000 ₽
├── Позиция ФМ: Монтаж щитового оборудования → 3,500,000 ₽
└── Позиция ФМ: Пусконаладочные работы → 1,200,000 ₽
Итого договор: 6,700,000 ₽
```

**Связующая таблица (уже есть: `ContractBudgetItem`):**

```
ContractBudgetItem {
  id: UUID
  contractId: UUID        // какой договор
  budgetItemId: UUID      // какая позиция ФМ
  allocatedQuantity: number  // сколько единиц из ЭТОГО договора
  allocatedAmount: number    // на какую сумму
  coveragePercent: number    // % покрытия позиции ФМ
}
```

**Вычисляемые поля на позиции ФМ:**

| Поле | Формула | Назначение |
|------|---------|------------|
| Законтрактовано | SUM(ContractBudgetItem.allocatedAmount) по подписанным договорам | Сколько уже обеспечено договорами |
| Незаконтрактовано | Бюджет − Законтрактовано | Сколько ещё нужно законтрактовать |
| % покрытия | Законтрактовано / Бюджет × 100 | Степень обеспеченности |

### 3.4 Статусы договоров

```
DRAFT → ON_APPROVAL → APPROVED → SIGNING → SIGNED → ACTIVE → COMPLETED
                                                       ↓
                                                  SUSPENDED
                                                  TERMINATED
                                                  DISPUTED
```

| Статус | Влияние на ФМ |
|--------|---------------|
| DRAFT | Нет — не видно в бюджете |
| ON_APPROVAL | Нет (опционально: мягкое резервирование) |
| APPROVED | Плановое обязательство (пунктиром) |
| SIGNED | **Законтрактовано** — суммы попадают в ФМ |
| ACTIVE | Законтрактовано + накопление факта |
| COMPLETED | Финальные суммы, удержания возвращены |
| TERMINATED | Обязательства снимаются по неисполненной части |

### 3.5 Допсоглашения (Change Orders)

Любое изменение подписанного договора оформляется как **ДС (Дополнительное соглашение)**:

```
ChangeOrder {
  contractId: UUID
  number: "ДС-1"
  reason: string
  items: [
    {
      budgetItemId: UUID
      changeType: ADD | MODIFY | REMOVE
      originalQty → newQty
      originalPrice → newPrice
      amountDelta: +/- number
    }
  ]
  status: DRAFT | APPROVED | SIGNED
}
```

**При подписании ДС** → автоматически обновляются суммы в ФМ:
- Если ДС увеличивает бюджет → система проверяет наличие резерва
- Если резерва нет → требуется согласование бюджетного комитета

### 3.6 Специфика для госконтрактов (44-ФЗ)

| Требование | Реализация |
|------------|------------|
| Цена контракта ≤ НМЦК | Валидация при создании |
| Изменения не более 10% | Автоматический контроль суммы ДС |
| Пени: 1/300 ставки ЦБ в день | Авторасчёт неустойки |
| Обеспечение 5-30% | Поле "гарантия" в договоре |
| Аванс ≤ 30% | Валидация графика платежей |

## 4. Коммерческое предложение / Себестоимость (КП)

### 4.1 Назначение КП

**Твоя идея:** КП — это раздел, который повторяет структуру ФМ (разделы, позиции, Материалы/Работы), и содержит столбец "Себестоимость". Заполняя его, стоимость попадает в столбец "Себестоимость" ФМ.

**Это правильно, с важным уточнением:** КП — это не просто "заполнение столбца". Это **процесс определения реальной себестоимости** через сбор и сравнение ценовых предложений от поставщиков. КП — промежуточное звено между счетами (от поставщиков) и ФМ.

```
Счета от поставщиков → КП (выбор лучшей цены) → ФМ (столбец "Себестоимость")
                        ↑                          ↑
                  утверждение                  только после
                  менеджером                   утверждения КП
```

### 4.2 Структура КП

КП зеркалирует ФМ:

```
КП по проекту "ЖК Солнечный"
├── Вкладка: МАТЕРИАЛЫ
│   ├── Раздел ЭМ
│   │   ├── Кабель ВВГнг 3×2.5, 15,000 м — себест. 45 ₽/м [из счёта №147]
│   │   ├── Автомат ABB S201 16А, 500 шт — себест. 320 ₽/шт [из счёта №152]
│   │   └── ...
│   ├── Раздел ОВ
│   │   └── ...
│   └── Раздел ВК
│       └── ...
├── Вкладка: РАБОТЫ
│   ├── Раздел ЭМ
│   │   ├── Прокладка кабеля — себест. 120 ₽/м [из КП субподрядчика]
│   │   └── ...
│   └── ...
└── Итоги
    ├── Всего себестоимость:     42,300,000 ₽
    ├── Всего цена заказчику:    55,000,000 ₽
    └── Маржа:                   12,700,000 ₽ (23.1%)
```

### 4.3 Откуда берётся себестоимость

**Для материалов (твоя идея верна):**

1. В систему вносятся **счета от поставщиков** (раздел "Счета")
2. Для каждой позиции КП система подбирает подходящие счета (smart matching)
3. Закупщик/менеджер выбирает самый выгодный
4. Выбранная цена становится себестоимостью позиции КП

**Иерархия источников цены (от менее точной к более точной):**

```
Сметная цена (ГЭСН/ТЕР + индексы)    — начальный placeholder
    ↓
Рыночная цена (мониторинг рынка)       — ориентир
    ↓
КП от поставщика (коммерческое предл.) — конкретное предложение
    ↓
Счёт на оплату (утверждённый)          — зафиксированная цена
    ↓
Факт оплаты                            — реальная себестоимость
```

**Для работ:**

Здесь два источника себестоимости:

**А) Субподрядчики (передача работ на субподряд):**
```
Работа: "Монтаж металлоконструкций, 250 т"

Субподрядчик А: 45,000 ₽/т = 11,250,000 ₽
Субподрядчик Б: 52,000 ₽/т = 13,000,000 ₽
Субподрядчик В: 41,000 ₽/т = 10,250,000 ₽  ← Лучшая цена

Сметная цена: 55,000 ₽/т
Себестоимость: 41,000 ₽/т (выбрали субподрядчика В)
Маржа: 25.5%
```

**Б) Собственные силы (хозяйственный способ):**
```
Работа: "Монтаж металлоконструкций, 250 т"

Расчёт себестоимости:
  ФОТ рабочих: ГЭСН норма 12.8 чел·ч/т × тариф = X ₽
  Механизмы: кран + сварка = Y ₽
  Накладные расходы: 112% от ФОТ = Z ₽
  Сметная прибыль: 65% от ФОТ = W ₽

  Итого себестоимость: X + Y + Z + W
```

> **Рекомендация:** Для работ сделать переключатель "Субподряд / Собственные силы". При субподряде — логика как с материалами (сбор КП от субподрядчиков). При собственных силах — калькулятор на основе сметных норм и фактических ставок бригад.

### 4.4 Статусы позиции КП

**Это критически важный момент, который ты правильно отметил: пока позиция не утверждена, себестоимость НЕ попадает в ФМ.**

```
UNPROCESSED → INVOICES_COLLECTED → PRICE_SELECTED → ON_APPROVAL → APPROVED → IN_FINANCIAL_MODEL
      │              │                    │              │              │              │
      │              │                    │              │              │              │
   Позиция       Собраны             Выбран          На согласо-   Утверждена     Цена
   создана       счета/КП от        лучший           вании у        менеджером    записана
                 поставщиков        вариант           ПМ/руков.                    в ФМ
```

| Статус | Влияние на ФМ |
|--------|---------------|
| UNPROCESSED | Нет — ФМ показывает сметную цену как placeholder |
| INVOICES_COLLECTED | Нет — идёт сбор данных |
| PRICE_SELECTED | Нет — выбор сделан, но не утверждён |
| ON_APPROVAL | Нет — ожидает решения |
| **APPROVED** | **Себестоимость попадает в ФМ** как "плановая себестоимость" |
| IN_FINANCIAL_MODEL | Подтверждено — себестоимость зафиксирована |

> **Уже реализовано:** Перечисление `CpItemStatus` в текущем коде содержит все эти статусы.

### 4.5 Маржинальный анализ

КП даёт трёхуровневый анализ маржи:

```
                    Себестоимость  Сметная цена  Цена заказчику
Позиция             (Факт/КП)    (Нормативная) (Контракт)
────────────────    ──────────   ────────────  ──────────────
Бетон В25           2,790,000    3,825,000     4,140,000
Арматура А500       2,496,000    3,040,000     3,200,000
Опалубка              980,000    1,200,000     1,350,000
Бетонирование       2,800,000    3,500,000     3,900,000
────────────────    ──────────   ────────────  ──────────────
ИТОГО               9,066,000   11,565,000    12,590,000

Экономия на закупках (смета vs себест.):  2,499,000 (21.6%)
Наценка контракта (смета vs заказч.):     1,025,000 (8.9%)
Валовая маржа (себест. vs заказч.):       3,524,000 (28.0%)
```

## 5. Счета

### 5.1 Структура раздела "Счета"

**Твоя идея:** Список всех счетов со столбцами (контрагент, объект, раздел и т.д.) + столбец привязки к позиции.

**Правильно. Полная структура:**

| Столбец | Описание |
|---------|----------|
| **Номер счёта** | Номер от поставщика |
| **Дата** | Дата выставления |
| **Контрагент** | Поставщик / Субподрядчик |
| **Проект (объект)** | К какому проекту относится |
| **Раздел** | ЭМ, ОВ, ВК... |
| **Категория** | Материал / Работа / Оборудование |
| **Сумма без НДС** | Нетто-сумма |
| **НДС** | Сумма НДС |
| **Сумма с НДС** | Итого к оплате |
| **Срок оплаты** | Дата, до которой нужно оплатить |
| **Привязка** | К какой позиции ФМ/КП привязан (или "Не привязан") |
| **Статус** | Текущий статус (см. ниже) |

### 5.2 Строки счёта (Invoice Lines)

Каждый счёт содержит **строки** (line items):

```
Счёт №СЧ-2024-0847 от ООО "МеталлТорг"
├── Строка 1: Арматура А500 d12, 8.5 т × 78,000 = 663,000 ₽
├── Строка 2: Арматура А500 d16, 12.0 т × 79,500 = 954,000 ₽
├── Строка 3: Арматура А500 d25, 11.5 т × 81,000 = 931,500 ₽
└── Строка 4: Доставка, 3 рейса × 25,000 = 75,000 ₽
────────────────────────────────────────────────────
Итого без НДС: 2,623,500 ₽  |  НДС: 524,700 ₽  |  Итого: 3,148,200 ₽
```

Каждая строка может быть **отдельно привязана** к позиции КП/ФМ. Это важно, потому что в одном счёте могут быть позиции для разных разделов.

### 5.3 Smart Matching (интеллектуальный подбор)

**Твоя идея:** Счета должны "осмысленно предлагаться" позициям — на основе проекта, раздела, материала и т.д.

**Алгоритм подбора (взвешенная оценка):**

```
Score = Σ(вес × совпадение_сигнала)

Сигналы:
  1. Совпадение наименования (fuzzy match)        вес: 0.30
  2. Совпадение единицы измерения                  вес: 0.10
  3. Совпадение поставщика (если уже работали)      вес: 0.15
  4. Близость цены к ожидаемой (сметной)           вес: 0.15
  5. Количество вписывается в остаток              вес: 0.10
  6. Совпадение проекта/раздела                    вес: 0.20
```

**Пример в интерфейсе:**

```
Позиция КП: "Арматура А500 d12, 25 тонн"

Подходящие счета:                         Score
┌──────────────────────────────────────────────────────┐
│ ✦ Счёт №147 — ООО "МеталлТорг"          94%        │
│   Арматура А500С d12мм, 8.5 т × 78,000₽           │
│   Раздел: КР | Проект: ЖК Солнечный               │
├──────────────────────────────────────────────────────┤
│   Счёт №193 — ООО "СталПрокат"           71%        │
│   Арматура А500 d10мм, 15 т × 76,000₽             │
│   Раздел: КР | Проект: ЖК Солнечный               │
├──────────────────────────────────────────────────────┤
│   Счёт №201 — ИП Петров                  65%        │
│   Арматура А400 d12мм, 5 т × 72,000₽              │
│   Раздел: — | Проект: ЖК Солнечный                 │
└──────────────────────────────────────────────────────┘
```

### 5.4 Несколько счетов на одну позицию (конкурентный лист)

**Твоя идея верна.** На одну позицию КП может приходить несколько счетов — нужно выбрать лучший.

**Конкурентный лист (Competitive List):**

```
Позиция КП: "Бетон В25 F150 W6, 450 м³"
Сметная цена: 8,500 ₽/м³

┌──────────────┬──────────┬──────────┬──────────┬────────┬────────────┐
│ Поставщик    │ Цена/м³  │ Доставка │ Оплата   │ Оценка │ Выбор      │
├──────────────┼──────────┼──────────┼──────────┼────────┼────────────┤
│ РБУ Бетонмикс│ 6,200 ₽  │ 2 дня    │ 30 дней  │ ★ Best │ ● Выбран   │
│ ЗАО БетонСтр │ 5,900 ₽  │ 7 дней   │ Предопл. │ 2-й    │ ○          │
│ ООО МосБетон │ 6,800 ₽  │ 1 день   │ 14 дней  │ 3-й    │ ○          │
│ ИП Иванов    │ 5,500 ₽  │ 14 дней  │ Предопл. │ —      │ ✗ Нет серт.│
└──────────────┴──────────┴──────────┴──────────┴────────┴────────────┘

Критерии оценки (настраиваемые):
  Цена: 40% | Срок поставки: 25% | Условия оплаты: 20% | Надёжность: 15%
```

**Частичное покрытие (split fulfillment):**

Одна позиция может быть покрыта несколькими счетами:

```
Позиция КП: Арматура А500 d12, 25 тонн

Счёт №147: МеталлТорг — 8.5 т × 78,000 ₽  [ОПЛАЧЕН]
Счёт №183: МеталлТорг — 12.0 т × 78,000 ₽ [УТВЕРЖДЁН]
Счёт №201: СталПрокат — 4.5 т × 80,000 ₽  [НА СОГЛАСОВАНИИ]
──────────────────────────────────────────────────────
Итого по счетам: 25.0 т / 25.0 т (100%)
Средневзвешенная цена: 78,540 ₽/т
```

### 5.5 Статусы счетов

**Твоя идея:** У счетов должны быть статусы, и пока счёт не согласован, себестоимость не улетает в ФМ.

**Абсолютно правильно. Цепочка статусов:**

```
RECEIVED → MATCHED → UNDER_REVIEW → APPROVED → SCHEDULED → PAID
    │          │           │              │
    ↓          ↓           ↓              ↓
 DUPLICATE  UNMATCHED   DISPUTED      ON_HOLD
                           │
                           ↓
                       REJECTED
```

| Статус | Описание | Влияние на КП | Влияние на ФМ |
|--------|----------|---------------|---------------|
| RECEIVED | Счёт внесён в систему | Видно как кандидат | Нет |
| MATCHED | Привязан к позиции КП | Видно как кандидат | Нет |
| UNDER_REVIEW | На проверке менеджером | Видно как кандидат | Нет |
| **APPROVED** | Утверждён к оплате | **Цена фиксируется в КП** | **Себестоимость → ФМ** (при утверждении КП) |
| SCHEDULED | Включён в платёжный календарь | — | Без изменений |
| **PAID** | Оплачен | — | **"Факт оплаты" → ФМ** |
| ON_HOLD | Оплата отложена | — | Обязательство, но флаг |
| REJECTED | Отклонён | Удаляется из кандидатов | Снимается из обязательств |
| DISPUTED | Спорный | Заблокирован | Заблокирован |

### 5.6 Столбец "Привязка" (к чему подвязан счёт)

**Твоя идея:** В таблице счетов должен быть столбец, показывающий привязку.

**Реализация:**

В списке счетов столбец "Привязка" показывает:
- **"Не привязан"** — серый индикатор
- **"ЭМ > Кабель ВВГнг 3×2.5 (8.5т из 15т)"** — привязан к конкретной позиции с прогрессом
- **"Множественная привязка (3 позиции)"** — для счетов с несколькими строками, привязанными к разным позициям

По клику — переход к позиции КП/ФМ.

### 5.7 Процесс утверждения счетов

**Многоуровневое согласование (по сумме):**

| Сумма | Цепочка утверждения |
|-------|---------------------|
| < 100,000 ₽ | Менеджер проекта |
| 100K – 500K ₽ | Менеджер → Руководитель закупок |
| 500K – 2M ₽ | Менеджер → Закупки → Финансовый директор |
| > 2M ₽ | Менеджер → Закупки → Фин. директор → Генеральный |
| Внебюджетный (любая сумма) | Менеджер → Бюджетный комитет → Фин. директор |

**Проверки перед утверждением:**
1. Цена соответствует договорной (±5% допустимое отклонение)
2. Количество не превышает остаток по позиции
3. Бюджет имеет достаточный лимит
4. Поставка физически подтверждена (для материалов)
5. Акт выполненных работ подписан (для работ)
6. Нет дубликата счёта

## 6. Сметы

### 6.1 Структура смет (как устроено в индустрии)

Российская система сметного ценообразования имеет три уровня:

```
Уровень 3: Сводный сметный расчёт (ССР)
├── Глава 1: Подготовка территории
├── Глава 2: Основные объекты строительства
│   ├── Объектная смета (ОС-01): Здание А
│   │   ├── Локальная смета (ЛС-01): Фундамент
│   │   │   ├── Позиция: Земляные работы (ГЭСН 01-01-001-01)
│   │   │   ├── Позиция: Бетонирование (ГЭСН 06-01-001-01)
│   │   │   └── Позиция: Арматурные работы (ГЭСН 06-01-008-01)
│   │   ├── ЛС-02: Каркас
│   │   └── ЛС-03: Инженерные системы
│   └── ОС-02: Здание Б
├── Глава 3-7: Вспомогательные объекты, сети, благоустройство
├── Глава 8: Временные здания и сооружения (ВЗиС)
├── Глава 9: Прочие работы
├── Глава 10-11: Содержание заказчика, подготовка кадров
├── Глава 12: Проектные и изыскательские работы
└── Резерв на непредвиденные расходы (2-3%)
```

**Каждая позиция локальной сметы содержит:**

| Поле | Описание |
|------|----------|
| Обоснование | Код нормы: ГЭСН 06-01-001-01 |
| Наименование | "Устройство бетонных фундаментов" |
| Ед. изм. | м³ |
| Количество | 500 |
| **Прямые затраты** | Оплата труда + Эксплуатация машин + Материалы |
| Оплата труда рабочих (ОЗП) | Чел·ч × тарифная ставка |
| Эксплуатация машин (ЭМ) | Маш·ч × стоимость маш·ч |
| Материалы (МАТ) | Кол-во × цена материала |
| Накладные расходы (НР) | % от ФОТ (по МДС 81-33.2004) |
| Сметная прибыль (СП) | % от ФОТ (по МДС 81-25.2001) |
| **Итого** | ПЗ + НР + СП |

### 6.2 Взаимодействие Смета ↔ ФМ

**Смета — это ОСНОВА для финансовой модели.** Данные из сметы импортируются в ФМ и формируют столбец "Сметная цена".

```
Смета (ЛС-01: Фундамент)                     ФМ (раздел: Фундамент)
──────────────────────────                     ───────────────────────────
Поз. 1: Земляные работы, 500 м³      →        Позиция ФМ: Земляные работы
  Прямые: 1,200,000 ₽                           Сметная цена: 1,500,000 ₽
  НР: 180,000 ₽                                 Себестоимость: (из КП)
  СП: 120,000 ₽                                  Цена заказч.: (из дог.)
  Итого: 1,500,000 ₽                   →

Поз. 2: Бетонирование, 450 м³        →        Позиция ФМ: Бетонирование
  Итого: 3,825,000 ₽                  →         Сметная цена: 3,825,000 ₽
```

**Ключевой момент:** Связь "позиция сметы → позиция ФМ" может быть:
- **1:1** — одна позиция сметы = одна позиция ФМ (для небольших проектов)
- **N:1** — несколько позиций сметы свёрнуты в одну позицию ФМ (чаще всего)
- **По правилам** — маппинг по коду раздела, типу работ или элементу затрат

### 6.3 Взаимодействие Смета ↔ Договоры

Смета — **юридическое обоснование** цены договора:

- **Приложение к договору** — сметная документация является приложением к договору подряда (ГК РФ ст. 743)
- **КС-2 акты** ссылаются на позиции сметы: объёмы из КС-2 не могут превышать сметные объёмы
- **Цена договора** = итого по смете (для контрактов с фиксированной ценой)

### 6.4 Взаимодействие Смета ↔ КП

Смета — **отправная точка** для расчёта себестоимости работ:

```
Позиция сметы: "Монтаж кабельных трасс, 1000 м"
  Сметная стоимость работ: 350 ₽/м (нормативная)

  → В КП:
    Вариант А (субподряд): 280 ₽/м (предложение субподрядчика)
    Вариант Б (свои силы):
      ОЗП: 85 ₽/м (факт. ставка бригады)
      Механизмы: 45 ₽/м (своя техника)
      НР: 95 ₽/м
      СП: 55 ₽/м
      Итого: 280 ₽/м

  Себестоимость (выбрано): 280 ₽/м
  Маржа: 350 - 280 = 70 ₽/м (20%)
```

### 6.5 Взаимодействие Смета ↔ Спецификация

Смета и спецификация берут данные из одного источника (проектной документации), но фокусируются на разном:

| Смета | Спецификация |
|-------|-------------|
| Сколько стоит (цена) | Что именно нужно (техтребования) |
| Нормативные объёмы работ | Конкретные марки/модели |
| Трудозатраты и механизмы | Только материалы и оборудование |

**Связь:** Количество материалов в смете должно совпадать с количеством в спецификации. Если расхождение — нужна сверка (reconciliation).

### 6.6 Интеграция с Гранд-Сметой

Большинство строительных компаний в России готовят сметы в **Гранд-Смете** (80%+ рынка). Наша система должна:

1. **Импортировать** сметы из Гранд-Сметы (форматы: XML/ARPS, .gge, Excel)
2. **Отображать** структуру (ССР → ОС → ЛС → позиции) в удобном виде
3. **Маппить** позиции сметы на позиции ФМ (автоматически + ручная корректировка)
4. **Обновлять** при новых версиях сметы (с отслеживанием изменений)

> **Примечание:** Мы НЕ создаём свой редактор смет — это требует огромных затрат (поддержка всех нормативных баз ГЭСН/ТЕР/ФЕР). Мы импортируем из Гранд-Сметы и работаем с результатом.

### 6.7 Индексы пересчёта

Критически важно для российского рынка:

Сметы составляются в **базисных ценах 2000 года** (или 2022 при переходе на ФСНБ-2022). Для перевода в текущие цены применяются **квартальные индексы** от Минстроя.

```
Базисная цена (2000): 450 ₽/м³
Индекс пересчёта (I кв. 2026): ×8.74
Текущая цена: 3,933 ₽/м³
```

Система должна:
- Хранить индексы по регионам и кварталам
- Автоматически пересчитывать сметные цены при обновлении индексов
- Показывать и базисную, и текущую цену

## 7. Проектная спецификация

### 7.1 Что содержит спецификация

Проектная спецификация (ведомость материалов и оборудования) — это **полный перечень всего, что нужно купить** для проекта, с техническими требованиями.

```
Спецификация проекта "ЖК Солнечный"
├── Раздел: ЭМ (Электромонтаж)
│   ├── Кабель ВВГнг-LS 3×2.5 мм², ГОСТ 31996-2012
│   │   Кол-во: 15,000 м | Чертёж: ЭМ-01, лист 3
│   ├── Автомат ABB S201 C16, IEC 60898
│   │   Кол-во: 500 шт | Чертёж: ЭМ-02, лист 1
│   └── ...
├── Раздел: ОВ (Отопление и вентиляция)
│   ├── Труба PPR PN20 d32, ГОСТ 32415-2013
│   │   Кол-во: 2,500 м | Чертёж: ОВ-01, лист 5
│   └── ...
└── ...
```

**Поля каждой позиции:**

| Поле | Описание |
|------|----------|
| Наименование | Полное название материала/оборудования |
| Марка/Модель | Конкретная марка из проекта |
| ГОСТ/ТУ | Ссылка на стандарт |
| Ед. изм. | шт, м, м², м³, т, кг, компл |
| Проектное количество | Из проектной документации |
| Кол-во с учётом потерь | Проектное + коэффициент потерь |
| Технические характеристики | Ключевые параметры |
| Тип | MATERIAL / EQUIPMENT |
| Ссылки на чертежи | Листы ПД/РД |

### 7.3 Статусы покрытия спецификации

Каждая позиция спецификации имеет **статус покрытия**:

| Статус | Описание | Индикатор |
|--------|----------|-----------|
| NOT_COVERED | Нет ни одного счёта/заказа | 🔴 Красный |
| PARTIALLY_COVERED | Заказано < 100% от нужного | 🟡 Жёлтый + прогресс-бар |
| FULLY_COVERED | Заказано ≥ 100% | 🟢 Зелёный |
| DELIVERED | Всё доставлено на склад | ✅ |
| INSTALLED | Материал смонтирован | ✔✔ |

> **Уже реализовано:** Поле `supplyStatus: 'FULLY_COVERED' | 'PARTIALLY_COVERED' | 'NOT_COVERED'` существует в текущих типах.

### 7.4 Управление аналогами

**Критически важная функция** (особенно в условиях импортозамещения):

```
Проектная позиция: ABB ACH580-01, 75 kW (ЧРП)
  ↓ Невозможно поставить
Предложение аналога:
  Аналог 1: Danfoss FC-302, 75 kW — 94% совместимость
  Аналог 2: ОВЕН ПЧВ3, 75 kW — 87% совместимость (отечественный)
  ↓ Техническая проверка
Инженер подтверждает: габариты ✓, интерфейсы ✓, сертификаты ✓
  ↓ Согласование проектировщика
Автор проекта утверждает замену (лист согласования)
  ↓ Обновление спецификации
Позиция обновлена на аналог с историей замены
```

### 7.5 Спецификация → Закупки → Склад (сквозной процесс)

```
Позиция спецификации
  → Заявка на закупку (автоматическая)
  → Запрос предложений (РФП поставщикам)
  → Сбор счетов/КП от поставщиков
  → Конкурентный лист (выбор лучшего)
  → Заказ поставщику (Purchase Order)
  → Доставка → Приёмка на склад (ТОРГ-12/УПД)
  → Выдача в производство (требование-накладная)
  → Спецификация: статус = INSTALLED
```

### 7.6 Связь Спецификация ↔ ФМ

| Спецификация | ФМ |
|-------------|-----|
| Количество материала | → Столбец "Количество" (для материалов) |
| Сметная цена материала (из сметы) | → Столбец "Сметная цена" |
| Фактическая цена (из счетов) | → через КП → Столбец "Себестоимость" |

**Цель:** Связать сметы с ФМ.

1. Импорт из Гранд-Сметы (XML/ARPS парсер)
2. Автомаппинг позиций сметы → позиции ФМ
3. Индексы пересчёта (справочник)
4. Сверка: позиции сметы vs позиции ФМ (reconciliation)

### Фаза 5: Спецификация v2

**Цель:** Реализовать интерфейс "позиция + счета рядом".

1. UI: позиция спецификации с прогресс-баром покрытия
2. Панель подобранных счетов (по smart matching)
3. Статусы покрытия: NOT_COVERED → PARTIALLY → FULLY → DELIVERED → INSTALLED
4. Управление аналогами (workflow замены)
5. Интеграция со складом (приёмка ↔ спецификация)
