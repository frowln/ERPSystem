# Согласование документа

> Как документ проходит путь от создания до утверждения и подписи — без бумаги и беготни по кабинетам.

---

## Обзор процесса

```
Создание → Отправка → Рецензия → Замечания → Исправление → Утверждение → Подпись
```

| | |
|---|---|
| **Участники** | Автор документа, Рецензенты, Согласующие, Утверждающий |
| **Время** | От нескольких часов (простой документ) до 2–3 недель (сложный, с несколькими кругами правок) |
| **Результат** | Документ утверждён и подписан, доступен всем участникам |

### Схема потока

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌───────────┐
│ Создание │───→│ Отправка │───→│ Рецензия │───→│ Замечания │
│ (автор)  │    │(на согл.) │    │(рецензент)│    │  (есть?)  │
└──────────┘    └──────────┘    └──────────┘    └───────────┘
                                                   │       │
                                                  Да      Нет
                                                   │       │
                                                   ↓       ↓
                                            ┌───────────┐  ┌────────────┐
                                            │Исправление│  │Утверждение │
                                            │  (автор)  │  │            │
                                            └─────┬─────┘  └─────┬──────┘
                                                  │              │
                                                  └──→ Рецензия  │
                                                       (повторно)│
                                                                 ↓
                                                          ┌──────────┐
                                                          │ Подпись  │
                                                          │  (ЭЦП)  │
                                                          └──────────┘
```

## Шаг 1: Создание документа

| | |
|---|---|
| **Кто** | Автор (инженер, менеджер, бухгалтер — зависит от типа документа) |
| **Где** | Главное меню → **Документы** → **Создать** |
| **Что делать** | Создать документ: выбрать категорию, заполнить название, описание. Загрузить файл (PDF, DOCX, DWG) или выбрать шаблон. Привязать к проекту. Указать дисциплину (архитектура, конструктив, инженерные сети). |
| **Результат** | Документ создан в статусе **«Черновик»** (Draft). Виден только автору. |
| **Подробнее** | [Документы](../by-module/documents.md) |

### Категории документов

| Категория | Примеры |
|-----------|---------|
| **Проектная документация** | Чертежи, спецификации, расчёты |
| **Рабочая документация** | Рабочие чертежи, ведомости объёмов |
| **Исполнительная** | АОСР, акты скрытых работ, журналы |
| **Договорная** | Контракты, доп. соглашения, акты |
| **Финансовая** | Сметы, КС-2, КС-3, счета |
| **Организационная** | Приказы, протоколы, инструкции |

---

## Шаг 2: Отправка на согласование

| | |
|---|---|
| **Кто** | Автор документа |
| **Где** | Карточка документа → кнопка **«Отправить на согласование»** |
| **Что делать** | Выбрать маршрут согласования: кто должен рассмотреть документ, в каком порядке. Можно использовать готовый шаблон маршрута или создать свой. Указать срок (SLA) — сколько дней даётся на рассмотрение. |
| **Результат** | Документ в статусе **«На рассмотрении»** (In Review). Рецензенты получают уведомление. |

### Типы маршрутов

| Тип | Описание | Пример |
|-----|----------|--------|
| **Последовательный** | Один за другим: сначала А, потом Б, потом В | Инженер → ГИП → Заказчик |
| **Параллельный** | Все одновременно, ждём всех | Электрик + Сантехник + Архитектор → все должны одобрить |
| **Смешанный** | Комбинация | Параллельно: 3 рецензента → последовательно: ГИП → Директор |

---

## Шаг 3: Рецензия

| | |
|---|---|
| **Кто** | Рецензент (указан в маршруте) |
| **Где** | Главное меню → **Согласования** → **Входящие** |
| **Что делать** | Открыть документ. Изучить содержание. Принять решение: **Утвердить** (всё в порядке), **Утвердить с замечаниями** (мелкие правки, но работу можно продолжать), **Отклонить** (серьёзные ошибки, нужно переделать), **Воздержаться** (не моя компетенция). |
| **Результат** | Рецензент оставил своё решение. Если все рецензенты утвердили — переход к утверждению. Если есть замечания — автору нужно исправить. |

### Инструменты рецензии

| Инструмент | Описание |
|-----------|----------|
| **Комментарии** | Текстовые замечания к документу |
| **Redline** (разметка) | Пометки прямо на чертеже: обвести проблему, подписать |
| **Ссылки на нормы** | «Не соответствует СНиП 3.03.01-87, п. 5.2» |
| **Прикреплённые файлы** | Примеры, как должно быть |

---

## Шаг 4: Замечания (если есть)

| | |
|---|---|
| **Кто** | Автор читает замечания рецензентов |
| **Где** | Карточка документа → вкладка **«Комментарии»** |
| **Что делать** | Изучить все замечания. Каждое замечание может быть: принято (нужно исправить), отклонено (с обоснованием, почему замечание некорректно), уточнено (нужна дополнительная информация). |
| **Результат** | Автор понимает, что нужно исправить. |

---

## Шаг 5: Исправление

| | |
|---|---|
| **Кто** | Автор документа |
| **Где** | Карточка документа → загрузить новую версию |
| **Что делать** | Внести исправления по замечаниям. Загрузить новую версию документа (система сохраняет все предыдущие версии — можно сравнить). Добавить комментарий: какие замечания устранены, какие отклонены с обоснованием. Отправить повторно на согласование. |
| **Результат** | Новая версия отправлена на повторную рецензию. Цикл повторяется (шаги 3–5), пока все не утвердят. |

### Версионность

| Версия | Что произошло |
|--------|-------------|
| v1.0 | Первоначальный документ |
| v1.1 | Исправлен по замечаниям рецензента 1 |
| v1.2 | Исправлен по замечаниям рецензента 2 |
| v2.0 | Утверждённая версия |

> **Все версии сохраняются.** Всегда можно посмотреть, что было в предыдущей версии и кто какие замечания оставил.

---

## Шаг 6: Утверждение

| | |
|---|---|
| **Кто** | Утверждающий (обычно руководитель: ГИП, директор, заказчик) |
| **Где** | Главное меню → **Согласования** → **Входящие** |
| **Что делать** | Финальная проверка. Все замечания устранены, все рецензенты дали «добро». Утверждающий ставит свою отметку. |
| **Результат** | Документ в статусе **«Утверждён»** (Approved). |

### Статусы документа в CDE (ISO 19650)

| Статус | Код | Описание |
|--------|-----|----------|
| **В работе** | WIP | Черновик, виден только автору и рабочей группе |
| **Общий доступ** | Shared | Отправлен на рецензию, виден рецензентам |
| **Опубликован** | Published | Утверждён, доступен всем участникам проекта |
| **Архив** | Archived | Устаревшая версия, заменена новой |

---

## Шаг 7: Подпись (ЭЦП)

| | |
|---|---|
| **Кто** | Подписант (уполномоченное лицо) |
| **Где** | Главное меню → **КЭП** → **Подписание** |
| **Что делать** | Для юридически значимых документов (контракты, акты, КС-2) — подписать квалифицированной электронной подписью (КЭП). Выбрать сертификат, подтвердить подпись. |
| **Результат** | Документ подписан электронной подписью. Имеет юридическую силу. |

> **Не все документы требуют ЭЦП.** Внутренние документы (инструкции, протоколы) достаточно утвердить в системе. КЭП нужна для: договоров, актов (КС-2, КС-3), счетов-фактур, документов для госорганов.

## Трансмиттал — отправка пакета документов

Для передачи нескольких документов заказчику или подрядчику используется **трансмиттал** — сопроводительный документ к пакету.

**Путь:** Главное меню → **CDE** → **Трансмитталы**

| Поле | Описание |
|------|----------|
| **Номер** | Уникальный номер трансмиттала |
| **Получатель** | Кому передаём документы |
| **Документы** | Список включённых документов (с версиями) |
| **Цель** | Для информации / На согласование / Для выполнения |
| **Примечание** | Сопроводительное письмо |

## Что может пойти не так

| Проблема | Где обнаружится | Что делать |
|----------|----------------|------------|
| **Рецензент не отвечает в срок** | Согласования → просрочка SLA | Система отправит напоминание. Если не помогает — эскалация руководителю. Можно перенаправить другому рецензенту |
| **Бесконечный цикл правок** | Версия v1.5, v1.6... и нет конца | Организовать очное совещание. Зафиксировать решения протоколом. Утвердить финальную версию |
| **Документ утверждён, но нужны изменения** | После утверждения | Создать новую ревизию (v2.0), отправить на согласование заново. Старая версия переходит в «Архив» |
| **Нет маршрута согласования** | Шаг 2 — не выбрать маршрут | Попросить администратора создать шаблон маршрута в настройках Workflow |
| **Замечания непонятны** | Шаг 4 — автор не понимает | Запросить разъяснение у рецензента через комментарии в карточке документа |
