# Управление дефектами

> От обнаружения дефекта на объекте до его полного устранения и закрытия.

---

## Обзор процесса

```
Обнаружение → Фиксация → Назначение → Исправление → Проверка → Закрытие
```

| | |
|---|---|
| **Участники** | Инженер по качеству, Прораб, Подрядчик (исполнитель), Заказчик (при приёмке) |
| **Время** | От 1 дня (незначительный) до 14 дней (критический) — по SLA |
| **Результат** | Дефект устранён, качество подтверждено |

### Схема потока

```
┌─────────────┐    ┌──────────┐    ┌────────────┐
│ Обнаружение │───→│ Фиксация │───→│ Назначение │
│  (на объекте)│    │(в системе)│    │(ответств.) │
└─────────────┘    └──────────┘    └────────────┘
                                         │
     ┌───────────┐    ┌──────────┐       │
     │ Закрытие  │←───│ Проверка │←──────┘
     │           │    │(инспектор)│  Исправление
     └───────────┘    └──────────┘
```

---

## Шаг 1: Обнаружение дефекта

| | |
|---|---|
| **Кто** | Любой: инженер по качеству, прораб, заказчик, инспектор |
| **Где** | На строительном объекте — во время обхода, инспекции, приёмки работ |
| **Что делать** | Заметить отклонение от проекта или нормативов. Это может быть: трещина в бетоне, неровная кладка, протечка, неправильная разводка, повреждение отделки — любой брак или несоответствие. Сфотографировать. |
| **Результат** | Дефект обнаружен. Нужно зафиксировать в системе. |

> **Где чаще всего находят дефекты:**
> - При ежедневном обходе прорабом
> - При плановой инспекции качества
> - При приёмке работ от подрядчика
> - При осмотре заказчиком
> - При закрытии Punch List перед сдачей

## Шаг 2: Фиксация в системе

| | |
|---|---|
| **Кто** | Тот, кто обнаружил (обычно инженер по качеству) |
| **Где** | Главное меню → **Качество** → **Реестр дефектов** → **Создать** |
| **Что делать** | Заполнить карточку дефекта: название, описание, проект, местоположение (этаж, секция, помещение). Загрузить фотографии. Отметить на чертеже точное место (pinX/pinY). Указать серьёзность и категорию. |
| **Результат** | Дефект создан в статусе **«Открыт»**. |
| **Подробнее** | [Дефекты](../by-module/defects.md) |

### Серьёзность дефектов

| Уровень | Цвет | SLA (срок устранения) | Пример |
|---------|------|-----------------------|--------|
| **Незначительный** (LOW) | Зелёный | 14 дней | Косметический дефект отделки |
| **Средний** (MEDIUM) | Жёлтый | 7 дней | Неровность кладки, мелкая протечка |
| **Высокий** (HIGH) | Оранжевый | 3 дня | Нарушение герметичности, дефект инженерных сетей |
| **Критический** (CRITICAL) | Красный | 1 день | Угроза безопасности, обрушение конструкции |

### Категории дефектов

| Категория | Примеры |
|-----------|---------|
| **Конструктивные** | Трещины, деформации, несоответствие проекту |
| **Архитектурные** | Отделка, фасад, витражи |
| **Электрические** | Проводка, освещение, щиты |
| **Сантехнические** | Трубы, сантехника, канализация |
| **Отделочные** | Штукатурка, покраска, плитка |
| **Пожарная безопасность** | Огнезащита, пожарная сигнализация |

## Шаг 3: Назначение ответственного

| | |
|---|---|
| **Кто** | Инженер по качеству или Прораб |
| **Где** | Карточка дефекта → поле **«Ответственный»** |
| **Что делать** | Назначить исполнителя — того, кто будет устранять дефект. Обычно это подрядчик, выполнявший работу. Установить крайний срок (по SLA). Добавить комментарий с инструкцией по устранению. |
| **Результат** | Дефект в статусе **«В работе»**. Ответственный получает уведомление (в системе + Telegram/email). |

> **Уведомления:** Система автоматически напомнит ответственному о приближении срока. Если срок просрочен — уведомит руководителя.

---

## Шаг 4: Устранение дефекта

| | |
|---|---|
| **Кто** | Подрядчик / исполнитель |
| **Где** | На объекте — физическое устранение |
| **Что делать** | Выполнить работы по устранению дефекта. Сфотографировать результат «до» и «после». Отметить в системе: дефект устранён, приложить фото, добавить комментарий о выполненных работах. |
| **Результат** | Дефект в статусе **«Исправлен»**. Ожидает проверки. |

---

## Шаг 5: Проверка (верификация)

| | |
|---|---|
| **Кто** | Инженер по качеству (тот, кто создавал дефект, или другой инспектор) |
| **Где** | На объекте — визуальный осмотр + в системе |
| **Что делать** | Прийти на объект и проверить устранение: качество работ соответствует требованиям? Сравнить фото «до» и «после». Если всё хорошо — подтвердить. Если нет — отклонить с комментарием (дефект вернётся в статус «В работе»). |
| **Результат** | Дефект в статусе **«Проверен»** (если принят) или обратно **«В работе»** (если не принят). |

---

## Шаг 6: Закрытие

| | |
|---|---|
| **Кто** | Инженер по качеству |
| **Где** | Карточка дефекта → кнопка **«Закрыть»** |
| **Что делать** | После успешной проверки — закрыть дефект. Он перейдёт в архив, но останется в статистике для анализа. |
| **Результат** | Дефект в статусе **«Закрыт»**. Счётчик «открытых дефектов» уменьшился. |

> **Процесс завершён.** Дефект устранён и задокументирован. Данные доступны для анализа.

## Аналитика: зачем отслеживать дефекты

### Парето-анализ

Главное меню → **Качество** → **Парето-анализ дефектов**

Диаграмма Парето показывает: **80% дефектов приходится на 20% причин**. Например:
- 45% дефектов — отделочные работы (подрядчик «Отделка-Плюс»)
- 25% — сантехника (бригада Иванова)
- 15% — электрика
- 15% — прочее

**Вывод:** нужно работать с качеством отделочных работ — провести обучение или сменить подрядчика.

### Дашборд дефектов

| Метрика | Что показывает |
|---------|---------------|
| **Всего дефектов** | Общее количество за период |
| **Открытых** | Сколько ещё не устранено |
| **Просроченных** | SLA нарушен — нужно срочно реагировать |
| **Среднее время устранения** | Эффективность подрядчиков |
| **По категориям** | Где больше всего проблем |
| **По ответственным** | У кого больше всего дефектов |

## Punch List — дефекты при сдаче

При сдаче объекта все незакрытые дефекты формируют **Punch List** — перечень замечаний, которые необходимо устранить перед приёмкой.

**Путь:** Главное меню → **Punch List**

| Отличие от обычных дефектов | Описание |
|----------------------------|----------|
| **Привязка к приёмке** | Punch List привязан к зоне/этажу/секции |
| **Процент закрытия** | Автоматический расчёт: закрытые ÷ все × 100% |
| **Блокировка сдачи** | Объект не принимается, пока Punch List не 100% |
| **Канбан-доска** | Удобное визуальное отслеживание по статусам |

## Что может пойти не так

| Проблема | Где обнаружится | Что делать |
|----------|----------------|------------|
| **Ответственный не реагирует** | Дефект просрочен (SLA) | Эскалация: уведомить руководителя подрядчика, при необходимости — привлечь другого исполнителя |
| **Дефект устранён некачественно** | Шаг 5 — проверка не пройдена | Вернуть в работу с детальным описанием недостатков. Если повторяется — рассмотреть замену подрядчика |
| **Слишком много дефектов** | Дашборд — красные метрики | Провести Парето-анализ: найти основную причину (подрядчик, материал, проект). Принять системные меры |
| **Дефект обнаружен после сдачи** | Гарантийный период | Оформить гарантийную претензию → [Жизненный цикл проекта](project-lifecycle.md), шаг 10 |
| **Заказчик нашёл дефект при приёмке** | Приёмочная комиссия | Добавить в Punch List, назначить срок, устранить до повторной приёмки |
