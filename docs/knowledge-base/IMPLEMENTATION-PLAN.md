## Философия

База знаний — не просто справка, а **обучающая платформа**, которая:
- Снижает порог входа (50-летний директор разберётся сам)
- Уменьшает нагрузку на поддержку (80% вопросов — в статьях)
- Ускоряет onboarding (чеклист + интерактивные подсказки)
- Является конкурентным преимуществом (единственная российская строительная ERP с такой глубиной)

## Технические решения

### Хранение контента

```
Вариант: Build-time генерация (рекомендуется)

docs/knowledge-base/*.md
       ↓ (scripts/generate-kb.ts при npm run build)
frontend/src/data/kb-index.json        ← Манифест: slug, title, category, excerpt
frontend/public/kb/articles/*.json     ← Контент каждой статьи (HTML)
frontend/public/kb/search-index.json   ← Индекс для Fuse.js
```

**Почему build-time, а не runtime:**
- Мгновенная загрузка (нет парсинга markdown на клиенте)
- Поисковый индекс готов заранее
- Можно code-split по категориям
- Работает офлайн (PWA)
- Нет нагрузки на бэкенд для статического контента

**Бэкенд используется для:**
- Обратная связь (👍👎) → POST /api/support/kb/feedback
- Счётчик просмотров → PATCH /api/support/kb/articles/:id/view
- Прогресс onboarding → GET /api/users/onboarding-progress
- FAQ администрирование (CRUD)

### Поиск

```
Fuse.js (клиентский, мгновенный)
  - Поля: title (вес 3), excerpt (вес 2), content (вес 1), tags (вес 2)
  - Порог: 0.3 (нечёткий поиск — «бюджит» найдёт «бюджет»)
  - Результаты: до 10, с подсветкой совпадений
  - Семантические связи: теги + related articles в метаданных
```

### Фаза 6: Админка для редактирования статей (НА БУДУЩЕЕ)

> **Когда:** Когда подключится техписатель, менеджер поддержки или другие авторы контента, которым неудобно работать через git.

**Сейчас:** Статьи редактируются через файлы в git (`docs/knowledge-base/by-module/*.md`). Это нормальный подход — так работают Stripe, Vercel, Cloudflare. Версионирование бесплатно через git history.
