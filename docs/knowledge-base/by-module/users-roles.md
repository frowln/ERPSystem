# Пользователи и роли

---

## Что это и зачем

**Пользователи и роли** — это раздел управления доступом. Кто может входить в систему, что может видеть, что может делать — всё настраивается здесь.

Без управления доступом:
- Бухгалтер видит (и может случайно удалить) техническую документацию
- Прораб может менять бюджеты проектов
- Уволенный сотрудник продолжает иметь доступ к системе
- Невозможно понять, кто именно изменил документ

Система ПРИВОД обеспечивает:
- **Роли** — типовые наборы прав (Администратор, Менеджер, Инженер, Бухгалтер, Наблюдатель)
- **Разрешения** — тонкая настройка: какой модуль видно, какие действия доступны
- **Управление пользователями** — создание, блокировка, сброс пароля
- **Журнал действий** — кто, когда и что сделал

## Часть 1: Роли

Роль — это **набор прав**, который назначается пользователю. Вместо того чтобы настраивать права каждому отдельно — вы назначаете роль.

### Базовые роли

| Роль | Описание | Что может |
|------|----------|-----------|
| **Администратор** (ADMIN) | Полный контроль | Всё: настройки, пользователи, все модули |
| **Менеджер** (MANAGER) | Руководитель проекта/отдела | Управление проектами, задачами, финансами, сотрудниками |
| **Инженер** (ENGINEER) | Технический специалист | Документация, качество, безопасность, BIM |
| **Бухгалтер** (ACCOUNTANT) | Финансовый учёт | Счета, платежи, бюджеты, КС-2/КС-3, 1С |
| **Наблюдатель** (VIEWER) | Только просмотр | Может смотреть, но не может создавать/редактировать |

### Что видит каждая роль

| Модуль | ADMIN | MANAGER | ENGINEER | ACCOUNTANT | VIEWER |
|--------|-------|---------|----------|------------|--------|
| Проекты | ✅ Всё | ✅ Свои | ✅ Свои | 👁 Просмотр | 👁 |
| Задачи | ✅ | ✅ | ✅ | 👁 | 👁 |
| Финансы | ✅ | ✅ | 👁 | ✅ | 👁 |
| Кадры | ✅ | ✅ | 👁 | ✅ ФОТ | 👁 |
| Документы | ✅ | ✅ | ✅ | ✅ Финансовые | 👁 |
| Безопасность | ✅ | ✅ | ✅ | ❌ | 👁 |
| BIM | ✅ | ✅ | ✅ | ❌ | 👁 |
| Настройки | ✅ | ❌ | ❌ | ❌ | ❌ |
| Пользователи | ✅ | ❌ | ❌ | ❌ | ❌ |

> ✅ = полный доступ, 👁 = только просмотр, ❌ = нет доступа

## Часть 2: Разрешения (Permissions)

**Путь:** Настройки → **Роли и права**

Помимо базовых ролей, можно настроить **тонкие разрешения**.

### Уровни разрешений

| Уровень | Описание | Пример |
|---------|----------|--------|
| **Модель** | Доступ к целому модулю | «Может видеть модуль Финансы» |
| **Запись** | Доступ к конкретным записям | «Может видеть только свои проекты» |
| **Поле** | Доступ к отдельным полям | «Не видит колонку Прибыль» |

### Типы действий

| Действие | Описание |
|----------|----------|
| **Просмотр** (VIEW) | Видит данные |
| **Создание** (CREATE) | Может создавать новые записи |
| **Редактирование** (EDIT) | Может изменять существующие |
| **Удаление** (DELETE) | Может удалять |
| **Экспорт** (EXPORT) | Может скачивать/выгружать |
| **Согласование** (APPROVE) | Может утверждать/отклонять |

### Группы разрешений

Разрешения организованы по модулям:

| Группа | Включает |
|--------|----------|
| **Проекты** | Просмотр / Создание / Редактирование / Удаление проектов |
| **Финансы** | Бюджеты / Счета / Платежи / КС-2 / КС-3 |
| **Кадры** | Сотрудники / Табели / Зарплаты |
| **Документы** | Просмотр / Загрузка / Удаление документов |
| **Отчёты** | Доступ к конструктору / Экспорт |
| **Настройки** | Управление системой |

## Часть 3: Управление пользователями

### Создание пользователя

| Поле | Обязательно? | Описание |
|------|-------------|----------|
| **ФИО** | Да | Полное имя сотрудника |
| **Email** | Да | Используется как логин |
| **Роль** | Да | Одна из базовых ролей |
| **Подразделение** | Нет | К какому отделу относится |
| **Телефон** | Нет | Для SMS-уведомлений |
| **Пароль** | Да | Временный, при первом входе — сменить |

### Статусы пользователя

| Статус | Описание |
|--------|----------|
| **Активен** (ACTIVE) | Может входить в систему и работать |
| **Заблокирован** (BLOCKED) | Не может войти (заблокирован вручную или автоматически) |
| **Ожидает активации** | Зарегистрирован, но ещё не подтвердил email |

### Действия с пользователем

| Действие | Описание |
|----------|----------|
| **Сменить роль** | Назначить другую роль |
| **Сбросить пароль** | Пользователь получит email со ссылкой для нового пароля |
| **Заблокировать** | Запретить вход (например, при увольнении) |
| **Разблокировать** | Вернуть доступ |
| **Удалить** | Полностью удалить аккаунт (необратимо) |

### Автоматическая блокировка

Система автоматически блокирует пользователя:
- После **5 неудачных попыток** входа → блокировка на 15 минут
- Администратор может разблокировать вручную

## Часть 4: Панель администратора

**Путь:** Настройки → **Панель администратора**

### Статистика

| Показатель | Описание |
|-----------|----------|
| **Всего пользователей** | Сколько аккаунтов в системе |
| **Активных** | Сколько могут входить |
| **Заблокированных** | Сколько заблокировано |
| **Онлайн сейчас** | Кто в системе прямо сейчас |
| **Последние входы** | Кто, когда, откуда входил |

### Журнал действий

Система записывает:
- Кто вошёл / вышел
- Кто создал / изменил / удалил запись
- Кто скачал документ
- Кто изменил настройки

Журнал нужен для:
- **Безопасности** — расследование инцидентов
- **Аудита** — кто за что отвечает
- **Отладки** — понять, что пошло не так

## Типичные сценарии

### Сценарий 1: Приём нового сотрудника

```
1. В компанию приходит новый инженер ПТО — Козлов И.А.

2. Администратор создаёт пользователя:
   → ФИО: Козлов Иван Александрович
   → Email: kozlov@stroykompaniya.ru
   → Роль: Инженер (ENGINEER)
   → Подразделение: ПТО
   → Временный пароль: автоматически сгенерирован

3. Козлов получает email:
   «Вам создан аккаунт в системе ПРИВОД.
    Логин: kozlov@stroykompaniya.ru
    Временный пароль: Xk9#mP2f
    При первом входе система попросит изменить пароль.»

4. Козлов входит → меняет пароль → настраивает 2FA
   → Имеет доступ: документы, качество, безопасность, BIM
   → Не имеет доступа: финансы, настройки, пользователи

5. Через месяц:
   → Козлова назначают руководителем проекта
   → Администратор меняет роль: Инженер → Менеджер
   → Теперь Козлов видит финансы своих проектов
```

### Сценарий 2: Увольнение сотрудника

```
1. Менеджер Сидоров увольняется

2. Администратор:
   → Открывает карточку пользователя Сидорова
   → Нажимает «Заблокировать»
   → Причина: «Увольнение с 15.04.2026»

3. С этого момента:
   → Сидоров не может войти в систему
   → Все его данные сохранены (задачи, комментарии, документы)
   → Его задачи нужно переназначить другим

4. Через полгода:
   → Если нужно — можно разблокировать (например, если вернулся)
   → Или удалить аккаунт полностью
```

### Сценарий 3: Настройка ограниченного доступа

```
1. Компания нанимает внешнего аудитора
   → Ему нужно видеть финансы, но НЕЛЬЗЯ ничего менять

2. Администратор:
   → Создаёт пользователя с ролью «Наблюдатель»
   → Дополнительно включает: просмотр модуля «Финансы»
   → Остальные модули — по умолчанию (только просмотр)

3. Аудитор:
   → Видит бюджеты, счета, платежи, отчёты
   → Может скачивать отчёты в Excel
   → НЕ может создавать, изменять, удалять

4. По окончании аудита:
   → Администратор блокирует аккаунт
```

## Частые вопросы

**Сколько ролей можно создать?**
Базовых ролей — 5 (Администратор, Менеджер, Инженер, Бухгалтер, Наблюдатель). Можно создавать дополнительные роли с индивидуальным набором прав.

**Может ли у пользователя быть несколько ролей?**
Пользователю назначается одна основная роль. Но можно дополнительно включить или выключить отдельные разрешения для конкретного пользователя.

**Что произойдёт, если заблокировать пользователя, который сейчас онлайн?**
Его текущая сессия будет завершена. При следующей попытке — не сможет войти.

**Можно ли восстановить удалённого пользователя?**
Нет. Удаление — необратимо. Рекомендуется сначала блокировать, а удалять — только если уверены.

**Видит ли администратор пароли пользователей?**
Нет. Пароли хранятся в зашифрованном виде. Администратор может только сбросить пароль — тогда пользователь получит ссылку для создания нового.

**Как узнать, кто изменил документ?**
В журнале действий. Каждое изменение записывается с указанием пользователя, времени и типа действия.
