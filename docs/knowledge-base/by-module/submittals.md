# Submittals (Согласование документации)

---

## Что это и зачем

**Submittal** — это документ (чертёж, расчёт, каталожный лист, образец), который подрядчик направляет заказчику (или проектировщику) на согласование перед использованием в работе.

Зачем это нужно:
- **Проверка соответствия** — заказчик убеждается, что подрядчик будет использовать правильные материалы и решения
- **Ответственность** — согласованный submittal = документальное подтверждение, что материал/решение одобрено
- **Предотвращение ошибок** — лучше согласовать ДО закупки и монтажа, чем переделывать ПОСЛЕ

Примеры submittal:
- Рабочий чертёж металлоконструкций (shop drawing)
- Каталожный лист на светильники
- Образец плитки для облицовки фасада
- Протокол испытаний бетонной смеси
- Расчёт несущей способности фундамента

## Типы submittal

| Тип | Что это | Пример |
|-----|---------|--------|
| **Рабочий чертёж** (SHOP_DRAWING) | Чертёж, разработанный подрядчиком | Деталировка металлоконструкций, раскладка арматуры |
| **Данные о продукции** (PRODUCT_DATA) | Каталожные листы, спецификации оборудования | Паспорт на насос, техлист светильника |
| **Образец** (SAMPLE) | Физический образец материала | Образец плитки, кусок обоев, фрагмент фасадной панели |
| **Макет** (MOCK_UP) | Натурный образец конструкции | Фрагмент фасада 1×1 м, образец кладки с расшивкой |
| **Протокол испытаний** (TEST_REPORT) | Результаты лабораторных испытаний | Протокол прочности бетона, сертификат огнестойкости |
| **Сертификат** (CERTIFICATE) | Документ, подтверждающий качество | Сертификат соответствия, пожарный сертификат |
| **Расчёт** (CALCULATION) | Инженерный расчёт | Расчёт фундамента, теплотехнический расчёт |
| **Подбор состава** (DESIGN_MIX) | Рецептура смеси | Подбор состава бетона В25 F150 W6 |
| **Прочее** (OTHER) | Всё остальное | Графики, схемы, инструкции |

## Как создать submittal

1. Перейдите **Submittals → Список**
2. Нажмите **«+ Новый submittal»**
3. Заполните:

| Поле | Обязательно? | Описание |
|------|-------------|----------|
| **Название** | Да | Краткое описание (например: «Каталожные листы светильников — офис 2 этаж») |
| **Проект** | Да | К какому проекту |
| **Тип** | Да | Из списка выше (9 типов) |
| **Описание** | Нет | Подробности: что согласуется, по какому разделу |
| **Дисциплина** | Нет | Раздел проекта (АР, КР, ОВ, ВК, ЭО и т.д.) |
| **Направлено на рассмотрение** | Нет | Кому (проектировщик, заказчик) |
| **Срок рассмотрения** | Нет | До когда нужен ответ |
| **Вложения** | Рекомендуется | Файлы: чертежи, каталоги, протоколы |

4. Нажмите **«Создать»** — submittal в статусе «Черновик»
5. Нажмите **«Отправить»** — submittal уходит на рассмотрение

## Статусы submittal

| Статус | Цвет | Что означает |
|--------|------|-------------|
| **Черновик** (DRAFT) | Серый | Создан, но ещё не отправлен |
| **Отправлен** (SUBMITTED) | Синий | Направлен на рассмотрение |
| **На рассмотрении** (UNDER_REVIEW) | Жёлтый | Рецензент изучает документ |
| **Согласовано** (APPROVED) | Зелёный | Полностью одобрено, можно использовать |
| **Согласовано с замечаниями** (APPROVED_AS_NOTED) | Жёлто-зелёный | Одобрено, но с замечаниями — нужно учесть |
| **Отклонено** (REJECTED) | Красный | Не согласовано. Нужно переделать и отправить заново |
| **Повторная подача** (RESUBMITTED) | Синий | Исправленная версия отправлена на повторное рассмотрение |
| **Закрыт** (CLOSED) | Серый | Процесс согласования завершён |

```
Черновик → Отправлен → На рассмотрении → Согласовано → Закрыт
                                        → Согласовано с замечаниями → Закрыт
                                        → Отклонено → Повторная подача → На рассмотрении → ...
```

## «Мяч на чьей стороне» (Ball in Court)

Одна из главных функций модуля — отслеживание, **кто сейчас должен действовать**.

| Ситуация | «Мяч» у кого |
|----------|-------------|
| Submittal создан, но не отправлен | Подрядчик |
| Отправлен на рассмотрение | Проектировщик / Заказчик |
| Отклонён — нужна переделка | Подрядчик |
| Согласован с замечаниями | Подрядчик (учесть замечания) |
| Согласован | Никто (процесс завершён) |

Система показывает в списке, у кого сейчас «мяч» — это помогает не терять документы и не затягивать согласование.

## Цепочка рецензий

К одному submittal может быть несколько рецензий — например, если документ проходит через нескольких специалистов:

1. **Проектировщик (раздел КР)** — проверяет расчёты → «Согласовано»
2. **Главный инженер заказчика** — проверяет соответствие ТЗ → «Согласовано с замечаниями»
3. **Авторский надзор** — финальное согласование → «Согласовано»

Каждая рецензия содержит:
- Кто рецензировал
- Дата
- Решение (согласовано / с замечаниями / отклонено)
- Комментарий
- Есть ли это официальное решение (флаг «Официальная рецензия»)

## Пакеты submittal

Связанные документы можно объединить в **пакет** (Submittal Package). Например:

**Пакет: «Металлоконструкции — каркас здания А»**
- Submittal 1: Рабочие чертежи (KMD)
- Submittal 2: Сертификаты на сталь
- Submittal 3: Протоколы испытаний сварных образцов
- Submittal 4: Аттестация сварщиков (НАКС)

Пакет удобен, когда для одного вида работ нужно согласовать несколько документов одновременно.

## Типичные сценарии

### Сценарий 1: Согласование рабочих чертежей

```
1. Подрядчик разработал рабочие чертежи металлоконструкций (KMD)

2. Создаёт submittal:
   → Тип: Рабочий чертёж
   → Название: «KMD — каркас 1 этажа, оси А-В/1-5»
   → Прикладывает PDF чертежей (15 листов)
   → Отправляет на рассмотрение проектировщику

3. Проектировщик (раздел КР) рассматривает 3 дня:
   → Замечание: «Узел У-5 — изменить фасонку, не проходит по расчёту»
   → Решение: «Отклонено»

4. Подрядчик исправляет узел У-5:
   → Повторная подача (RESUBMITTED)
   → Прикладывает исправленные чертежи

5. Проектировщик проверяет:
   → Узел исправлен ✅
   → Решение: «Согласовано»

6. Подрядчик может заказывать металл и начинать изготовление
```

### Сценарий 2: Согласование материалов отделки

```
1. Заказчик хочет утвердить отделочные материалы

2. Подрядчик создаёт пакет «Отделка — офис 3 этаж»:

   Submittal 1: Образцы плитки (3 варианта) → Тип: Образец
   Submittal 2: Каталог краски Tikkurila → Тип: Данные о продукции
   Submittal 3: Образцы обоев (2 варианта) → Тип: Образец

3. Заказчик рассматривает:
   → Плитка: вариант 2 согласован ✅
   → Краска: согласована с замечанием «RAL 9010 вместо 9003»
   → Обои: отклонены, «не соответствуют дизайн-проекту»

4. Подрядчик:
   → Плитка: заказывает вариант 2
   → Краска: заказывает RAL 9010
   → Обои: ищет другие варианты → повторная подача
```

## Частые вопросы

**Обязательно ли согласовывать submittal?**
По условиям большинства строительных контрактов — да. Это стандартная практика на крупных объектах. Без согласования заказчик может не принять работу.

**Кто рассматривает submittal?**
Зависит от типа: рабочие чертежи — проектировщик, образцы отделки — заказчик (дизайнер), сертификаты — инженер по качеству.

**Сколько времени на рассмотрение?**
Обычно 5–14 дней по контракту. Система отслеживает сроки и показывает просроченные submittal.

**Что означает «Согласовано с замечаниями»?**
Документ в целом принят, но нужно учесть замечания. Можно использовать, но исправления обязательны.

**Можно ли использовать материал до согласования submittal?**
Формально — нет. На практике бывают исключения (например, типовые материалы), но это риск: если заказчик не согласует — придётся менять за свой счёт.

**Зачем нужны пакеты?**
Для удобства: все документы по одному виду работ в одном месте. Проектировщику не нужно искать связанные submittal по отдельности.

**Чем submittal отличается от RFI?**
Submittal — это «вот документ, согласуйте». RFI — это «у нас вопрос, ответьте». Submittal = документ на согласование. RFI = запрос информации.
