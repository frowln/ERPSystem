# Планирование

---

## Что это и зачем

**Планирование** — это модуль для управления графиком строительства. Здесь вы строите **диаграмму Ганта**, определяете последовательность работ, назначаете ресурсы и отслеживаете выполнение.

Без модуля планирования:
- График в Excel — нет связей между работами, забыли обновить
- Подрядчик опоздал на 3 дня — никто не понял, что это сдвигает весь проект на 2 недели
- Ресурсы (люди, техника) назначены «на глаз» — то простаивают, то не хватает
- Заказчик спрашивает: «Укладываетесь в сроки?» — а ответить нечем

Модуль «Планирование» в системе ПРИВОД:
- Диаграмма Ганта с зависимостями между работами
- Метод критического пути (CPM) — какие работы нельзя задерживать
- Освоенный объём (EVM) — объективная оценка прогресса
- Базовые планы — сравнение «как планировали» vs. «как идёт»
- Назначение ресурсов — люди, техника, материалы
- Учёт объёмов работ — план и факт

## Часть 1: Структура работ (WBS)

Прежде чем строить график, нужно **разбить проект на части**. Это называется WBS — структура декомпозиции работ.

### Уровни разбивки

| Уровень | Тип | Пример |
|---------|-----|--------|
| **Проект** | PROJECT | ЖК «Южный парк» |
| **Этап** | PHASE | Подземная часть |
| **Раздел** | SECTION | Монолитные работы |
| **Работа** | WORK_PACKAGE | Бетонирование фундамента блок А |
| **Операция** | ACTIVITY | Армирование, опалубка, бетонирование |

```
ЖК «Южный парк»
├── Подземная часть
│   ├── Земляные работы
│   │   ├── Разработка котлована
│   │   └── Обратная засыпка
│   ├── Монолитные работы
│   │   ├── Фундамент блок А
│   │   ├── Фундамент блок Б
│   │   └── Стены подвала
│   └── Гидроизоляция
├── Надземная часть
│   ├── Монолит каркаса
│   ├── Кладочные работы
│   └── Фасад
└── Инженерные системы
    ├── Электрика
    ├── Водоснабжение
    └── Вентиляция
```

## Часть 2: Диаграмма Ганта

**Путь:** Планирование → **Ганта**

Диаграмма Ганта — это **календарный график**, где каждая работа показана горизонтальной полосой.

### Что видно на диаграмме

| Элемент | Описание |
|---------|----------|
| **Полоска работы** | Длина = длительность (в днях) |
| **Синяя полоска** | Плановые сроки |
| **Зелёная заливка** | Фактический прогресс (%) |
| **Красная полоска** | Работа на критическом пути |
| **Стрелки** | Зависимости между работами |
| **Ромбик** | Веха (milestone) — ключевая дата |

### Зависимости между работами

| Тип связи | Обозначение | Описание | Пример |
|-----------|-------------|----------|--------|
| **Финиш-Старт** | FS | Б начинается после завершения А | Бетонирование после армирования |
| **Старт-Старт** | SS | Б начинается одновременно с А | Электрика и сантехника одновременно |
| **Финиш-Финиш** | FF | Б завершается одновременно с А | Отделка и уборка |
| **Старт-Финиш** | SF | Б завершается при начале А | Редко используется |

Связи можно задать с **задержкой** (лаг). Например: «Бетонирование → (+ 28 дней набор прочности) → Снятие опалубки».

## Часть 3: Критический путь (CPM)

**Критический путь** — это самая длинная цепочка связанных работ в проекте. Если любая работа на критическом пути задерживается — задерживается **весь проект**.

### Пример

```
Котлован (15 дн.) → Фундамент (30 дн.) → Каркас (90 дн.) → Фасад (45 дн.) → Сдача
         ↓                                      ↑
     Сети (20 дн.) ─────────────────────────────┘

Критический путь: Котлован → Фундамент → Каркас → Фасад → Сдача
(15 + 30 + 90 + 45 = 180 дней)

Сети (20 дней) — НЕ на критическом пути.
У них есть ЗАПАС (float): 30 - 20 = 10 дней.
Сети могут опоздать на 10 дней без последствий для проекта.
```

### Что показывает система

| Показатель | Описание |
|-----------|----------|
| **Критический путь** | Выделен красным на Ганте |
| **Общий запас (Total Float)** | На сколько дней можно задержать работу без влияния на проект |
| **Свободный запас (Free Float)** | На сколько дней можно задержать без влияния на следующую работу |
| **Раннее начало / окончание** | Самые ранние возможные даты |
| **Позднее начало / окончание** | Самые поздние допустимые даты |

> Правило: если у работы **запас = 0** — она на критическом пути. Задерживать нельзя.

## Часть 4: Освоенный объём (EVM)

**Путь:** Планирование → **Освоенный объём**

EVM (Earned Value Management) — это метод, который отвечает на три вопроса:
1. **Сколько планировали сделать** к этой дате?
2. **Сколько реально сделали?**
3. **Сколько реально потратили?**

### Три базовых показателя

| Показатель | Обозначение | Вопрос |
|-----------|-------------|--------|
| **Плановый объём** | PV | Сколько работ планировали выполнить к сегодня? |
| **Освоенный объём** | EV | Сколько работ реально выполнили? |
| **Фактические затраты** | AC | Сколько реально потратили денег? |

### Показатели отклонений

| Показатель | Формула | Что значит | Хорошо |
|-----------|---------|-----------|--------|
| **Отклонение по срокам** (SV) | EV − PV | Опережаем или отстаём? | > 0 |
| **Отклонение по стоимости** (CV) | EV − AC | Экономим или перерасходуем? | > 0 |
| **Индекс сроков** (SPI) | EV ÷ PV | На сколько выполнены сроки? | ≥ 1.0 |
| **Индекс стоимости** (CPI) | EV ÷ AC | Эффективность расходования? | ≥ 1.0 |

### Прогнозы

| Показатель | Формула | Что значит |
|-----------|---------|-----------|
| **Бюджет проекта** (BAC) | — | Общий утверждённый бюджет |
| **Прогноз затрат** (EAC) | BAC ÷ CPI | Сколько потратим в итоге? |
| **Осталось потратить** (ETC) | EAC − AC | Сколько ещё нужно денег? |
| **Отклонение при завершении** (VAC) | BAC − EAC | Уложимся в бюджет? |
| **Индекс для завершения** (TCPI) | (BAC − EV) ÷ (BAC − AC) | Какая эффективность нужна до конца? |

### Как читать графики EVM

```
Пример: Проект «ЖК Южный парк», 6-й месяц из 12

BAC (бюджет): 100 млн руб.
PV (план):     50 млн (к 6 мес. планировали сделать на 50 млн)
EV (факт):     40 млн (реально сделали на 40 млн)
AC (потратили): 48 млн (потратили 48 млн)

SV = 40 − 50 = −10 млн → отстаём от графика ❌
CV = 40 − 48 = −8 млн → перерасход ❌
SPI = 40 ÷ 50 = 0.80 → выполнено 80% плана по срокам
CPI = 40 ÷ 48 = 0.83 → 1 рубль работы стоит 1.20 руб.

EAC = 100 ÷ 0.83 = 120 млн → прогноз: потратим 120 млн вместо 100
VAC = 100 − 120 = −20 млн → перерасход 20 млн на конец проекта

Вывод: проект отстаёт И перерасходует бюджет.
Нужны меры: ускорить работы, сократить затраты.
```

## Часть 5: Базовые планы (Baseline)

Когда график утверждён, создаётся **базовый план** — «фотография» графика в определённый момент. Потом можно сравнивать: «как планировали» vs. «как идёт».

### Статусы базового плана

| Статус | Описание |
|--------|----------|
| **Черновик** (DRAFT) | Только создан |
| **На согласовании** (PENDING) | Отправлен на утверждение |
| **Утверждён** (APPROVED) | Принят как эталон |
| **Активный** (ACTIVE) | Текущий действующий базовый план |
| **Заменён** (SUPERSEDED) | Был активным, но заменён новым |
| **Архивный** (ARCHIVED) | Сохранён для истории |
| **Отклонён** (REJECTED) | Не утверждён |

### Зачем нужно несколько базовых планов

На практике график корректируется при серьёзных изменениях:
- Изменился состав работ (заказчик добавил этаж)
- Критическая задержка (поставщик подвёл)
- Пересмотр бюджета

Каждый раз создаётся новый базовый план. Старый сохраняется для сравнения.

На диаграмме Ганта можно включить **отображение базовой линии** — серая полоска под фактической, чтобы видеть отклонения.

## Часть 6: Ресурсы

**Путь:** Планирование → **Ресурсы**

### Типы ресурсов

| Тип | Описание | Примеры |
|-----|----------|---------|
| **Трудовые** (LABOR) | Люди | Бетонщик, монтажник, электрик |
| **Техника** (EQUIPMENT) | Машины и механизмы | Кран, экскаватор, бетононасос |
| **Материалы** (MATERIAL) | Строительные материалы | Бетон, арматура, кирпич |

### Назначение ресурсов

Для каждой работы указывается:
- **Какие ресурсы** нужны
- **Сколько** (количество людей, единиц техники, объём материалов)
- **На какой период** (может быть не на весь срок работы)

### Ресурсная гистограмма

Система строит **графики загрузки** ресурсов:
- Ось X — время (дни, недели)
- Ось Y — количество ресурса
- Красная линия — максимум доступного ресурса

Если столбик выше красной линии — **перегрузка**: нужно либо увеличить ресурсы, либо сдвинуть работы.

## Часть 7: Учёт объёмов работ

**Путь:** Планирование → **Объёмы работ**

Для каждой работы можно вести учёт **физических объёмов**:

| Поле | Пример |
|------|--------|
| **Работа** | Бетонирование фундамента блок А |
| **Единица измерения** | м³ |
| **Плановый объём** | 450 м³ |
| **Выполнено** | 320 м³ |
| **Процент** | 71% |
| **Остаток** | 130 м³ |

Данные вносятся ежедневно или еженедельно на основании **актов выполненных работ** и **журнала производства работ**.

## Типичные сценарии

### Сценарий 1: Составление графика нового проекта

```
1. ГИП (главный инженер проекта) создаёт WBS:
   → ЖК «Южный парк»
   → 4 этапа: подготовка, подземная часть,
     надземная часть, инженерные системы
   → 45 работ в общей сложности

2. Для каждой работы:
   → Длительность (дни)
   → Зависимости (что за чем)
   → Ресурсы (бригады, техника, материалы)

3. Система автоматически:
   → Строит диаграмму Ганта
   → Рассчитывает критический путь (красный)
   → Показывает дату окончания: 15.12.2026
   → Выявляет перегрузку крана в июле

4. ГИП корректирует:
   → Сдвигает кладочные работы на неделю
   → Добавляет второй кран на июль
   → Итого: дата окончания 20.12.2026

5. Утверждает базовый план (Baseline 1) ✅
```

### Сценарий 2: Анализ отставания по EVM

```
1. Директор открывает EVM (6-й месяц):
   → SPI = 0.85 (отстаём на 15%)
   → CPI = 0.92 (перерасход 8%)

2. Разбирается, где проблема:
   → Монтаж каркаса — задержка 12 дней
   → Причина: арматуру привезли с дефектами,
     ждали замену

3. Принимает решение:
   → Увеличить бригаду монтажников (ускорить)
   → Выйти на работу в субботу (2 дополнительных дня)
   → Через месяц: SPI = 0.95 (почти догнали) ✅

4. Создаёт новый базовый план (Baseline 2)
   → Сравнение: Baseline 1 vs. Baseline 2
   → Сдвиг сдачи: +5 дней (было 15.12, стало 20.12)
```

### Сценарий 3: Еженедельный ввод объёмов

```
1. Каждую пятницу прораб вводит выполненные объёмы:
   → Бетонирование перекрытия 3 этажа: 120 м³ из 200
   → Кладка стен 2 этажа: 85 м² из 340
   → Электромонтаж 1 этажа: 100% ✅

2. Система обновляет прогресс на Ганте:
   → Зелёная полоска показывает фактический прогресс
   → Перекрытие: 60% (отстаёт от плана 75%)
   → Кладка: 25% (в графике)

3. ГИП видит отставание бетонных работ:
   → Проверяет критический путь — да, работа критическая
   → Запрашивает у прораба причину
   → Прораб: «Бетононасос сломался, ждём запчасть»
   → ГИП: заказывает подменный бетононасос
```

## Частые вопросы

**Что такое критический путь простыми словами?**
Это цепочка работ, от которых зависит дата сдачи проекта. Если любая из этих работ задерживается — задерживается весь проект. Остальные работы имеют «запас» по времени.

**Зачем нужен базовый план?**
Чтобы иметь эталон для сравнения. Без базового плана невозможно ответить на вопрос «мы отстаём или нет?» — потому что нет точки отсчёта.

**Сложно ли разобраться в EVM?**
На первый взгляд — да. Но по сути это всего три числа: «планировали», «сделали», «потратили». Всё остальное — производные от них. Система считает автоматически, вам нужно только вводить данные.

**Как часто нужно обновлять график?**
Рекомендуется — еженедельно. Минимум — раз в 2 недели. Без регулярного обновления график теряет актуальность.

**Можно ли импортировать график из MS Project?**
Система поддерживает импорт стандартных форматов. Обратитесь к администратору для настройки.

**Что делать, если критический путь поменялся?**
Это нормально. При обновлении данных критический путь пересчитывается. Важно следить за ним и реагировать, если новые работы стали критическими.
