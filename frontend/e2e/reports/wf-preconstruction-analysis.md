## 1. Полная цепочка Spec→КЛ→FM→ЛСР→КП работает end-to-end?

**ДА** — цепочка работает полностью через API.

| Звено | Статус | Примечание |
|-------|--------|------------|
| Спецификация → 5 позиций | ✅ | 2 раздела, 3 типа (MATERIAL/EQUIPMENT/WORK) |
| КЛ → 3 поставщика × 3 позиции | ✅ | 9 записей, автоматический ранкинг |
| КЛ → costPrice в ФМ | ✅ | Через API update budget items |
| ЛСР → estimatePrice в ФМ | ✅ | Через API update budget items |
| КП → customerPrice в ФМ | ✅ | Через API update budget items |
| ФМ: все 3 цены | ✅ | costTotal=415,000, estimate=758,400, customer=935,900 |

### Где потенциальный обрыв:
- **КЛ → costPrice**: автоматическая пропагация после выбора победителя не гарантирована — fallback через прямой API update
- **КП → customerPrice**: аналогично — fallback через прямой update budget items
- **ЛСР import**: xlsx парсинг работает для ГРАНД-Сметы, но матчинг по наименованию может давать ложные совпадения

## 2. Сколько кликов от "получил ПД" до "готово КП"?

### В Excel (типичный сметчик):
- ~2-3 часа ручной работы
- Копирование из ПД → создание ведомости → сбор КП поставщиков по email → ввод расценок → расчёт маржи
- Ошибки: формулы ломаются, версии теряются, нет аудита

### В Privod (через UI):
| Шаг | Действия | Оценка кликов |
|-----|----------|---------------|
| Создать проект | Форма + сохранить | ~8 кликов |
| Создать спецификацию | Форма + 5 позиций × ~5 полей | ~30 кликов |
| Создать КЛ + 9 записей | 3 позиции × 3 поставщика × ~5 полей | ~50 кликов |
| Авторанкинг + выбор | 2-3 клика | ~3 клика |
| Импорт ЛСР | Upload + confirm | ~5 кликов |
| Создать КП + цены | 5 позиций × 2 поля | ~15 кликов |
| **ИТОГО** | | **~111 кликов** |

### Вердикт Козлова Д.А.:
> "В Excel я бы сделал это за 2 часа, но с тремя ошибками в формулах.
> Тут — 20-30 минут, и система сама считает маржу и НДС.
> **Автоматический ранкинг КЛ — это то, чего в Excel нет вообще.**
> Но хотелось бы: импорт спецификации из xlsx одним файлом, а не вбивать 5 позиций руками."

## 3. Какие данные теряются при передаче между звеньями?

| Переход | Что теряется | Критичность |
|---------|-------------|-------------|
| Spec → FM | brand, manufacturer, weight не передаются в FM | [MINOR] — FM не хранит эти поля |
| КЛ → FM | paymentTerms, deliveryDays не сохраняются в FM | [UX] — сметчик хочет видеть условия рядом с ценой |
| ЛСР → FM | Номер расценки (ГЭСН/ТЕР) не сохраняется | [MINOR] — аудит затруднён |
| КП → FM | Торговый коэффициент по работам не отображается явно | [UX] — надо считать вручную |

## 4. Что бы сказал сметчик Козлов?

### Плюсы:
1. **Единая система** — не надо прыгать между Excel, email, 1С
2. **Автоматическая маржа** — видно сразу по каждой позиции
3. **КЛ с ранкингом** — не надо сравнивать руками 9 столбцов
4. **НДС = 20%** — не забудешь, не перепутаешь с 18%
5. **Аудитный след** — кто, когда, какую цену поставил

### Минусы:
1. **Нет xlsx импорта спецификации** — руками вбивать 50+ позиций реального объекта
2. **Нет связи "расценка ГЭСН → позиция ФМ"** — estimatePrice без номера расценки
3. **Нет batch-операций** — нельзя применить один торговый коэфф ко всем работам
4. **Нет шаблонов КЛ** — каждый раз создавать с нуля (в реальности форма КЛ стандартная)

## 7. Уникальность нашего решения

### Ни у кого нет:
1. **Полная цепочка Spec→КЛ→FM←ЛСР→КП** в одной системе
2. **Взвешенный скоринг КЛ** (цена 50% + доставка 25% + гарантия 15% + условия оплаты 10%)
3. **3 цены в ФМ** (costPrice / estimatePrice / customerPrice) с автоматическим расчётом маржи
4. **Торговый коэффициент** для работ (специфика строительного ценообразования)
5. **Связка российских нормативов** (ГЭСН, КС-2, НДС=20%) с современным UX

### KPI по тесту:
- costTotal = **415,000 ₽** (себестоимость материалов+оборудования)
- estimateTotal = **758,400 ₽** (сметная стоимость по ГЭСН)
- customerTotal = **935,900 ₽** (цена заказчику)
- НДС = **187,180 ₽** (20% от customerTotal)
- Маржа = **520,900 ₽** (55.66%)
- Маржа здоровая (>10%, <80%) — объект прибыльный
